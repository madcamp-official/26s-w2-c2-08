## 1. 문서 목적과 기준

본 문서는 [IA](./IA.md)에 정의된 화면이 무엇을 표시하고 사용자 동작과 시스템 상태에 어떻게 반응하는지 설명한다. 화면 구조와 상태 검토를 위한 정적 구현은 [Prototype index](../prototypes/index.html)에서 확인한다.

세부 기능과 계약은 다음 문서를 기준으로 하며 이 문서에서 반복하지 않는다.

- 제품 범위: [기획안](./기획안.md), [기능명세서](./기능명세서.md)
- HTTP·WebSocket: [API 명세서](../api/API_명세서.md), [OpenAPI](../api/openapi.yaml)
- 데이터 제약: [DB 스키마](../database/DB_스키마.md), [ERD](../database/ERD.md)
- 실행 경계: [기술명세서](../architecture/기술명세서.md), [시스템 구성도](../architecture/시스템구성도.md)

### 1.1 공통 원칙

- 사용자 역할은 계정 전체가 아니라 Course별 `PROFESSOR` 또는 `STUDENT`로 표시한다.
- 세션 표시 상태는 `시작 전`, `진행 중`, `정리 중`, `완료`를 사용한다.
- 실제 API와 연결되지 않은 Prototype 데이터에는 `Static prototype`을 표시한다.
- 로딩, 빈 상태, 부분 오류, 전체 오류와 권한 없음 상태를 정상 화면과 함께 설계한다.
- AI 기능 실패가 Course 탐색, Transcript와 질문 같은 핵심 기능을 막지 않게 한다.
- 임시 partial Transcript와 DB에 저장된 final Transcript를 시각적으로 구분한다.
- 모바일에서도 핵심 기능을 제거하지 않고 재배치하거나 접을 수 있게 한다.

### 1.2 공통 접근성·반응형

- 키보드 사용자를 위한 본문 건너뛰기 링크와 명확한 focus 표시를 제공한다.
- 입력에는 시각적 label을 연결하고 상태 메시지는 `aria-live` 또는 `role="alert"`로 알린다.
- Dialog는 Escape 종료, focus 순환과 열기 전 요소로의 focus 복귀를 지원한다.
- 동작 가능한 요소는 원칙적으로 최소 44px 높이를 확보한다.
- `prefers-reduced-motion`에서는 장식 animation과 smooth scroll을 최소화한다.
- 기준 viewport는 데스크톱 1440px, 태블릿 768px, 모바일 375px이다.

## 3. 서비스 진입

### 3.1 `MAIN_PAGE` · 비로그인 메인

#### 기본 정보

| 항목          | 내용                                                           |
| ------------- | -------------------------------------------------------------- |
| 목적          | GOAL의 핵심 가치를 설명하고 Google 로그인 흐름으로 진입시킨다. |
| 역할          | 비로그인 사용자                                                |
| 진입          | 비인증 상태로 서비스 URL 접속 또는 로그아웃 완료               |
| 이탈          | `LOGIN_PAGE`, 인증된 세션 복원 시 `MAIN_PAGE_AUTH`             |
| 기능·IA       | `MAIN_PAGE`, `LOGIN_BUTTON_AREA`                               |
| 실제 API 계약 | `GET /api/v1/auth/google/start`                                |
| Prototype     | [main.html](../prototypes/main.html)                           |

#### 화면 구성

- Header: GOAL 브랜드, 주요 기능·미리보기·Prototype 목록, Google 로그인 CTA
- Hero: 서비스 가치, 실시간 Transcript·익명 질문·AI 따라잡기 소개
- 실시간 class 미리보기: final 형태의 누적 문장과 임시 partial 문장, STT 연결 상태
- AI 미리보기: 현재 강의 요약과 익명 질문 작성 도움
- 핵심 가치: 수업 중 질문, 놓친 맥락 복구, 수업 후 복습
- 검토 도구: OAuth 기본·진행·오류·반환 성공 상태 전환

#### 상태

| 상태             | 표시                              | 가능한 동작                           |
| ---------------- | --------------------------------- | ------------------------------------- |
| 정상             | 가치 제안과 로그인 CTA            | 로그인 흐름 시작, 페이지 내 기능 탐색 |
| 세션 확인 중     | 향후 앱 초기화 skeleton·진행 상태 | 완료 대기                             |
| 로그인 진입 실패 | 인증 시작 실패 안내               | 다시 시도                             |
| 이미 인증됨      | 대시보드로 이동한다는 안내        | `MAIN_PAGE_AUTH` 이동                 |

정적 Prototype은 세션 확인 네트워크 요청을 만들지 않으며 로그인 상태 검토 도구로 화면만 전환한다.

#### 반응형·접근성

- 태블릿에서 Hero와 강의 미리보기를 세로로 배치한다.
- 모바일에서 Transcript 아래에 AI 요약과 질문 미리보기를 유지한다.
- 주요 기능 링크와 로그인 CTA는 키보드로 접근할 수 있다.
- 인식 중 문장에는 저장되지 않은 partial임을 텍스트로 함께 표시한다.

### 3.2 `LOGIN_PAGE` · Google 로그인 인계 상태

별도의 Google 화면을 복제하지 않고 `main.html`의 상태 영역에서 서비스가 책임지는 인계·반환 상태만 표현한다.

#### 기본 정보

| 항목           | 내용                                                                |
| -------------- | ------------------------------------------------------------------- |
| 목적           | OAuth 인계, callback 확인과 오류 복구를 사용자에게 설명한다.        |
| 역할           | 비로그인 사용자                                                     |
| 진입           | `MAIN_PAGE`의 Google 로그인 CTA                                     |
| 이탈           | 성공 시 `MAIN_PAGE_AUTH`, 취소·실패 시 `MAIN_PAGE`                  |
| 실제 API 계약  | `GET /api/v1/auth/google/start`, `GET /api/v1/auth/google/callback` |
| Prototype 상태 | `main.html?auth=default`, `loading`, `error`, `success`             |

#### 상태와 동작

| 상태      | 표시                                   | 가능한 동작         |
| --------- | -------------------------------------- | ------------------- |
| `default` | Google 로그인 CTA와 Course별 역할 안내 | 로그인 시작         |
| `loading` | Google 이동 중·callback 대기 안내      | 성공·실패 모의 전환 |
| `error`   | 취소 또는 제공자 오류                  | 다시 시도           |
| `success` | callback 확인 완료                     | 대시보드 이동       |

- 진행 중에는 중복 로그인 요청을 만들지 않는다.
- 실제 인증 provider 응답과 token을 화면이나 log에 표시하지 않는다.
- 오류 메시지는 계정 존재 여부 등 민감한 내부 상태를 노출하지 않는다.
- 정적 Prototype에서는 실제 OAuth·Cookie·네트워크 요청을 사용하지 않는다.

#### 미정 사항

- 실제 앱에서 초기 server Session 복원 중 사용할 skeleton 범위
- OAuth provider 장애와 사용자 취소를 서로 다른 문구로 구분할지 여부

### 3.3 `MAIN_PAGE_AUTH` · 로그인 후 대시보드

#### 기본 정보

| 항목          | 내용                                                                             |
| ------------- | -------------------------------------------------------------------------------- |
| 목적          | 사용자가 Course별 역할과 현재 class 상태를 파악하고 다음 작업으로 진입하게 한다. |
| 역할          | 인증된 사용자 공통, 각 카드 안에서 Course별 역할 표시                            |
| 진입          | Google callback 성공, 인증 Session이 남은 상태로 재접속                          |
| 이탈          | Course 생성·참여, `MY_INFO_PAGE`, 역할별 Course·class                            |
| 기능          | `PRE-T-01`, `PRE-S-01~03`, `POST-07`, `SYS-01`, `SYS-03`                         |
| 실제 API 계약 | `GET /api/v1/me`, `GET /api/v1/courses`                                          |
| Prototype     | [dashboard.html](../prototypes/dashboard.html)                                   |

#### 화면 구성

- 공통 Header: 홈, Course, 완료 강의 기록, 알림, Profile disclosure
- 인사 영역: 현재 시각에 맞는 인사와 날짜
- 진행 중 class: Course, class 제목, STT 상태, 참여·질문 수와 입장 동작
- 빠른 실행: Course 생성과 참여 진입
- 관리 중인 Course: 해당 Course에서 `PROFESSOR`인 항목
- 참여 중인 Course: 해당 Course에서 `STUDENT`인 항목
- 상태 검토 도구: 두 목록의 상태를 서로 독립적으로 전환

#### Course 카드 표시

| 데이터           | 표시 규칙                                                    |
| ---------------- | ------------------------------------------------------------ |
| Course 이름·학기 | 모든 카드에 표시                                             |
| 역할             | `교수자` 또는 `학생`, 계정 전역 역할처럼 표시하지 않음       |
| 현재 class       | `시작 전`, `진행 중`, `정리 중`, `완료` 사용자 문구로 표시   |
| 참여 코드        | 관리 중인 Course 카드에만 예시 표시, 학생 카드에는 표시 금지 |
| 완료 기록        | 학생 Course의 최근 완료 기록과 개수 표시                     |

#### 목록 상태

관리 목록과 참여 목록은 별도의 요청 결과로 간주해 한쪽 오류가 다른 쪽을 가리지 않게 한다.

| 상태      | 표시                             | 가능한 동작            |
| --------- | -------------------------------- | ---------------------- |
| `normal`  | 역할별 Course 카드               | Course 진입, 목록 탐색 |
| `loading` | 카드 형태 skeleton과 `aria-busy` | 완료 대기              |
| `empty`   | 역할별 안내와 생성 또는 참여 CTA | 해당 흐름 진입         |
| `error`   | 실패한 목록만 오류 표시          | 해당 목록만 다시 시도  |
| 인증 만료 | Course 내용을 숨기고 로그인 안내 | `MAIN_PAGE` 이동       |

Prototype은 `dashboard.html?owned=normal`, `loading`, `empty`, `error`와 `joined` query를 조합해 상태를 재현한다. 인증 만료는 `dashboard.html?auth=expired`로 확인한다.

#### 동작과 오류

- 현재 PR에는 후속 Course 화면이 없으므로 생성·참여·Course 진입은 후속 Prototype 안내 Toast만 표시한다.
- Profile disclosure는 click, 바깥 영역 click과 Escape로 닫을 수 있다.
- 인증 만료 상태에서는 Course 이름이나 참여 코드 목 데이터도 표시하지 않는다.
- 실제 앱에서는 401 응답 시 비로그인 메인으로 이동하고 다시 로그인을 요청한다.

#### 반응형·접근성

- 940px 이하에서 상단 중앙 nav 대신 가로 스크롤 가능한 모바일 nav를 제공한다.
- 태블릿에서 진행 중 class와 빠른 실행을 세로로 배치한다.
- 모바일에서 Course 카드를 한 열로 배치하고 진행 중 class의 메타정보와 CTA를 세로로 정렬한다.
- Profile은 복잡한 ARIA menu가 아닌 disclosure navigation으로 구현한다.
- 목록 로딩 메시지는 시각적으로 숨겨도 스크린 리더에 전달한다.

### 3.4 `MY_INFO_PAGE` · 내 정보

#### 기본 정보

| 항목          | 내용                                                                |
| ------------- | ------------------------------------------------------------------- |
| 목적          | 인증 계정의 기본 정보와 Course별 역할 요약을 확인하고 로그아웃한다. |
| 역할          | 인증된 사용자 공통                                                  |
| 진입          | 대시보드의 Profile disclosure에서 내 정보 선택                      |
| 이탈          | 대시보드, 역할별 Course, 로그아웃 후 `MAIN_PAGE`                    |
| 기능          | `SYS-01`, `SYS-03`, 인증·로그아웃                                   |
| 실제 API 계약 | `GET /api/v1/me`, `GET /api/v1/courses`, `POST /api/v1/auth/logout` |
| Prototype     | [my-info.html](../prototypes/my-info.html)                          |

#### 화면 구성

- 계정 기본 정보: avatar, 표시 이름, Google 계정 이메일과 연결 상태
- 계정 상세: 로그인 방식, 가입일과 고정 계정 역할이 없다는 안내
- Course 요약: 관리 중인 Course와 참여 중인 Course 개수·대시보드 이동
- Session 보안: 공용 기기 안내와 로그아웃
- 상태 검토 도구: 정상, 로딩, Course 없음과 조회 오류

계정 정보 수정은 IA에서 MVP 선택 기능이므로 1차 MVP 확정 동작으로 제공하지 않는다.

#### 상태

| 상태      | 표시                                    | 가능한 동작             |
| --------- | --------------------------------------- | ----------------------- |
| `normal`  | 계정 정보와 역할별 Course 요약          | 대시보드 이동, 로그아웃 |
| `loading` | 계정·Course 요약 skeleton과 `aria-busy` | 완료 대기               |
| `empty`   | 계정 정보와 Course 0개 안내             | 대시보드 이동           |
| `error`   | 민감한 계정 정보를 숨긴 오류            | 다시 시도               |
| 인증 만료 | 모든 계정 정보를 숨기고 로그인 안내     | `MAIN_PAGE` 이동        |

Prototype은 `my-info.html?state=normal`, `loading`, `empty`, `error`에 해당하는 개별 query 값으로 상태를 재현하며, 인증 만료는 `my-info.html?auth=expired`로 확인한다.

#### 로그아웃

1. 사용자가 로그아웃을 선택한다.
2. 확인 Dialog가 열리고 focus가 Dialog 안으로 이동한다.
3. 실제 앱은 `POST /api/v1/auth/logout` 성공 후에만 비로그인 화면으로 이동한다.
4. 실패하면 현재 화면과 Session을 유지하고 오류를 알린다.
5. Prototype은 성공·실패 결과를 로컬 이동과 Toast로 모의한다.

#### 반응형·접근성

- 태블릿 이하에서 계정 카드와 Course 요약을 한 열로 배치한다.
- 모바일에서는 계정 avatar와 텍스트를 세로로 배치한다.
- 로그아웃 Dialog는 Escape, focus trap과 trigger focus 복귀를 지원한다.
- 이름·이메일 조회 실패 시 이전 목 데이터를 남겨 성공처럼 보이지 않게 한다.

#### 미정 사항

#### 미정 사항

Prototype은 `course-join.html?state=form`, `checking`, `success`, `already`, `invalid`, `conflict`, `error`, `expired`로 상태를 재현하며 실제 코드 검증 요청을 만들지 않는다.

- 상태 전환은 `aria-live` 영역과 Toast로 전달한다.
- 태블릿 이하에서는 코드 입력과 안내를 한 열로, 모바일에서는 결과 동작을 세로로 배치한다.

#### 미정 사항

- 참여 코드의 alphabet·길이, 자동 만료와 회전 정책

- 태블릿에서는 Course 소개·코드, 현재 class·관리 영역을 각각 한 열로 바꾼다.
- 모바일에서는 class 메타와 동작, 기록 행을 세로 중심으로 재배치한다.

#### 미정 사항

## 5. class 생성과 수업 준비

### 5.1 `CLASS_CREATE_PAGE` · class 생성·복수 PDF 준비

#### 기본 정보

| 항목      | 내용                                                                                       |
| --------- | ------------------------------------------------------------------------------------------ |
| 목적      | Course 교수자가 `READY` class를 생성하고 여러 PDF를 업로드·처리한 뒤 시작 상태를 확인한다. |
| 역할      | 해당 Course의 `PROFESSOR`                                                                  |
| 진입      | 교수자 Course에서 새 class 만들기 또는 `READY` class 준비 선택                             |
| 이탈      | 취소 시 `COURSE_PAGE_PROF`, 시작 성공 시 `LIVE_CLASS_PAGE_PROF`                            |
| 기능·IA   | `CLASS_CREATE_PAGE`, `CLASS_MATERIAL_UPLOAD`, `LIVE_CLASS_START`, `PRE-T-03~05`            |
| Prototype | [class-create.html](../prototypes/class-create.html)                                       |

#### 실제 API 순서

1. `GET /api/v1/courses/{course_id}`로 교수자 역할과 다른 `LIVE` class 여부를 확인한다.
2. `POST /api/v1/courses/{course_id}/sessions`에 제목과 `lecture_date`를 보내 `READY` Session과 `session_id`를 얻는다.
3. 재진입 시 `GET /api/v1/sessions/{session_id}`로 상태를 확인하고 `LIVE`, `PROCESSING`, `COMPLETED`이면 해당 화면으로 이동한다.
4. 여러 PDF를 선택하더라도 `POST /api/v1/sessions/{session_id}/materials`는 파일 하나당 한 번 호출한다.
5. `202 Accepted` 응답의 Material과 `MATERIAL_PROCESSING` Job을 기준으로 자료 목록·Material·Job 조회를 갱신한다.
6. 실패 Job은 `POST /api/v1/jobs/{job_id}/retry`로 같은 행의 `attempt`를 증가시킨다.
7. `POST /api/v1/sessions/{session_id}/start` 성공 시 `READY → LIVE`로 전환한다.

Prototype은 위 요청을 실행하지 않고 목 상태와 로컬 파일 선택만 바꾼다.

#### class 생성 상태

| 상태                                   | 표시와 동작                                               |
| -------------------------------------- | --------------------------------------------------------- |
| `draft`                                | 제목·날짜 필수 입력, Course 복귀와 생성                   |
| `creating`                             | Session 생성 진행 상태와 중복 제출 방지                   |
| `ready`                                | Session ID 확보 후 PDF 업로드와 시작 준비 제공            |
| `create-error`                         | 입력을 다시 확인하고 생성 요청 재시도                     |
| `live`, `processing`, `completed` 응답 | 편집을 제공하지 않고 해당 실시간·정리·완료 화면 이동 안내 |

`class-create.html?stage=draft|creating|ready|create-error|live|processing|completed`로 확인한다.

#### PDF 선택·업로드·처리 상태

- `<input type="file" multiple accept="application/pdf,.pdf">`와 연결된 label을 제공한다.
- 서버가 수락하기 전 로컬 선택 파일만 파일명 단위로 취소할 수 있다.
- 수락 후 생성된 Material에는 확정되지 않은 삭제 동작을 제공하지 않는다.
- 같은 PDF 재업로드를 허용하고 class당 여러 Material을 표시한다.
- 파일별 상태는 `UPLOADED`, `PROCESSING`, `READY`, `FAILED`를 사용한다.
- 한 파일 실패가 다른 파일의 상태, class 정보와 시작 상태를 가리지 않는다.
- 진행률을 계산할 수 없으면 숫자를 꾸며내지 않고 불확정 `<progress>`와 처리 단계 텍스트를 쓴다.
- 실패 Material 재시도는 새 Job처럼 표현하지 않고 같은 Job의 `attempt 1 → 2`로 갱신한다.

| Prototype 상태 | 표시                                                      |
| -------------- | --------------------------------------------------------- |
| `empty`        | 파일 선택, 서버 수락 전 선택 취소                         |
| `uploading`    | 파일별 독립 업로드와 202 수락 구분                        |
| `processing`   | `UPLOADED` 대기와 `PROCESSING` 진행                       |
| `mixed`        | `READY`, `PROCESSING`, `FAILED` 동시 표시와 파일별 재시도 |
| `all-ready`    | 복수 PDF 처리 완료                                        |
| `failed`       | 재시도 가능한 실패와 안전한 오류 안내                     |
| `load-error`   | class·시작 정보는 유지하고 자료 영역만 재시도             |

`class-create.html?stage=ready&materials=empty|uploading|processing|mixed|all-ready|failed|load-error`로 확인한다.

#### 시작 상태와 권한

- Course당 `LIVE` class는 최대 하나이며 경쟁 시작은 `409 SESSION_STATE_CONFLICT`로 안내한다.
- `start=idle|starting|conflict|error|success`로 시작 요청 상태를 재현한다.
- 일반 시작 실패에도 `READY` 정보와 PDF 결과를 유지한다.
- 인증 만료와 교수자 권한 없음에서는 class 입력, 파일명과 처리 상태를 모두 숨긴다.
- `auth=valid|expired|forbidden`으로 접근 경계를 확인한다.

#### 접근성·반응형

- 제목·날짜 입력 오류를 설명과 연결하고 첫 오류 필드로 focus를 이동한다.
- PDF가 아닌 파일은 파일명을 포함한 `role="alert"` 오류로 알린다.
- 파일 목록은 semantic `ul`·`li`를 사용하고 재시도 버튼 이름에 파일명을 포함한다.
- 로딩·업로드·처리 영역에 `aria-busy`, `aria-live`, `<progress>` 또는 명시적 처리 텍스트를 제공한다.
- 태블릿 이하에서는 준비 영역과 시작·API 안내를 한 열로 바꾸고 모바일에서도 모든 파일 상태와 동작을 유지한다.

#### 미정 사항

- 강의자료 없이 class 시작을 허용할지 여부
- `UPLOADED`·`PROCESSING` Material이 있을 때 시작을 허용할지 여부
- 생성 후 class 제목·날짜 수정과 class 취소 정책
- 서버가 수락한 Material 삭제와 스토리지 보상 삭제 UI
- PDF 개수와 용량의 숫자 제한

#### API·OpenAPI 후속 동기화 필요

- API 명세와 OpenAPI에는 Course당 동시 `LIVE` 수가 아직 TBD로 남아 있으나 DB 문서는 최대 1개로 확정했다.
- API 명세와 OpenAPI의 PDF 카디널리티 설명은 TBD지만 DB 문서는 class당 복수 PDF와 동일 파일 재업로드를 허용한다.
- OpenAPI의 Job retry 설명은 새 행 또는 기존 행이 TBD라고 되어 있으나 DB 문서는 같은 Job 행의 `attempt` 증가로 확정했다.
- 위 세 항목을 API 명세와 OpenAPI에서 동기화하기 전까지 화면 구현은 최신 DB 결정과 이 화면설계서를 따른다.

#### 기본 정보

| --------- | ------------------------------------------------------------------------------------------------------------- |
| 목적      | 교수자가 Audio·STT 상태를 확인하며 익명 질문을 선택하고 final Transcript로 답변을 연결한 뒤 class를 종료한다. |
| 역할      | 해당 Course의 `PROFESSOR`, Session 상태 `LIVE`                                                                |
| 진입      | class 시작 성공 또는 진행 중 class 재입장                                                                     |
| 이탈      | 종료 성공 시 `CLASS_PROCESSING_STATE`, 상태 불일치 시 역할별 Course·class 화면                                |
| 기능·IA   | `LIVE-T-01~08`, `LIVE_AUDIO_STREAM`, `LIVE_TRANSCRIPT_AREA`, `LIVE_QUESTION_AREA`, `LIVE_CLASS_QUIT`          |

#### 독립된 세 상태 축

다음 상태는 원인과 복구 방식이 다르므로 하나의 “연결 상태”로 합치지 않는다.

| 축               | Prototype 상태                                                                                                               | 의미                                          |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------- |
| Session event WS | `connected`, `reconnecting`, `resync`, `stopped`                                                                             | 질문·Answer·Session 변경 알림과 resume cursor |
| Audio WS         | `permission`, `denied`, `unavailable`, `connecting`, `listening`, `degraded`, `reconnecting`, `gap`, `finalizing`, `stopped` | 교수자 마이크 권한, chunk 전송·ack·drain      |
| STT              | `listening`, `degraded`, `finalizing`, `finalized`, `stopped`                                                                | 음성 인식 처리와 final 생성 완료 여부         |

Prototype query는 각각 `events`, `audio`, `stt`를 사용한다. 마이크 권한 버튼은 사용자 gesture 이후의 UI만 모의하며 `getUserMedia`, 실제 장치, WebSocket을 사용하지 않는다. Audio 또는 STT 장애가 질문 조회·선택을 막지 않는다.

#### Transcript 규칙

- `transcript.partial`은 `utterance_id`별 더 큰 `revision`만 같은 표시를 치환한다.
- partial에는 `인식 중`, `저장되지 않음`, `답변 후보 제외`를 함께 표시하고 `aria-live="off"`로 둔다.
- 재연결·resync 시 partial을 제거하며 replay·REST 복구 대상으로 사용하지 않는다.
- `transcript.final`은 DB commit 뒤에만 받고 `sequence` 순서로 누적한다.
- final 수신 시 같은 `utterance_id` partial을 제거하고 Answer가 CAPTURING이면 선택 경계 이후 final만 후보로 표시한다.
- 재접속 시 `GET /sessions/{id}/transcript?after_sequence=`가 canonical 결과다.
- 복구 불가능한 audio 구간은 추측한 문장 대신 gap으로 표시한다.
- Transcript 영역의 `normal`, `loading`, `empty`, `resync`, `error`는 질문 영역과 독립적이다.

#### 질문·클러스터·정렬

- 질문은 `reaction_count`를 질문별로 표시하고 `POPULAR` 또는 `RECENT` 정렬을 제공한다.
- Cluster는 title·summary와 원본 질문 disclosure를 제공하지만 반응 수를 임의 합산하지 않는다.
- 선택 시점의 Cluster membership 변경 이력을 화면에서 재구성하지 않는다.
- `normal`, `unclustered`, `cluster-failed`, `loading`, `empty`, `error`를 독립적으로 표시한다.
- 클러스터링 실패에도 원본 질문 조회와 답변 기능을 유지한다.
- 교수자에게 학생 작성자 ID·이름·이메일을 표시하지 않는다.

#### Answer 캡처

1. 단일 질문 또는 현재 Cluster를 선택해 `POST /sessions/{id}/answers`를 요청한다.
2. 서버 응답의 `question_ids`와 `capture_started_after_sequence`를 선택 결과의 진실로 사용한다.
3. 첫 새 final 전에는 `CAPTURING · waiting`, 이후에는 final만 candidate로 표시한다.
4. 완료 시 자동 범위를 확정해 Answer를 `COMPLETED`, 연결 질문을 `ANSWERED`로 바꾼다.
5. `ANSWER_TRANSCRIPT_NOT_READY`이면 오류를 표시하되 CAPTURING을 유지한다.
6. 취소 시 Answer를 `CANCELLED`, 질문을 `OPEN`으로 되돌린다.

Session당 CAPTURING Answer는 하나이고 질문당 취소되지 않은 active Answer는 하나이므로 캡처 중에는 다른 답변 시작을 비활성화한다. partial은 후보가 아니다. Cluster 과거 제목 snapshot은 현재 OpenAPI 응답에 없어 화면에서는 “선택 질문 N개”만 안전하게 표시한다. 실시간 텍스트 답변 입력은 제공하지 않는다.

#### class 종료

- CAPTURING Answer가 있으면 종료 Dialog를 열지 않고 완료 또는 취소를 먼저 요청한다.
- 정상 순서는 `audio.stop → audio.stopped + STT FINALIZED → Idempotency-Key가 있는 POST /sessions/{id}/end → PROCESSING`이다.
- Dialog는 Escape, focus trap과 trigger focus 복귀를 지원한다.
- `class-processing.html`은 PR 6 전까지 깨진 링크를 만들지 않고 성공 목 상태와 Toast로 후속 계약만 알린다.
- stop timeout과 강제 종료 세부 UX는 확정하지 않는다.

#### 전체·부분 상태와 API/WS

- 전체 `view=normal|loading|error|forbidden|expired|wrong-state`를 제공하고 권한·인증 오류에는 수업 데이터를 숨긴다.
- Session events ticket과 Audio ticket은 각각 `SESSION_EVENTS_READ`, `SESSION_AUDIO_WRITE` scope의 별도 1회용 티켓이다.
- Session WS는 `transcript.*`, `question.*`, `reaction.updated`, `answer.updated`, `session.updated` 알림을 전달한다.
- REST `GET /sessions/{id}`, `/transcript`, `/questions`, `/answers`가 재연결·resync의 최종 진실이다.
- 답변 시작·완료·취소 API와 멱등 종료 API를 화면 동작에 매핑한다.
- 참여자 수 API가 없으므로 실제 수치를 표시하지 않는다.

#### 접근성·반응형

- 중요한 final·연결·Answer 변경은 별도 `aria-live="polite"` 영역으로 알린다.
- Cluster disclosure는 `aria-expanded`와 `aria-controls`를 동기화한다.
- 정렬 버튼은 `aria-pressed`, Answer bar는 `tabindex="-1"` focus와 후보 설명을 제공한다.
- 데스크톱은 Transcript `1.35fr`, 질문 `380~400px`, active Answer 전체 폭으로 배치한다.
- 900px 이하에서는 Transcript·질문·Answer를 한 열로 배치하며 768px과 375px에서도 질문·상태를 숨기지 않는다.
- reduced-motion에서는 skeleton·Toast·scroll animation을 최소화한다.

#### 미정 사항

- 중복 탭·장치의 audio publisher takeover 정책
- `DEGRADED` 전환 지연 임계값
- Answer 취소의 최종 확인 UX와 수동 Transcript 범위 조정 수준
- 실시간 교수자 텍스트 답변 포함 여부
- Cluster 품질·점수 표시와 OpenAPI의 Cluster snapshot title field
- Audio stop timeout과 강제 종료 UI
- 교수자의 LIVE AI 요약·Chat 권한과 보관 정책

### 6.2 `LIVE_CLASS_PAGE_STUD` · 학생 실시간 class

#### 기본 정보

| 항목      | 내용                                                                                    |
| --------- | --------------------------------------------------------------------------------------- |
| 목적      | 학생이 final·partial Transcript를 확인하고 익명 질문·반응·개인 AI 요약·Chat을 이용한다. |
| 역할      | 해당 Course의 `STUDENT`, Session `LIVE`                                                 |

#### 연결과 Transcript

- 학생은 `SESSION_EVENTS_READ` 1회용 ticket과 Session event WS만 사용하며 Audio WS·마이크 UI를 제공하지 않는다.
- Session 연결 `connected|reconnecting|resync|error|stopped`와 STT `listening|degraded|finalizing|finalized|stopped`를 별도로 표시한다.
- partial은 같은 `utterance_id`의 더 큰 revision만 치환하고 저장·AI 근거에 포함하지 않으며 재연결 시 제거한다.

- 전체 `view=normal|loading|error|forbidden|expired|wrong-role|processing|completed`와 영역별 상태를 독립 제공한다.

#### 익명 질문·반응

- 질문 textarea에 label과 익명성 안내를 제공하며 trim 결과가 비면 오류를 연결하고 focus한다.
- 최대 길이나 동일 의미 중복 거부 규칙은 임의로 만들지 않는다.
- 전송 중 중복 제출을 비활성화하고 성공 후에만 입력을 비운다. 실패하면 원문을 유지한다.
- 질문 저장과 클러스터링은 독립이므로 클러스터 실패에도 질문·반응을 유지한다.
- 반응 PUT·DELETE는 `aria-pressed`와 count를 함께 낙관 갱신하고 실패 시 `previousPressed`와 이전 count로 정확히 rollback한다.
- `reaction.updated`의 count는 개인 `reacted_by_me` 상태를 덮지 않는다.
- API에 `is_mine`·`can_react`가 없어 내 질문 badge나 재접속 후 사전 비활성화를 계약으로 만들지 않는다. 서버 거부 후 rollback만 모의한다.

#### AI 질문 문장 작성 도움

- `idle → requesting → suggestions → editing`으로 진행하며 빈 초안과 실패 상태를 제공한다.
- 후보는 radio fieldset으로 선택하고 editable textarea에서 학생이 수정한 뒤 별도 질문 등록을 실행한다.
- 후보 선택만으로 공개 질문을 저장하지 않는다. 실패 시 원문을 유지해 다시 요청한다.
- `POST /sessions/{id}/question-drafts`는 직접 `200` 후보 응답이므로 AIJob·Job retry와 연결하지 않는다.

#### 개인 LIVE Summary와 Chat

- Summary는 요청 시점 마지막 final 범위를 확정하고 `202 LIVE_SUMMARY Job → PENDING/RUNNING → GET Job polling → REST Summary`로 처리한다.
- partial은 Summary 근거가 아니며 `idle|pending|running|complete|error|retry` 상태를 제공한다.
- Chat은 사용자 Message 저장과 `CHAT_RESPONSE` Job 생성 후 polling하고 REST Message를 조회한다. 한 Chat에는 진행 Job 하나만 허용한다.
- Chat 실패에도 기존 대화를 유지하며 retry는 같은 Job의 `attempt`를 증가시킨다.
- `idle|pending|evidence|no-evidence|error|retry`를 구분하고 근거 없음도 저장 완료 결과로 표시한다.
- Summary·Chat은 requester-only라 shared Session WS에 broadcast하지 않는다. private transport가 미정이므로 token delta를 구현하지 않는다.
- Evidence는 DB KnowledgeChunk를 기준으로 하지만 OpenAPI와 불일치하므로 화면에는 안전한 자료·Transcript category label만 표시한다.
- AI 실패 중에도 Transcript·질문·반응을 계속 사용할 수 있다.

#### 접근성·반응형

- final과 중요 상태는 polite announcer, partial은 `aria-live="off"`를 사용한다.
- 질문은 오류·`aria-busy`, draft는 fieldset·radio, reaction은 최소 44px·`aria-pressed`·count를 제공한다.
- Chat은 `role="log"`, 처리 상태와 오류 alert를 제공하고 결과 도착 시 강제 focus·scroll 이동을 하지 않는다.
- 데스크톱은 Transcript 주 영역과 질문·AI rail, 900px 이하에서는 한 열로 배치한다.
- 375px 순서는 Header→연결→Transcript→질문 작성→질문 목록→Summary→Chat이며 AI 영역을 숨기지 않는다.

#### 미정 사항

- `Question.is_mine`·`can_react` 공개 필드와 자기 질문 반응 오류 code
- 질문·초안·Chat 입력 최대 글자 수
- requester-only AI private streaming transport
- final Transcript가 하나도 없을 때 Summary의 정확한 오류 code
- KnowledgeChunk 기반 Evidence 공개 field와 안전한 link 형식
- LIVE Chat·Summary 보관 기간과 삭제 정책

## 7. class 기록 정리 중

### 7.1 `CLASS_PROCESSING_STATE`

- 목적: Session `PROCESSING` 동안 독립 후처리 Job과 이미 저장된 기록을 동시에 제공한다.
- 역할: Course 교수자·학생. `class-processing.html?role=professor|student`로 권한별 CTA와 retry를 확인한다.
- `GET /sessions/{id}/record`는 PROCESSING부터 PDF, final Transcript, 원본 질문·반응과 LIVE에서 확정한 Answer range를 canonical하게 반환한다.
- Audio 연결과 partial은 없으며 선택적 Session event WS의 `transcript.final/status`, `question/answer/job/session.updated` 알림 뒤 REST로 복구한다.
- 흐름은 CAPTURING 해소 → audio drain·추가 final → PROCESSING → blocking Job → Transcript FINALIZED → FINAL Summary·마지막 clustering → 모든 blocking Job terminal 후 서버 COMPLETED다.
- 독립 작업은 `SESSION_POSTPROCESSING`, `FINAL_SUMMARY`, 필요 시 마지막 `QUESTION_CLUSTERING`이며 별도 `ANSWER_LINKING` Job을 만들지 않는다.
- Job은 `PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `retrying`, attempt를 표시한다. percent가 제공될 때만 progress를 표시한다.
- 교수자만 FAILED·retryable 공유 Job을 같은 `job_id`, `attempt + 1 → PENDING`으로 재시도한다. 학생은 교수자 재시도 안내만 본다.
- Summary·clustering 실패가 PDF·final Transcript·원본 질문·Answer를 가리지 않는다. 새 질문·반응은 금지한다.
- 최종 Cluster는 현재 확정본만 표시하고 과거 membership 이력은 제공하지 않는다.
- 완료는 클라이언트 Job count가 아니라 `session.updated=COMPLETED` 또는 REST Session이 진실이다. 일부 실패여도 모두 terminal이면 COMPLETED이며 자동 이동 대신 역할별 CTA를 제공한다.
- 상태 query: `session=processing|completed`, `transcript=finalizing|finalized|failed|empty`, `postprocess`, `summary`, `clustering`의 독립 Job 상태와 `connection` 복구 상태를 사용한다.
- 900px 이하에서 작업과 저장 기록을 한 열로 배치하되 저장 기록을 spinner 아래 숨기지 않는다. terminal 변화는 polite, 실패는 alert, retry 이름에는 작업명을 포함한다.

미정·API gap: `blocks_session_completion`, Cluster `is_final/finalized_at`, Session `completed_at`, progress stage 공개값, SLO, 자동 전환, 공유 Job retry 범위와 Transcript gap API를 동기화해야 한다.

## 8. 완료 class 기록

### 8.1 `ENDED_CLASS_PAGE_PROF` · 교수자 완료 기록

- `COMPLETED` Session에서 `GET /sessions/{id}/record`의 materials, final Transcript, FINAL Summary, questions, answers와 shared jobs를 canonical하게 표시한다. `view=loading|normal|error|forbidden|expired|processing`을 제공한다.
- 복수 PDF는 파일명·크기·쪽 수·처리 상태와 권한 확인된 원본 열람만 제공하며 업로드·삭제·음성 재생·다운로드를 제공하지 않는다.
- Transcript는 final만 sequence 오름차순으로 전체 표시한다. Answer는 자유 텍스트가 아니라 final range이며 같은 Answer가 여러 질문에 연결돼도 한 번만 `연결 질문 N개`로 렌더링한다.
- Answer 범위 선택 시 해당 sequence에 focus하고 highlight한다. `source_cluster_id`를 현재 Cluster에 다시 join하지 않고 snapshot title 공개 field가 없어 과거 제목을 꾸며내지 않는다.
- 질문·반응·Answer 시작은 모두 읽기 전용이다. 최종 현재 Cluster와 원본 질문만 표시하며 과거 membership 이력을 제공하지 않는다.
- 영역 상태는 materials `normal|multiple|empty|failed|error`, transcript `normal|empty|error`, summary `normal|empty|failed|retrying`, questions `normal|empty|cluster-failed|error`, jobs `success|partial-failure|retrying`이다.
- `summary=null`은 Job과 함께 해석하며 SUCCEEDED인데 null이면 데이터 불일치다. Summary·clustering 실패가 PDF·Transcript·원본 질문·Answer를 가리지 않는다.
- 유일한 쓰기는 교수자의 FAILED+retryable shared Job 재시도다. 같은 `job_id`의 attempt를 증가시키고 Session은 COMPLETED를 유지한다. 성공 FINAL Summary 다시 생성과 교수자 REVIEW Chat은 제공하지 않는다.
- 완료 화면은 WS가 필수가 아니며 REST가 진실이다. 900px 이하 한 열에서도 PDF→Summary→Transcript→Q&A→Jobs 순서를 유지한다.

미정: 교수자 REVIEW Chat, Cluster `is_final` 공개 field와 정렬, final clustering Job 구분, Answer snapshot title·CANCELLED 포함 정책, summary null/no Job, pagination을 후속 동기화한다.
