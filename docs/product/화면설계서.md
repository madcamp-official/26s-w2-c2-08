## 1. 문서 목적과 기준

본 문서는 [IA](./IA.md)에 정의된 화면이 무엇을 표시하고 사용자 동작과 시스템 상태에 어떻게 반응하는지 설명한다. 화면 구조와 상태 검토를 위한 정적 구현은 [Prototype index](../prototypes/index.html)에서 확인한다.

세부 기능과 계약은 다음 문서를 기준으로 하며 이 문서에서 반복하지 않는다.

- 제품 범위: [기획안](./기획안.md), [기능명세서](./기능명세서.md)
- HTTP·WebSocket: [API 명세서](../api/API_명세서.md), [OpenAPI](../api/openapi.yaml)
- 데이터 제약: [DB 스키마](../database/DB_스키마.md), [ERD](../database/ERD.md)
- 실행 경계: [기술명세서](../architecture/기술명세서.md), [시스템 구성도](../architecture/시스템구성도.md)

### 1.1 공통 원칙

- 로그인 직후의 인증 사용자는 계정 전역 역할을 갖지 않으며 Course 생성과 참여를 모두 이용할 수 있다.
- Course 생성자는 해당 Course의 유일한 `PROFESSOR` owner가 된다. Course당 교수자는 한 명이며 추가하거나 이전하지 않는다.
- Course에는 `READY`, `LIVE`, `PROCESSING`을 합쳐 active class가 최대 하나만 존재하고 `current_session`은 그 class 또는 `null`이다.
- 세션 표시 상태는 `시작 전`, `진행 중`, `정리 중`, `완료`를 사용한다.
- 같은 `lecture_date`의 완료 class는 실제 시작 시각을 함께 표시하고 `started_at` 내림차순으로 구분한다.
- class 제목은 모든 상태에서 수정할 수 있지만 날짜와 시작·종료·완료 시각은 수정할 수 없다.
- Material 삭제는 사용자에게는 PDF 삭제로 표현하되, 성공 응답 뒤에는 Session 연결이 즉시 해제되어 목록·열람·새 AI 검색에서 사라지는 것으로 처리한다.
- AI는 현재 연결된 `READY` Material만 참고한다. `UPLOADED`, `PROCESSING`, `FAILED`와 삭제된 Material을 새 근거로 사용하지 않는다.
- 교수자 마이크 입력은 고정 실시간 Audio WS와 브라우저 로컬 녹음의 두 경로로 분기하며, 각 경로의 상태와 실패를 독립적으로 표시한다.
- 저장 녹음 playback은 요청마다 서버 권한을 다시 확인한다. 교수자·학생 완료 화면은 같은 player 구조와 접근 거부 상태를 제공하지만 실제 허용 역할은 확정하지 않는다.
- 내부 녹음 storage key, 물리 경로와 provider 원문 오류는 DOM·오류·log에 노출하지 않는다.
- 실제 API와 연결되지 않은 Prototype 데이터에는 `Static prototype`을 표시한다.
- 로딩, 빈 상태, 부분 오류, 전체 오류와 권한 없음 상태를 정상 화면과 함께 설계한다.
- AI 기능 실패가 Course 탐색, Transcript와 질문 같은 핵심 기능을 막지 않게 한다.
- 임시 partial Transcript와 DB에 저장된 final Transcript를 시각적으로 구분한다.
- 완료 기록은 compact `/record` manifest를 먼저 읽고 대형 목록을 영역별로 점진 로딩한다. 한 영역의 로딩·오류·재시도로 다른 영역을 가리지 않는다.
- AI Evidence는 API가 제공한 `source_kind`·사용자용 `label`·권한 검사 공개 `link`만 DOM에 표시하고 내부 KnowledgeChunk·storage 정보로 이동 경로를 조합하지 않는다. detached Material 또는 폐기된 AI 대표질문의 기존 Evidence는 `label`만 유지하고 `link=null`인 이동 control을 비활성화한다.
- 모바일에서도 핵심 기능을 제거하지 않고 재배치하거나 접을 수 있게 한다.

### 1.2 공통 접근성·반응형

- 키보드 사용자를 위한 본문 건너뛰기 링크와 명확한 focus 표시를 제공한다.
- 입력에는 시각적 label을 연결하고 상태 메시지는 `aria-live` 또는 `role="alert"`로 알린다.
- Dialog는 Escape 종료, focus 순환과 열기 전 요소로의 focus 복귀를 지원한다.
- 동작 가능한 요소는 원칙적으로 최소 44px 높이를 확보한다.
- `prefers-reduced-motion`에서는 장식 animation과 smooth scroll을 최소화한다.
- 기준 viewport는 데스크톱 1440px, 태블릿 768px, 모바일 375px이다.

## 3. 서비스 진입

### 3.1 `MAIN_PAGE` · 비로그인 메인

#### 기본 정보

| 항목          | 내용                                                           |
| ------------- | -------------------------------------------------------------- |
| 목적          | GOAL의 핵심 가치를 설명하고 Google 로그인 흐름으로 진입시킨다. |
| 역할          | 비로그인 사용자                                                |
| 진입          | 비인증 상태로 서비스 URL 접속 또는 로그아웃 완료               |
| 이탈          | `LOGIN_PAGE`, 인증된 세션 복원 시 `MAIN_PAGE_AUTH`             |
| 기능·IA       | `MAIN_PAGE`, `LOGIN_BUTTON_AREA`                               |
| 실제 API 계약 | `GET /api/v1/auth/google/start`                                |
| Prototype     | [main.html](../prototypes/main.html)                           |

#### 화면 구성

- Header: GOAL 브랜드, 주요 기능·미리보기·Prototype 목록, Google 로그인 CTA
- Hero: 서비스 가치, 실시간 Transcript·익명 질문·AI 따라잡기 소개
- 실시간 class 미리보기: final 형태의 누적 문장과 임시 partial 문장, STT 연결 상태
- AI 미리보기: 현재 강의 요약과 익명 질문 작성 도움
- 핵심 가치: 수업 중 질문, 놓친 맥락 복구, 수업 후 복습
- 검토 도구: OAuth 기본·진행·오류·반환 성공 상태 전환

#### 상태

| 상태             | 표시                              | 가능한 동작                           |
| ---------------- | --------------------------------- | ------------------------------------- |
| 정상             | 가치 제안과 로그인 CTA            | 로그인 흐름 시작, 페이지 내 기능 탐색 |
| 세션 확인 중     | 향후 앱 초기화 skeleton·진행 상태 | 완료 대기                             |
| 로그인 진입 실패 | 인증 시작 실패 안내               | 다시 시도                             |
| 이미 인증됨      | 대시보드로 이동한다는 안내        | `MAIN_PAGE_AUTH` 이동                 |

정적 Prototype은 세션 확인 네트워크 요청을 만들지 않으며 로그인 상태 검토 도구로 화면만 전환한다.

#### 반응형·접근성

- 태블릿에서 Hero와 강의 미리보기를 세로로 배치한다.
- 모바일에서 Transcript 아래에 AI 요약과 질문 미리보기를 유지한다.
- 주요 기능 링크와 로그인 CTA는 키보드로 접근할 수 있다.
- 인식 중 문장에는 저장되지 않은 partial임을 텍스트로 함께 표시한다.

### 3.2 `LOGIN_PAGE` · Google 로그인 인계 상태

별도의 Google 화면을 복제하지 않고 `main.html`의 상태 영역에서 서비스가 책임지는 인계·반환 상태만 표현한다.

#### 기본 정보

| 항목           | 내용                                                                |
| -------------- | ------------------------------------------------------------------- |
| 목적           | OAuth 인계, callback 확인과 오류 복구를 사용자에게 설명한다.        |
| 역할           | 비로그인 사용자                                                     |
| 진입           | `MAIN_PAGE`의 Google 로그인 CTA                                     |
| 이탈           | 성공 시 `MAIN_PAGE_AUTH`, 취소·실패 시 `MAIN_PAGE`                  |
| 실제 API 계약  | `GET /api/v1/auth/google/start`, `GET /api/v1/auth/google/callback` |
| Prototype 상태 | `main.html?auth=default`, `loading`, `error`, `success`             |

#### 상태와 동작

| 상태      | 표시                                   | 가능한 동작         |
| --------- | -------------------------------------- | ------------------- |
| `default` | Google 로그인 CTA와 Course별 역할 안내 | 로그인 시작         |
| `loading` | Google 이동 중·callback 대기 안내      | 성공·실패 모의 전환 |
| `error`   | 취소 또는 제공자 오류                  | 다시 시도           |
| `success` | callback 확인 완료                     | 대시보드 이동       |

- 진행 중에는 중복 로그인 요청을 만들지 않는다.
- 실제 인증 provider 응답과 token을 화면이나 log에 표시하지 않는다.
- 오류 메시지는 계정 존재 여부 등 민감한 내부 상태를 노출하지 않는다.
- 정적 Prototype에서는 실제 OAuth·Cookie·네트워크 요청을 사용하지 않는다.

#### 미정 사항

- 실제 앱에서 초기 server Session 복원 중 사용할 skeleton 범위
- OAuth provider 장애와 사용자 취소를 서로 다른 문구로 구분할지 여부

### 3.3 `MAIN_PAGE_AUTH` · 로그인 후 대시보드

#### 기본 정보

| 항목          | 내용                                                                             |
| ------------- | -------------------------------------------------------------------------------- |
| 목적          | 사용자가 Course별 역할과 현재 class 상태를 파악하고 다음 작업으로 진입하게 한다. |
| 역할          | 인증된 사용자 공통, 각 카드 안에서 Course별 역할 표시                            |
| 진입          | Google callback 성공, 인증 Session이 남은 상태로 재접속                          |
| 이탈          | Course 생성·참여, `MY_INFO_PAGE`, 역할별 Course·class                            |
| 기능          | `PRE-T-01`, `PRE-S-01~03`, `POST-07`, `SYS-01`, `SYS-03`                         |
| 실제 API 계약 | `GET /api/v1/me`, `GET /api/v1/courses`                                          |
| Prototype     | [dashboard.html](../prototypes/dashboard.html)                                   |

#### 화면 구성

- 공통 Header: 홈, Course, 완료 강의 기록, 알림, Profile disclosure
- 인사 영역: 현재 시각에 맞는 인사와 날짜
- 현재 active class: Course, class 제목, `READY`·`LIVE`·`PROCESSING` 상태와 역할별 진입 동작. Course마다 최대 한 건만 표시
- 빠른 실행: Course 생성과 참여 진입
- 관리 중인 Course: 해당 Course에서 `PROFESSOR`인 항목
- 참여 중인 Course: 해당 Course에서 `STUDENT`인 항목
- 상태 검토 도구: 두 목록의 상태를 서로 독립적으로 전환

#### Course 카드 표시

| 데이터           | 표시 규칙                                                               |
| ---------------- | ----------------------------------------------------------------------- |
| Course 이름·학기 | 모든 카드에 표시                                                        |
| 역할             | `교수자` 또는 `학생`, 계정 전역 역할처럼 표시하지 않음                  |
| 현재 class       | `current_session`의 `시작 전`, `진행 중`, `정리 중` 또는 없음을 표시    |
| 참여 코드        | 관리 중인 Course 카드에만 `[A-Z]{6}` 코드 표시, 학생 카드에는 표시 금지 |
| 완료 기록        | 학생 Course의 최근 완료 기록과 개수 표시                                |

#### 목록 상태

관리 목록과 참여 목록은 별도의 요청 결과로 간주해 한쪽 오류가 다른 쪽을 가리지 않게 한다.

| 상태      | 표시                             | 가능한 동작            |
| --------- | -------------------------------- | ---------------------- |
| `normal`  | 역할별 Course 카드               | Course 진입, 목록 탐색 |
| `loading` | 카드 형태 skeleton과 `aria-busy` | 완료 대기              |
| `empty`   | 역할별 안내와 생성 또는 참여 CTA | 해당 흐름 진입         |
| `error`   | 실패한 목록만 오류 표시          | 해당 목록만 다시 시도  |
| 인증 만료 | Course 내용을 숨기고 로그인 안내 | `MAIN_PAGE` 이동       |

Prototype은 `dashboard.html?owned=normal`, `loading`, `empty`, `error`와 `joined` query를 조합해 상태를 재현한다. 인증 만료는 `dashboard.html?auth=expired`로 확인한다.

#### 동작과 오류

- 정적 Prototype은 Course 생성·참여, 교수자 LIVE class와 완료 기록 화면으로 실제 로컬 링크를 제공하고, owner 카드의 참여 코드는 `[A-Z]{6}` 안전한 fixture만 사용한다. API·OAuth·WebSocket은 호출하지 않는다.
- Profile disclosure는 click, 바깥 영역 click과 Escape로 닫을 수 있다.
- 인증 만료 상태에서는 Course 이름이나 참여 코드 목 데이터도 표시하지 않는다.
- 실제 앱에서는 401 응답 시 비로그인 메인으로 이동하고 다시 로그인을 요청한다.

#### 반응형·접근성

- 940px 이하에서 상단 중앙 nav 대신 가로 스크롤 가능한 모바일 nav를 제공한다.
- 태블릿에서 진행 중 class와 빠른 실행을 세로로 배치한다.
- 모바일에서 Course 카드를 한 열로 배치하고 진행 중 class의 메타정보와 CTA를 세로로 정렬한다.
- Profile은 복잡한 ARIA menu가 아닌 disclosure navigation으로 구현한다.
- 목록 로딩 메시지는 시각적으로 숨겨도 스크린 리더에 전달한다.

### 3.4 `MY_INFO_PAGE` · 내 정보

#### 기본 정보

| 항목          | 내용                                                                |
| ------------- | ------------------------------------------------------------------- |
| 목적          | 인증 계정의 기본 정보와 Course별 역할 요약을 확인하고 로그아웃한다. |
| 역할          | 인증된 사용자 공통                                                  |
| 진입          | 대시보드의 Profile disclosure에서 내 정보 선택                      |
| 이탈          | 대시보드, 역할별 Course, 로그아웃 후 `MAIN_PAGE`                    |
| 기능          | `SYS-01`, `SYS-03`, 인증·로그아웃                                   |
| 실제 API 계약 | `GET /api/v1/me`, `GET /api/v1/courses`, `POST /api/v1/auth/logout` |
| Prototype     | [my-info.html](../prototypes/my-info.html)                          |

#### 화면 구성

- 계정 기본 정보: avatar, 표시 이름, Google 계정 이메일과 연결 상태
- 계정 상세: 로그인 방식, 가입일과 고정 계정 역할이 없다는 안내
- Course 요약: 관리 중인 Course와 참여 중인 Course 개수·대시보드 이동
- Session 보안: 공용 기기 안내와 로그아웃
- 상태 검토 도구: 정상, 로딩, Course 없음과 조회 오류

계정 정보 수정은 IA에서 MVP 선택 기능이므로 1차 MVP 확정 동작으로 제공하지 않는다.

#### 상태

| 상태      | 표시                                    | 가능한 동작             |
| --------- | --------------------------------------- | ----------------------- |
| `normal`  | 계정 정보와 역할별 Course 요약          | 대시보드 이동, 로그아웃 |
| `loading` | 계정·Course 요약 skeleton과 `aria-busy` | 완료 대기               |
| `empty`   | 계정 정보와 Course 0개 안내             | 대시보드 이동           |
| `error`   | 민감한 계정 정보를 숨긴 오류            | 다시 시도               |
| 인증 만료 | 모든 계정 정보를 숨기고 로그인 안내     | `MAIN_PAGE` 이동        |

Prototype은 `my-info.html?state=normal`, `loading`, `empty`, `error`에 해당하는 개별 query 값으로 상태를 재현하며, 인증 만료는 `my-info.html?auth=expired`로 확인한다.

#### 로그아웃

1. 사용자가 로그아웃을 선택한다.
2. 확인 Dialog가 열리고 focus가 Dialog 안으로 이동한다.
3. 실제 앱은 `POST /api/v1/auth/logout` 성공 후에만 비로그인 화면으로 이동한다.
4. 실패하면 현재 화면과 Session을 유지하고 오류를 알린다.
5. Prototype은 성공·실패 결과를 로컬 이동과 Toast로 모의한다.

#### 반응형·접근성

- 태블릿 이하에서 계정 카드와 Course 요약을 한 열로 배치한다.
- 모바일에서는 계정 avatar와 텍스트를 세로로 배치한다.
- 로그아웃 Dialog는 Escape, focus trap과 trigger focus 복귀를 지원한다.
- 이름·이메일 조회 실패 시 이전 목 데이터를 남겨 성공처럼 보이지 않게 한다.

#### 미정 사항

- MVP에서 사용자가 변경할 수 있는 계정 정보의 범위
- 계정 탈퇴 및 Course 데이터 처리 정책

## 4. Course 생성과 참여

### 4.1 `COURSE_CREATE_PAGE` · Course 생성

#### 기본 정보

| 항목          | 내용                                                                          |
| ------------- | ----------------------------------------------------------------------------- |
| 목적          | 과목명과 표시용 학기를 입력해 Course와 유일한 교수자 owner 멤버십을 생성한다. |
| 역할          | 인증 사용자, 생성된 Course에서는 `PROFESSOR`                                  |
| 진입          | 대시보드의 Course 만들기 동작                                                 |
| 이탈          | 취소 시 `MAIN_PAGE_AUTH`, 성공 시 `COURSE_PAGE_PROF`                          |
| 기능·IA       | `PRE-T-01`, `PRE-T-02`, `COURSE_CREATE_PAGE`                                  |
| 실제 API 계약 | `POST /api/v1/courses`                                                        |
| Prototype     | [course-create.html](../prototypes/course-create.html)                        |

#### 화면 구성과 입력

- 과목명과 학기는 공백이 아닌 문자열을 필수로 입력한다.
- 학기는 현재 API 계약처럼 구조화하지 않은 표시용 문자열로 받는다.
- 생성 결과에는 현재 사용자가 이 Course의 유일한 `교수자` owner임을 표시한다.
- 서버가 발급하는 무기한 `[A-Z]{6}` 참여 코드는 성공 화면에서만 보여주며 owner 전용 정보임을 알린다.
- 문서와 Prototype 예시 코드는 숫자·구분자 없이 `ABCDEF`처럼 영문 대문자 6자를 사용한다.

#### 상태

| 상태         | 표시                                              | 가능한 동작            |
| ------------ | ------------------------------------------------- | ---------------------- |
| `form`       | Course 입력 폼과 생성 후 동작 안내                | 입력, 취소, 생성       |
| `submitting` | Course·교수자 멤버십 생성 진행 상태               | 완료 대기              |
| `success`    | Course 정보, 유일한 교수자 owner 역할과 참여 코드 | 코드 복사, Course 이동 |
| `error`      | 입력은 유지한다는 오류 안내                       | 입력 화면에서 재시도   |
| 인증 만료    | 입력과 생성 결과를 숨긴 안내                      | 로그인 화면 이동       |

Prototype은 `course-create.html?state=form`, `submitting`, `success`, `error`, `expired`로 상태를 재현하고 실제 네트워크 요청이나 저장을 만들지 않는다. 로그인 직후 누구나 생성할 수 있다는 공통 진입, 성공한 Course에서만 owner 역할이 생긴다는 경계와 무기한 `[A-Z]{6}` 코드·owner 전용 후속 회전을 안내한다.

#### 검증·접근성·반응형

- 공백만 입력하면 필드별 오류를 연결하고 첫 오류 입력으로 focus를 이동한다.
- API에 정의되지 않은 최대 길이를 클라이언트에서 임의로 추가하지 않는다.
- 생성 중에는 실제 구현에서 제출 버튼을 비활성화하고 멱등 키로 중복 생성을 방지한다.
- 결과 참여 코드 복사 성공·실패를 `aria-live` Toast로 알린다.
- 태블릿 이하에서는 입력 폼과 안내를 한 열로 재배치하고 모바일에서는 결과 동작을 세로로 배치한다.

#### 미정 사항

- 과목명과 학기의 최대 길이
- 학기를 연도·계절처럼 구조화할지 여부

### 4.2 `COURSE_JOIN_PAGE` · Course 참여

#### 기본 정보

| 항목          | 내용                                                                  |
| ------------- | --------------------------------------------------------------------- |
| 목적          | 교수자에게 받은 참여 코드를 검증해 Course의 학생 멤버십을 만든다.     |
| 역할          | 인증 사용자, 참여한 Course에서는 `STUDENT`                            |
| 진입          | 대시보드의 코드로 참여하기 동작                                       |
| 이탈          | 취소 시 `MAIN_PAGE_AUTH`, 성공·기존 멤버십 확인 시 `COURSE_PAGE_STUD` |
| 기능·IA       | `PRE-S-01`, `COURSE_JOIN_PAGE`, `SYS-01`, `SYS-03`                    |
| 실제 API 계약 | `POST /api/v1/courses/join`                                           |
| Prototype     | [course-join.html](../prototypes/course-join.html)                    |

#### 입력과 역할 규칙

- 참여 코드는 앞뒤 공백을 제거하고 영문을 대문자로 정규화한 뒤 `[A-Z]{6}`인지 검증한다.
- 숫자와 공백을 포함한 구분자를 허용하지 않으며 입력에는 `maxlength="6"`과 영문 대문자 6자 안내를 제공한다.
- 참여 코드는 자동 만료되지 않고 owner가 회전한 이전 코드는 즉시 무효가 된다.
- 새 참여는 학생 멤버십을 만들고, 기존 학생의 재요청은 같은 멤버십을 반환한다.
- 같은 Course의 기존 교수자 역할은 학생으로 덮어쓰지 않는다.
- 학생 성공 화면과 Course 화면에는 참여 코드를 다시 표시하지 않는다.

#### 상태

| 상태       | 표시                                          | 가능한 동작               |
| ---------- | --------------------------------------------- | ------------------------- |
| `form`     | 참여 코드 입력과 Course별 역할 안내           | 입력, 취소, 참여 요청     |
| `checking` | 코드와 접근 가능 여부 확인 중                 | 완료 대기                 |
| `success`  | Course 이름과 학생 역할                       | 대시보드·학생 Course 이동 |
| `already`  | 기존 학생 멤버십 사용 안내                    | 학생 Course 이동          |
| `invalid`  | 코드 존재·회전 여부를 구분하지 않는 공통 안내 | 다시 입력                 |
| `conflict` | 기존 교수자 역할을 유지한다는 안내            | 관리 Course 이동          |
| `error`    | 요청 실패와 입력 유지 안내                    | 재시도                    |
| 인증 만료  | 코드와 Course 정보를 숨긴 안내                | 로그인 화면 이동          |

Prototype은 `course-join.html?state=form`, `checking`, `success`, `already`, `invalid`, `conflict`, `error`, `expired`로 상태를 재현하며 실제 코드 검증 요청을 만들지 않는다.

#### 검증·접근성·반응형

- 공백뿐인 입력은 필드 오류를 연결하고 입력으로 focus를 이동한다.
- 코드 오류는 Course 존재 여부, 회전된 이전 코드 여부와 교수자 정보를 노출하지 않는다.
- 확인 중에는 실제 구현에서 제출을 비활성화하고 같은 요청의 중복 멤버십 생성을 막는다.
- 상태 전환은 `aria-live` 영역과 Toast로 전달한다.
- 태블릿 이하에서는 코드 입력과 안내를 한 열로, 모바일에서는 결과 동작을 세로로 배치한다.

### 4.3 `COURSE_PAGE_PROF` · 교수자 Course 상세

#### 기본 정보

| 항목          | 내용                                                                                             |
| ------------- | ------------------------------------------------------------------------------------------------ |
| 목적          | Course 정보·참여 코드, 단일 active class와 완료 기록을 유일한 owner 권한으로 관리한다.           |
| 역할          | 해당 Course를 생성한 유일한 `PROFESSOR` owner                                                    |
| 진입          | 관리 Course 선택 또는 Course 생성 성공                                                           |
| 이탈          | class 생성·준비, 교수자 실시간 class, 교수자 완료 class                                          |
| 기능·IA       | `COURSE_PAGE_PROF`, `COURSE_CODE_AREA`, `COURSE_DELETE`, `PRE-T-02~05`, `MGT-T-01~04`, `POST-07` |
| 실제 API 계약 | Course 조회·코드 회전·삭제와 class 목록·제목 수정·삭제 API                                       |
| Prototype     | [course-professor.html](../prototypes/course-professor.html)                                     |

#### 화면 구성과 권한

- Course 이름·학기와 이 Course의 유일한 `교수자` owner 역할을 표시하며 owner 추가·이전 동작은 제공하지 않는다.
- 참여 코드는 owner 권한을 확인한 정상 화면에서만 표시·복사·회전할 수 있다.
- 현재 class와 완료 class 목록은 서로 독립적으로 조회해 한쪽 오류가 다른 영역을 가리지 않는다.
- 현재 class는 `시작 전(READY)`, `진행 중(LIVE)`, `정리 중(PROCESSING)`, 없음으로 구분한다.
- Course에는 active class가 최대 하나이므로 `READY`, `LIVE`, `PROCESSING` 중에는 새 class 동작을 비활성화한다. `current_session`은 그 단일 class 또는 `null`이다.
- 완료 class는 `started_at` 내림차순으로 정렬하고 날짜와 실제 시작 시각을 함께 표시해 같은 날짜의 class를 구분한다.

#### 참여 코드 회전과 Course 삭제

- 참여 코드 회전은 owner만 실행한다. 확인 Dialog에 이전 코드가 즉시 무효화됨을 알리고, 성공하면 새 `[A-Z]{6}` 코드로 교체하며 실패하면 기존 코드를 유지한다.
- 참여 코드는 만료되지 않고 구분자를 사용하지 않으며 회전 감사·이력을 보관하거나 화면에 제공하지 않는다.
- Course에는 종료·보관 상태와 owner 이전이 없다. owner에게 삭제 확인 동작을 제공하고 성공하면 대시보드로 이동한다.
- active class가 있는 Course의 삭제 허용 여부와 삭제 후 복구 정책은 미정이다. 정책 확정 전에는 active class가 있는 경우 삭제를 실행하지 않고 안내한다.

#### 전체·부분 상태

| 범위       | 상태                                  | 표시와 동작                                                    |
| ---------- | ------------------------------------- | -------------------------------------------------------------- |
| Course     | `normal`, `loading`, `error`          | 정상 상세, 전체 skeleton, 이전 데이터를 숨긴 재시도            |
| Course     | `forbidden`, 인증 만료                | Course·참여 코드·class 정보를 모두 숨기고 안전한 화면으로 이동 |
| 현재 class | `live`, `ready`, `processing`, `none` | 단일 `current_session`의 입장·준비·정리 상태 또는 없음         |
| 현재 class | `loading`, `error`                    | 이 영역만 대기하거나 재시도하고 참여 코드·기록 유지            |
| 완료 기록  | `normal`, `loading`, `empty`, `error` | 목록 탐색, 빈 안내 또는 이 목록만 재시도                       |

Prototype은 `course-professor.html?view=normal`, `loading`, `error`, `forbidden`, `expired`를 지원한다. 정상 화면에서는 `session=live|ready|processing|none|loading|error`, `action=limited|available`과 `history=normal|loading|empty|error`를 조합한다.

#### 보안·접근성·반응형

- 권한 없음·인증 만료·전체 오류에는 참여 코드나 이전 Course 목 데이터를 남기지 않는다.
- 참여 코드는 영문 대문자 6자로 표시하고 복사·회전 결과를 `aria-live`로 알린다.
- 각 상태 전환기는 대상 영역과 연결되고 로딩 영역에는 `aria-busy`와 숨김 설명을 제공한다.
- 태블릿에서는 Course 소개·코드, 현재 class·관리 영역을 각각 한 열로 바꾼다.
- 모바일에서는 class 메타와 동작, 기록 행을 세로 중심으로 재배치한다.

#### 미정 사항

- active class가 있는 Course의 삭제 허용 여부와 삭제 후 복구 정책
- 참여자 수를 Course 상세 API에 포함할지 별도 API로 조회할지 여부

### 4.4 `COURSE_PAGE_STUD` · 학생 Course 상세

#### 기본 정보

| 항목          | 내용                                                                          |
| ------------- | ----------------------------------------------------------------------------- |
| 목적          | 학생이 현재 class 상태를 확인·입장하고 날짜별 완료 기록으로 이동한다.         |
| 역할          | 해당 Course의 `STUDENT`                                                       |
| 진입          | 참여 Course 선택 또는 코드 참여 성공                                          |
| 이탈          | 학생 실시간 class, class 정리 중, 학생 완료 class·복습 AI                     |
| 기능·IA       | `COURSE_PAGE_STUD`, `PRE-S-02~03`, `POST-07`, `SYS-01`, `SYS-03`              |
| 실제 API 계약 | `GET /api/v1/courses/{course_id}`, `GET /api/v1/courses/{course_id}/sessions` |
| Prototype     | [course-student.html](../prototypes/course-student.html)                      |

#### 화면 구성과 권한

- Course 이름·학기와 이 Course의 `학생` 역할을 표시한다.
- 학생 화면에는 참여 코드와 class 생성·시작·종료 같은 교수자 전용 정보를 표시하지 않는다.
- 단일 `current_session`이 `LIVE`이면 실시간 화면 입장을, `READY`이면 교수자 시작 대기를, `PROCESSING`이면 정리 상태 확인을 제공한다.
- 현재 class가 없으면 교수자가 새 class를 시작할 때까지 기다린다는 빈 상태를 제공한다.
- 완료 class는 `started_at` 내림차순으로 정렬하고 날짜와 실제 시작 시각을 함께 표시해 같은 날짜 기록을 구분하며 완료 기록 화면으로 진입시킨다.
- 현재 class와 완료 목록을 독립적으로 조회해 한쪽 오류가 다른 영역을 가리지 않는다.

#### 전체·부분 상태

| 범위       | 상태                                  | 표시와 동작                                          |
| ---------- | ------------------------------------- | ---------------------------------------------------- |
| Course     | `normal`, `loading`, `error`          | 정상 상세, 전체 skeleton, 이전 데이터를 숨긴 재시도  |
| Course     | `forbidden`, 인증 만료                | Course·class·기록을 모두 숨기고 안전한 화면으로 이동 |
| 현재 class | `live`, `ready`, `processing`, `none` | 학생 입장, 시작 대기, 정리 확인 또는 빈 안내         |
| 현재 class | `loading`, `error`                    | 이 영역만 대기·재시도하고 완료 기록 유지             |
| 완료 기록  | `normal`, `loading`, `empty`, `error` | 기록 탐색, 빈 안내 또는 이 목록만 재시도             |

Prototype은 `course-student.html?view=normal`, `loading`, `error`, `forbidden`, `expired`를 지원한다. 정상 화면에서는 `session=live|ready|processing|none|loading|error`와 `history=normal|loading|empty|error`를 조합한다.

#### 접근성·반응형

- 권한 없음·인증 만료·전체 오류에는 이전 Course나 class 목 데이터를 남기지 않는다.
- class 상태는 색만이 아니라 `진행 중`, `시작 전`, `정리 중` 텍스트로 전달한다.
- 로딩 영역에는 `aria-busy`와 스크린 리더용 설명을 제공하고 부분 오류는 해당 영역만 `role="alert"`로 알린다.
- 태블릿에서는 Course 소개와 학생 기능 안내, 현재 class와 이용 안내를 각각 한 열로 바꾼다.
- 모바일에서는 class 메타·동작과 완료 기록 행을 세로로 재배치한다.

#### 미정 사항

- 학생용 Course 알림의 종류와 브라우저 알림 권한 범위

## 5. class 생성과 수업 준비

### 5.1 `CLASS_CREATE_PAGE` · class 생성·선택적 복수 PDF 준비

#### 기본 정보

| 항목      | 내용                                                                                                     |
| --------- | -------------------------------------------------------------------------------------------------------- |
| 목적      | Course 교수자가 `READY` class를 생성하고 선택적으로 여러 PDF를 관리하며 시작 가능 상태를 확인한다.       |
| 역할      | 해당 Course의 `PROFESSOR`                                                                                |
| 진입      | 교수자 Course에서 새 class 만들기 또는 `READY` class 준비 선택                                           |
| 이탈      | 취소 시 `COURSE_PAGE_PROF`, 시작 성공 시 `LIVE_CLASS_PAGE_PROF`                                          |
| 기능·IA   | `CLASS_CREATE_PAGE`, `CLASS_MATERIAL_UPLOAD`, `CLASS_MATERIAL_DETACH`, `LIVE_CLASS_START`, `PRE-T-03~05` |
| Prototype | [class-create.html](../prototypes/class-create.html)                                                     |

#### 실제 API 순서

1. `GET /api/v1/courses/{course_id}`로 owner 역할과 `current_session`을 확인한다. active class가 있으면 새 class를 만들지 않는다.
2. `POST /api/v1/courses/{course_id}/sessions`에 선택 제목과 필수 `lecture_date`를 보내 `READY` Session과 `session_id`를 얻는다. 제목을 생략하거나 trim 결과가 비면 Course 제목·class 날짜·시각을 포함한 서버 자동 제목을 사용한다.
3. 재진입 시 `GET /api/v1/sessions/{session_id}`로 상태를 확인하고 `LIVE`, `PROCESSING`, `COMPLETED`이면 해당 화면으로 이동한다.
4. PDF는 선택 사항이다. 업로드하면 `POST /api/v1/sessions/{session_id}/materials`를 파일 하나당 한 번 호출한다.
5. `202 Accepted` 응답의 Material과 `MATERIAL_PROCESSING` Job을 기준으로 자료 목록·Material·Job 조회를 갱신한다.
6. 실패 Job은 `POST /api/v1/jobs/{job_id}/retry`로 같은 행의 `attempt`를 증가시킨다.
7. 삭제는 `DELETE /api/v1/materials/{material_id}` 성공 뒤 목록을 다시 조회해 detach 결과를 확정한다.
8. attached Material 중 `PROCESSING`이 없을 때 `POST /api/v1/sessions/{session_id}/start`로 `READY → LIVE`를 요청한다.

Prototype은 위 요청을 실행하지 않고 목 상태와 로컬 입력만 바꾼다. `READY` 생성 화면은 선택 제목·필수 날짜, `ACTIVE_SESSION_EXISTS`, 제목 수정·자동 제목 복귀와 삭제 확인을 재현한다. READY Material은 PDF 0개 시작, active `N/10`, 파일당 100 MB 사전 검사, 표시 이름 suffix, 파일별 상태·재시도·권한 확인 원본 열람·즉시 detach 확인과 오류 상태를 재현한다. LIVE·완료 화면의 Material 관리는 각 상태 화면 동기화 단위에서 반영한다.

#### class 생성 상태

| 상태                                   | 표시와 동작                                                      |
| -------------------------------------- | ---------------------------------------------------------------- |
| `draft`                                | 선택 제목·필수 날짜 입력, Course 복귀와 생성                     |
| `creating`                             | Session 생성 진행 상태와 중복 제출 방지                          |
| `ready`                                | Session ID 확보 후 PDF 업로드와 시작 준비 제공                   |
| `create-error`                         | 입력을 다시 확인하고 생성 요청 재시도                            |
| `active-conflict`                      | `409 ACTIVE_SESSION_EXISTS`, Course의 현재 active class로 이동   |
| `live`, `processing`, `completed` 응답 | 해당 실시간·정리·완료 화면으로 이동하고 owner에게 제목 수정 제공 |

`class-create.html?stage=draft|creating|ready|create-error|active-conflict|live|processing|completed`로 확인한다.

#### PDF 선택·업로드·처리 상태

- `<input type="file" multiple accept="application/pdf,.pdf">`와 연결된 label을 제공하고 파일당 `100 MB (100,000,000 bytes)` 제한을 설명한다.
- 서버가 수락하기 전 로컬 선택 파일만 파일명 단위로 취소할 수 있다. 선택 결과가 남은 active 자리보다 많으면 서버에 보내기 전에 개수 초과를 알린다.
- 삭제되지 않고 연결된 `UPLOADED`, `PROCESSING`, `READY`, `FAILED` Material을 모두 active 개수에 포함하고 `현재 N/10개`를 표시한다.
- 동일 content의 PDF 재업로드를 허용한다. 표시 이름이 겹치면 서버 응답의 basename suffix를 사용해 `강의자료.pdf`, `강의자료 (1).pdf`, `강의자료 (2).pdf`처럼 표시하고, 삭제 후 남은 이름을 재번호화하지 않는다.
- 업로드 성공 뒤에는 서버가 확정한 표시 이름만 사용한다. 내부 storage key, 물리 경로와 cleanup 상태는 DOM·오류·log에 넣지 않는다.
- 파일별 상태는 `UPLOADED`, `PROCESSING`, `READY`, `FAILED`를 사용한다.
- `READY`만 AI 참고 가능으로 표시한다. 연결된 `UPLOADED`, `PROCESSING`, `READY` 행에는 권한 확인 원본 열람을 제공한다. `UPLOADED`는 처리 대기, `PROCESSING`은 처리 중, `FAILED`는 “업로드 실패 · 사용할 수 없음”으로 표시하고 원본 열람·AI 참고·업로드 완료 개수에 포함하지 않는다.
- 한 파일 실패가 다른 파일의 상태와 class 정보를 가리지 않으며 class 시작도 막지 않는다. 단, 실패 행은 detach 전까지 active 10개 제한에는 포함한다.
- 진행률을 계산할 수 없으면 숫자를 꾸며내지 않고 불확정 `<progress>`와 처리 단계 텍스트를 쓴다.
- 실패 Material 재시도는 새 Job처럼 표현하지 않고 같은 Job의 `attempt 1 → 2`로 갱신한다.

| Prototype 상태    | 표시                                                                           |
| ----------------- | ------------------------------------------------------------------------------ |
| `empty`           | PDF 0개도 정상 준비 상태로 표시, 파일 선택 또는 바로 시작                      |
| `uploading`       | 파일별 독립 업로드와 202 수락 구분                                             |
| `processing`      | `UPLOADED` 대기와 `PROCESSING` 진행, 시작 차단                                 |
| `mixed`           | `READY`, `PROCESSING`, `FAILED` 동시 표시와 파일별 열람·재시도·삭제, 시작 차단 |
| `all-ready`       | 복수 PDF 처리 완료와 동일 이름 suffix, 파일별 열람·삭제                        |
| `failed`          | 업로드되지 않은 것처럼 사용할 수 없음을 알리고 재시도·삭제 제공                |
| `full`            | active 10/10, 새 파일 선택 차단과 기존 자료 삭제 안내                          |
| `size-error`      | `FILE_TOO_LARGE`, 100 MB(100,000,000 bytes) 제한과 전송하지 않은 파일 표시     |
| `mime-error`      | `UNSUPPORTED_MEDIA_TYPE`, 유효한 PDF만 허용하고 전송하지 않은 파일 표시        |
| `delete-conflict` | `MATERIAL_DELETE_CONFLICT`, 기존 행 유지와 canonical 목록 재조회 안내          |
| `delete-missing`  | `MATERIAL_NOT_FOUND`, 기존 행 제거와 canonical 목록 갱신 안내                  |
| `load-error`      | class·시작 정보는 유지하고 자료 영역만 재시도                                  |

`class-create.html?stage=ready&materials=empty|uploading|processing|mixed|all-ready|failed|full|size-error|mime-error|delete-conflict|delete-missing|load-error`로 확인한다.

#### 시작 상태와 권한

- PDF가 0개여도 시작 버튼을 활성화한다. attached Material이 없거나 `READY`, `UPLOADED`, `FAILED`만 있으면 시작할 수 있다.
- attached Material 중 `PROCESSING`이 하나라도 있으면 시작 버튼을 비활성화하고 처리 완료·실패를 기다리거나 해당 Material을 삭제하라고 안내한다.
- `UPLOADED`·`FAILED`가 있어도 시작할 수 있지만 두 상태의 자료는 AI 참고에서 제외된다는 안내를 시작 전에 제공한다.
- 서버가 `MATERIAL_PROCESSING_ACTIVE`를 반환하면 최신 Material 목록을 다시 조회하고 같은 차단 사유를 표시한다.
- Course의 `READY` class는 이미 그 Course의 유일한 active class다. 다른 상태 경쟁과 잘못된 전이는 해당 API 오류에 맞춰 안내한다.
- `start=idle|starting|material-processing|conflict|error|success`로 시작 요청 상태를 재현한다.
- 일반 시작 실패에도 `READY` 정보와 PDF 결과를 유지한다.
- 인증 만료와 교수자 권한 없음에서는 class 입력, 파일명과 처리 상태를 모두 숨긴다.
- `auth=valid|expired|forbidden`으로 접근 경계를 확인한다.

#### class 제목 수정과 삭제

- owner는 `PATCH /api/v1/sessions/{session_id}`로 `READY`, `LIVE`, `PROCESSING`, `COMPLETED`의 제목만 수정할 수 있다.
- 제목을 생략하거나 trim 후 비워 저장하면 Course 제목·class 날짜·시각을 포함한 서버 자동 제목으로 되돌린다. 정확한 문자열 형식, `READY`에서 사용할 시각 원장과 timezone은 미정이다.
- `lecture_date`, `started_at`, `ended_at`, `completed_at`은 화면에서 읽기 전용이며 수정 입력을 제공하지 않는다.
- `DELETE /api/v1/sessions/{session_id}`는 `READY`와 `COMPLETED`에서만 owner에게 제공한다. 성공하면 Course 화면으로 이동하고 실패하면 현재 데이터와 상태를 유지한다.
- `LIVE`에서는 먼저 class 종료를 안내하고, `PROCESSING`에서는 삭제를 제공하지 않는다. 후처리 완료로 `COMPLETED`가 된 뒤 삭제할 수 있다.
- Course 학생에게는 제목 수정과 class 삭제 control을 렌더링하지 않는다.
- `class-create.html?stage=ready`는 READY 제목 수정·자동 제목 복귀와 삭제 확인 Dialog를 모의하고, 삭제 성공 시 `session=none`인 교수자 Course로 이동한다. 실제 API·삭제는 실행하지 않는다.

#### Material 삭제

- owner는 Session `READY`, `LIVE`, `COMPLETED`에서 attached Material의 삭제 확인 Dialog를 열 수 있다. Session `PROCESSING`에서는 삭제 control을 제공하지 않는다.
- 삭제 확인에는 자료가 목록·열람·새 AI 검색에서 즉시 사라지고 백그라운드 물리 정리를 기다리지 않는다고 설명한다.
- 성공 응답 뒤 해당 행을 즉시 제거하고 active 개수를 줄인 다음 canonical 목록을 다시 조회한다. object·Knowledge cleanup 실패는 삭제를 되돌리거나 행을 복원하지 않는다.
- 이미 detach됐거나 접근할 수 없는 응답도 기존 행을 제거하고 목록을 갱신한다. 상태·version 경합은 목록 갱신 뒤 사용자가 다시 삭제하도록 한다.
- 업로드·삭제 멱등성 replay는 새로운 행이나 중복 성공 알림을 만들지 않고 기존 결과를 재사용한다.
- `class-create.html?stage=ready`의 각 attached `UPLOADED`, `PROCESSING`, `READY` 행은 권한 확인 원본 열기 목 동작과 삭제 확인 Dialog를 제공한다. `FAILED`는 원본 열기 없이 재시도·삭제만 제공하고, 성공 삭제는 행과 active 개수를 즉시 줄인 뒤 canonical 재조회 상태를 알린다.

#### Material 오류 문구

|  HTTP | 오류 code                    | UI 책임과 사용자 문구                                                                                                  |
| ----: | ---------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `409` | `MATERIAL_PROCESSING_ACTIVE` | 시작 영역에 “처리 중인 강의자료가 있습니다. 완료·실패를 기다리거나 삭제한 뒤 시작해 주세요.”를 표시하고 최신 목록 조회 |
| `409` | `MATERIAL_LIMIT_EXCEEDED`    | 업로드 영역에 “강의자료는 최대 10개까지 연결할 수 있습니다. 기존 자료를 삭제한 뒤 다시 시도해 주세요.” 표시            |
| `409` | `SESSION_STATE_CONFLICT`     | 업로드·삭제 영역에 “class 기록 정리 중에는 강의자료를 추가하거나 삭제할 수 없습니다.” 표시 후 Session 상태 갱신        |
| `413` | `FILE_TOO_LARGE`             | 해당 파일에 “PDF 한 개는 100 MB(100,000,000 bytes) 이하여야 합니다.” 표시                                              |
| `415` | `UNSUPPORTED_MEDIA_TYPE`     | 해당 파일에 “내용과 형식이 유효한 PDF 파일만 업로드할 수 있습니다.” 표시                                               |
| `404` | `MATERIAL_NOT_FOUND`         | 해당 행을 제거하고 “이미 삭제되었거나 접근할 수 없는 자료입니다. 최신 목록을 불러왔습니다.” 표시                       |
| `409` | `MATERIAL_DELETE_CONFLICT`   | 최신 목록을 조회하고 “다른 변경과 겹쳐 삭제하지 못했습니다. 목록을 확인한 뒤 다시 시도해 주세요.” 표시                 |

내부 storage cleanup 실패나 provider 원문 오류는 표시하지 않는다. cleanup 실패는 detach 성공을 되돌리지 않는다.

#### 접근성·반응형

- 날짜 입력 오류를 설명과 연결하고 첫 오류 필드로 focus를 이동한다. 제목은 선택 입력임을 label과 설명에 표시한다.
- PDF가 아닌 파일, 파일 크기와 active 개수 오류는 해당 파일·입력 설명과 연결한 `role="alert"`로 알린다.
- 파일 목록은 semantic `ul`·`li`를 사용하고 재시도 버튼 이름에 파일명을 포함한다.
- 로딩·업로드·처리 영역에 `aria-busy`, `aria-live`, `<progress>` 또는 명시적 처리 텍스트를 제공한다.
- 태블릿 이하에서는 준비 영역과 시작·API 안내를 한 열로 바꾸고 모바일에서도 모든 파일 상태와 동작을 유지한다.

#### 미정 사항

- 자동 class 제목의 정확한 형식과 적용 시간대
- Material detach 뒤 기존 AI Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책. 안전한 `label` snapshot 유지와 `link=null`은 확정한다.

## 6. 실시간 class

교수자와 학생은 Transcript·질문·개인 AI를 같은 정보 구조와 UI로 사용한다. 교수자 화면은 이 공통 틀에 audio publisher·녹음·자료 관리·Answer·class 종료 control을 추가한다.

### 실시간 class 공통 개인 LIVE Summary와 Chat

- 개인 AI 영역은 두 역할 모두에게 같은 위치·상태·동작으로 제공한다. Summary와 Chat의 Job·Message·결과는 요청자에게만 보인다.
- Summary는 요청 시점 마지막 live final 범위를 확정한다. Segment가 1개 이상이면 `202 LIVE_SUMMARY Job → PENDING/RUNNING → GET Job polling → REST Summary`로 처리한다. 저장된 live final Segment가 0개이면 Job을 만들지 않고 `409 SUMMARY_TRANSCRIPT_NOT_READY`와 “아직 확정된 강의 내용이 없습니다. 잠시 후 다시 시도해 주세요.”를 표시한다.
- Chat 요청은 USER Message와 그 Message를 입력으로 고정한 `CHAT_RESPONSE` Job·outbox를 `202` 응답 전에 하나의 transaction으로 반드시 저장한 뒤 polling한다. 생성 중 assistant 부분 문장은 저장하지 않으며, `SUCCEEDED`한 최종 assistant Message·Evidence만 REST로 다시 조회한다. 한 Chat에는 진행 Job 하나만 허용하고 실패 retry는 같은 Job의 `attempt`를 증가시키되 동일 USER Message를 사용한다.
- Chat USER 입력은 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 1자 이상 2,000자 이하다. 공백뿐이거나 초과한 입력은 자르지 않고 `422 VALIDATION_ERROR`로 거부하며, 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 사용한다. textarea 원문은 그대로 두고 정규화 결과의 글자 수와 오류를 표시한다. 이 제한을 assistant Message에 적용하지 않는다.
- partial Transcript는 Summary·Chat 근거가 아니다. Summary는 `idle|pending|running|complete|error|retry`, Chat은 `idle|pending|evidence|no-evidence|error|retry`를 구분하며 근거 없음도 저장 완료 결과로 표시한다.
- 개인 AI는 `202 + AIJob → requester-only GET Job polling → 성공한 저장 결과 REST 조회`만 사용한다. token delta, SSE, streaming HTTP와 private WebSocket을 제공하지 않고 shared Session WebSocket에 개인 Job 상태·결과를 broadcast하지 않는다.
- PDF Evidence 검색은 현재 연결된 `READY` Material의 KnowledgeChunk로 한정한다. Evidence는 공개 `source_kind`·사용자용 `label`·권한을 다시 검사하는 안정적인 `link`만 표시하며 내부 KnowledgeChunk ID·storage key를 노출하지 않는다.
- Session이 `LIVE`에서 `PROCESSING`으로 전이하면 모든 LIVE Summary·Chat·Message·Evidence와 연결된 요청자 전용 Job이 삭제된다. `session.updated=PROCESSING`을 받는 즉시 선택한 Job·Chat ID와 화면 cache를 비우고 기록 정리 화면으로 이동한다. 이후 예전 LIVE 결과·Job 조회의 `404`를 복구 대상으로 취급하거나 완료 기록으로 승격하지 않는다.
- FINAL Summary와 `COMPLETED`의 개인 `REVIEW` Chat은 이 LIVE 정리의 영향을 받지 않는다. AI 실패 중에도 Transcript·질문·반응을 계속 사용할 수 있다.

#### class 정보 관리

- owner는 화면 Header에서 제목을 수정할 수 있다. 빈 제목 저장은 서버 자동 제목으로 되돌리고 날짜와 `started_at`은 읽기 전용으로 표시한다.
- `LIVE`에서는 삭제 동작을 제공하지 않고 class 종료를 먼저 안내한다. 종료 후 `PROCESSING` 완료로 `COMPLETED`가 되어야 삭제할 수 있다.

#### LIVE 강의자료 관리

- owner에게 attached Material 목록, `N/10` active 개수, PDF 선택과 Material별 삭제를 제공한다. 파일당 100 MB, 동일 content 허용, 표시 이름 suffix와 상태 표현은 5.1절을 그대로 사용한다.
- 업로드·삭제 뒤 REST Material 목록을 다시 조회한다. `UPLOADED`, `PROCESSING`, `READY`, `FAILED` 상태가 실시간 Transcript·질문 기능을 막지 않는다.
- 교수자·학생 개인 AI는 현재 연결된 `READY` Material만 참고한다. `UPLOADED`, `PROCESSING`, `FAILED` 행은 처리·실패 상태로만 보이고 새 AI 근거가 되지 않는다.
- 삭제 성공은 행·열람·새 AI 검색에서 즉시 반영하고 background cleanup 상태는 표시하지 않는다. 오류 code와 문구는 5.1절의 Material 오류 표를 공통으로 사용한다.
- 현재 `class-live-professor.html` Prototype에는 이 관리 영역이 없으며 후속 Prototype 동기화 대상이다.

#### 녹음 이중 경로와 단일 publisher

- 한 번의 교수자 마이크 입력을 실시간 Audio WS와 브라우저 로컬 녹음으로 동시에 분기한다. 실시간 전송은 MVP v1 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk로 표시하고 저장 녹음 codec·container를 이 값으로 추정하지 않는다.
- Session에서 첫 `audio.start`에 성공한 교수자 탭만 active publisher로 표시한다. 두 번째 publisher 시도에는 음성 전송 거부를 alert로 알리되 Transcript·질문·강의자료 등 조회 영역은 그대로 유지한다.
- active publisher가 끊기면 같은 `client_stream_id`의 재연결·resume 상태를 표시한다. 다른 탭의 lease 재획득이나 takeover control은 제공하지 않는다.
- active publisher 탭에서 브라우저 닫기나 화면 이탈을 시도하면 실시간 전송과 로컬 녹음이 중단될 수 있음을 경고한다. 브라우저 기본 이탈 경고의 문구를 임의로 보장하지 않는다.
- 실시간 전송 실패와 로컬 녹음 실패를 별도 alert로 표시한다. 한 경로의 실패가 다른 경로, 질문과 이미 저장된 final Transcript를 자동 실패 상태로 바꾸지 않는다.

#### 독립된 상태 축

다음 상태는 원인과 복구 방식이 다르므로 하나의 “연결 상태”로 합치지 않는다.

| 축                 | 목표 UI 상태                                                                              | 의미                                                                  |
| ------------------ | ----------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| Session event WS   | `connected`, `reconnecting`, `resync`, `stopped`                                          | 질문·Answer·Session 변경 알림과 resume cursor                         |
| audio publisher    | `checking`, `active`, `conflict`, `reconnecting`, `resumed`                               | 첫 성공 탭, 중복 publisher 전송 거부와 동일 `client_stream_id` resume |
| 실시간 Audio WS    | `permission`, `denied`, `unavailable`, `connecting`, `listening`, `reconnecting`, `error` | 마이크 권한과 고정 PCM chunk 전송 상태                                |
| 브라우저 로컬 녹음 | `idle`, `recording`, `finalizing`, `ready-upload`, `failed`                               | 실시간 전송과 독립된 저장 원본 준비 상태                              |
| 실시간 STT         | `listening`, `finalizing`, `finalized`, `error`                                           | 실시간 음성 인식과 final 저장 상태                                    |

위 이름은 화면 검토용 상태이며 API enum을 먼저 확정하지 않는다. 현재 Prototype은 `events`, `audio`, `stt`만 모의하고 publisher·로컬 녹음·이탈 경고가 없으며, 오래된 `DEGRADED`·gap·stop 표현을 포함한다. PR10에서 정적 상태를 교체하되 `getUserMedia`, 실제 녹음, WebSocket과 upload는 호출하지 않는다.

이 표의 실시간 STT `finalized`는 한 발화의 live final 저장을 뜻한다. 수업 후 전체 녹음 재처리의 영구 Transcript `FINALIZED`와는 다른 상태 축이며, LIVE 화면에서 두 의미를 하나의 완료 상태로 합치지 않는다.

#### Transcript 규칙

- `transcript.partial`은 `utterance_id`별 더 큰 `revision`만 같은 표시를 치환한다.
- partial에는 `인식 중`, `저장되지 않음`, `답변 후보 제외`를 함께 표시하고 `aria-live="off"`로 둔다.
- 재연결·resync 시 partial을 제거하며 replay·REST 복구 대상으로 사용하지 않는다.
- `transcript.final`은 발화 단위 live final Segment가 DB commit된 뒤에만 받고 `sequence` 순서로 누적한다. 이는 수업 후 canonical Transcript가 아니다.
- final 수신 시 같은 `utterance_id` partial을 제거하고 Answer가 CAPTURING이면 선택 경계 이후 final만 후보로 표시한다.
- 재접속 시 `GET /sessions/{id}/transcript?after_sequence=`의 저장된 live Segment가 LIVE 복구 결과다. 수업 후 canonical version 선택과 혼동하지 않는다.
- audio replay 실패와 gap 표현은 정책 확정 전까지 클라이언트가 추정하지 않는다. 서버가 공개한 확정 정보만 표시한다.
- Transcript 영역의 `normal`, `loading`, `empty`, `resync`, `error`는 질문 영역과 독립적이다.

#### 질문·클러스터·정렬

- 질문은 `reaction_count`를 질문별로 표시하고 `POPULAR` 또는 `RECENT` 정렬을 제공한다.
- Cluster는 immutable AI 대표질문을 중앙에 두고, 학생 실제 질문과 `CAPTURING` 또는 `COMPLETED` Answer로 보존된 과거 AI 대표질문을 유형이 구분된 child 노드로 연결한 마인드맵으로 표시한다.
- Cluster 목록은 `ordinal ASC`, 동률은 `id ASC`를 사용한다. child는 서버가 제공한 위치 순서를 사용하고 반응 수를 Cluster 단위로 임의 합산하지 않는다.
- 질문 commit이 성공하면 화면은 클러스터링 pending을 표시한다. `active_job_id`와 `retry_job_id`가 모두 없을 때만 시스템이 fresh Job을 자동 시작한다. active Job이 있으면 새 질문이 다음 Job 하나로 합쳐질 것임을 알리고, `active_job_id=null`, `retry_job_id`가 있으면 같은 실패 행의 `attempt + 1` scheduler 예약에 합쳐졌음을 표시한다. 사용자가 클러스터링을 직접 실행하는 control은 제공하지 않는다.
- LIVE 결과는 새 질문만 기존 Cluster에 배치하고 기존 질문을 다른 Cluster로 옮기지 않는다. 새 질문이 들어간 Cluster만 새 대표질문으로 교체하며 대표질문 생성이 끝나기 전에 Job을 완료로 표시하지 않는다.
- 같은 semantic Cluster는 LIVE generation 사이에 공개 logical ID를 유지하므로 마인드맵 node identity를 재사용한다. 새 LIVE Cluster와 FINAL full rebuild Cluster만 새 공개 ID를 사용한다.
- 교체된 과거 대표질문은 `CAPTURING` 또는 `COMPLETED` Answer가 있으면 child로 보존하고 Answer가 없으면 즉시 표시·직접 이동·새 AI 검색에서 제거한다. 기존 Evidence가 참조 중이면 내부 provenance tombstone만 유지하되 Evidence의 안전한 `label`은 남기고 `link=null`인 이동 control을 비활성화한다. 캡처 중인 대표질문은 완료·취소가 확정될 때까지 유지한다.
- 현재 generation과 마지막 클러스터링 Job의 mode·상태·attempt를 표시해 새로고침 후에도 현재 결과와 실행·실패 상태를 구분한다.
- `normal`, `unclustered`, `clustering-pending`, `clustering-running`, `clustering-retry-reserved`, `cluster-failed`, `loading`, `empty`, `error`를 독립적으로 표시한다. `clustering-retry-reserved`는 `active_job_id=null`, `retry_job_id!=null`인 시스템 재시도 예약이며 사용자가 실행하는 retry 상태가 아니다.
- 선택 시점의 일반 Cluster membership 이력은 재구성하지 않고, Answer 대상으로 보존된 과거 대표질문만 typed child로 노출한다.
- 클러스터링 실패에도 원본 질문 조회와 답변 기능을 유지한다.
- 교수자에게 학생 작성자 ID·이름·이메일을 표시하지 않는다.

#### Answer 캡처

1. 미답변 학생 질문 또는 현재 AI 대표질문 하나를 선택해 `POST /sessions/{id}/answers`를 요청한다. Cluster 전체는 Answer target이 아니다.
2. 서버 응답의 target 식별자, `target_text_snapshot`과 `capture_started_after_sequence`를 선택 결과의 진실로 사용한다.
3. 첫 새 final 전에는 `CAPTURING · waiting`, 이후에는 final만 candidate로 표시한다.
4. 완료 시 자동 범위를 확정해 Answer를 `COMPLETED`로 바꾸고 선택한 target 하나만 답변됨으로 표시한다. 대표질문에 답해도 child 질문은 미답변 상태를 유지한다.
5. `ANSWER_TRANSCRIPT_NOT_READY`이면 오류를 표시하되 CAPTURING을 유지한다.
6. 취소 시 `CAPTURING` Answer를 hard delete한다. 학생 질문과 현재 `ACTIVE` 대표질문은 미답변 표시로 되돌리고, 이미 `PRESERVED`인 과거 대표질문은 membership과 화면·직접 이동·새 AI 검색에서 제거한다. 기존 Evidence가 참조하면 내부 provenance tombstone만 유지하고 해당 Evidence는 `label`을 남긴 채 이동을 비활성화한다. `answer.deleted`로 해당 node와 Answer UI를 무효화하고 이어지는 `clustering.updated`에 따라 현재 generation·revision을 REST로 다시 조회한다. 취소 Answer 행·badge·filter는 만들지 않는다.

Session당 `CAPTURING` Answer는 하나이고 target당 Answer도 하나이므로 캡처 중에는 다른 답변 시작을 비활성화한다. partial은 후보가 아니다. `target_text_snapshot`은 선택 당시 target 문구로 표시하고 현재 Cluster 대표질문과 다르더라도 다시 join해 바꾸지 않는다. LIVE에서는 음성 캡처만 제공하고 텍스트 Answer 작성은 제공하지 않는다.

#### class 종료

- CAPTURING Answer가 있으면 종료 Dialog를 열지 않고 완료 또는 취소를 먼저 요청한다.
- 종료 요청이 확정되면 Session은 즉시 `PROCESSING`으로 전환하고 새 Audio·질문·반응·Answer 입력을 비활성화한다. 클라이언트가 Job 개수나 Audio 상태로 전이를 추정하지 않고 서버 Session을 진실로 사용한다.
- 같은 전이에서 개인 LIVE Summary·Chat·Message·Evidence와 관련 Job을 삭제한다. `session.updated=PROCESSING`을 받으면 두 역할 모두 선택한 개인 Job·Chat ID와 cache를 지우며, 교수자도 이전 결과를 완료 기록으로 넘기지 않는다.
- `PROCESSING` 전환으로 실행 중 LIVE 클러스터링을 중단하고 느린 결과를 화면에 commit하지 않는다. 종료 시점 질문 sequence 이하의 학생 실제 질문 전체와 종료 시각까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 snapshot한 FINAL Job을 자동 시작하고, 미답변 AI 대표질문은 제외한다.
- 후처리 coordinator는 LIVE에서 완료된 음성 Answer를 먼저 Job ID 없는 source 대기 상태로 두고, 재매핑·source가 terminal이 되면 공유 완료 차단 `ANSWER_ORGANIZATION` Job을 자동 생성한다. 사용자가 실행 버튼으로 시작하지 않으며, 각 Answer는 별도 상태로 표시한다.
- active publisher 탭은 로컬 녹음을 마감해 resumable upload와 원본 저장에 넘긴다. 녹음 upload 완료 뒤 HQ STT 후속 처리를 시작하며 세부 상태는 7.1절에서 표시한다.
- Dialog는 Escape, focus trap과 trigger focus 복귀를 지원한다.
- 현재 Prototype의 `audio.stop → audio.stopped` 선행 순서와 기록 정리 중 이동은 목표 계약이 아니다. 종료·녹음 handoff와 PR10 정적 상태를 별도 동기화한다.

#### 전체·부분 상태와 API/WS

- 전체 `view=normal|loading|error|forbidden|expired|wrong-state`를 제공하고 권한·인증 오류에는 수업 데이터를 숨긴다.
- Session events ticket과 Audio ticket은 각각 `SESSION_EVENTS_READ`, `SESSION_AUDIO_WRITE` scope의 별도 1회용 티켓이다.
- Audio 연결은 first-success publisher 제약을 적용하고 같은 `client_stream_id` resume만 허용한다. lease·takeover 방식은 화면에서 추정하지 않는다.
- Session WS는 `transcript.*`, `question.*`, `reaction.updated`, `clustering.updated`, `answer.updated`, `answer.deleted`, `job.updated`, `session.updated` 알림을 전달한다.
- REST `GET /sessions/{id}`, `/transcript`, `/questions`, `/answers`, `/sessions/{id}/question-clusters`, `/sessions/{id}/question-clusters/{cluster_id}/members`가 재연결·resync의 최종 진실이다.
- Answer REST 응답은 교수자 text, AI 정리 결과, immutable 원본 LIVE 범위와 선택적 HQ 재매핑 범위를 서로 다른 개념으로 제공한다. 화면은 하나의 답변 text나 범위로 병합하거나 서로 덮어쓰지 않는다.
- 답변 시작·완료·취소 API와 멱등 종료 API를 화면 동작에 매핑한다.
- 참여자 수 API가 없으므로 실제 수치를 표시하지 않는다.
- 개인 Summary·Chat의 상태와 결과는 Session WS의 `job.updated`로 복구하지 않고 요청자 전용 Job polling·REST 결과만 사용한다.

#### 접근성·반응형

- 중요한 final·연결·Answer 변경은 별도 `aria-live="polite"` 영역으로 알린다.
- publisher 충돌, 로컬 녹음 실패와 이탈 경고는 원인과 남아 있는 기능을 함께 설명한다. 충돌 탭의 조회 영역으로 focus를 강제 이동하지 않는다.
- Cluster disclosure는 `aria-expanded`와 `aria-controls`를 동기화한다.
- 정렬 버튼은 `aria-pressed`, Answer bar는 `tabindex="-1"` focus와 후보 설명을 제공한다.
- 데스크톱은 Transcript `1.35fr`, 질문 `380~400px`, active Answer 전체 폭으로 배치한다.
- 900px 이하에서는 Transcript·질문·Answer를 한 열로 배치하며 768px과 375px에서도 질문·상태를 숨기지 않는다.
- reduced-motion에서는 skeleton·Toast·scroll animation을 최소화한다.

#### 미정 사항

- 저장 녹음 codec·container와 브라우저 local storage 방식
- publisher lease·재획득·takeover 정책
- `DEGRADED` 전환, stop timeout·강제 종료, audio replay와 live gap 생성 기준

- 학생은 `SESSION_EVENTS_READ` 1회용 ticket과 Session event WS만 사용하며 Audio WS·마이크 UI를 제공하지 않는다.

- 전체 `view=normal|loading|error|forbidden|expired|wrong-role|processing|completed`와 영역별 상태를 독립 제공한다.

#### 익명 질문·반응

- 질문 textarea에 label·익명성 안내·글자 수를 제공하고, 앞뒤 공백 제거와 Unicode NFC 정규화 후 Unicode code point를 세어 1자 이상 300자 이하만 전송한다. 공백뿐이거나 초과한 입력의 `422 VALIDATION_ERROR` 뒤에도 원문과 정규화 결과 글자 수를 유지한다.
- 빈 입력과 300자 초과는 오류를 입력에 연결하고 focus한다. 초과 문자를 조용히 잘라서 전송하지 않고, 같은 문구의 다른 학생 질문은 허용한다.
- 전송 중 중복 제출을 비활성화하고 성공 후에만 입력을 비운다. 실패하면 원문을 유지한다.
- 질문 commit 후에는 클러스터링 pending watermark가 자동 갱신된다. `active_job_id`·`retry_job_id`가 모두 없으면 fresh Job을 시작하고, active Job이나 같은 실패 행의 retry 예약이 있으면 새 질문을 다음 실행에 합친다. 화면은 직접 실행 버튼 대신 pending·running·failed·retry-reserved와 마지막 Job의 mode·상태·attempt를 표시한다.
- 질문 저장과 클러스터링은 독립이므로 클러스터 실패에도 질문·반응을 유지한다. 마인드맵 중앙에는 AI 대표질문, child에는 학생 질문과 `CAPTURING`·`COMPLETED` Answer로 보존된 과거 AI 대표질문의 유형을 구분한다.
- 반응 PUT·DELETE는 `aria-pressed`와 count를 함께 낙관 갱신하고 실패 시 `previousPressed`와 이전 count로 정확히 rollback한다.
- `reaction.updated`의 count는 개인 `reacted_by_me` 상태를 덮지 않는다.
- API에 `is_mine`·`can_react`가 없어 내 질문 badge나 재접속 후 사전 비활성화를 계약으로 만들지 않는다. 서버 거부 후 rollback만 모의한다.

#### AI 질문 문장 작성 도움

- `idle → requesting → suggestions → editing`으로 진행하며 공백뿐인 초안·500자 초과·실패 상태를 제공한다. 제한은 앞뒤 공백 제거와 Unicode NFC 정규화 후 Unicode code point로 세며 초과 문자를 자르지 않고 `422 VALIDATION_ERROR`를 표시한다.
- 후보는 radio fieldset으로 선택하고 editable textarea에서 학생이 수정한 뒤 별도 질문 등록을 실행한다.

## 7. class 기록 정리 중

### 7.1 `CLASS_PROCESSING_STATE`

- 목적: Session `PROCESSING` 동안 녹음 resumable upload, 전체 녹음 HQ STT, 기존 Answer 시간 범위 재매핑 검증, canonical Transcript 전환, 완료된 음성 Answer별 AI 정리, FINAL 질문 재클러스터링과 독립 AI 후처리 상태를 이미 저장된 기록과 함께 제공한다.
- 역할: Course 교수자·학생. `class-processing.html?role=professor|student`로 권한별 CTA와 retry를 확인한다.
- `GET /sessions/{id}/record`는 기록 요약·각 처리 상태·영역별 개수·공개 조회 경로만 제공하는 compact manifest다. Material·Transcript·질문·Answer·Cluster·공유 Job 전체 배열은 embed하지 않고 각 경로로 독립 조회하며, Transcript는 서버가 지정한 canonical version과 영구 상태를 진실로 사용한다.
- manifest 로딩 성공 후 기록이 있는 영역을 각각 로딩한다. 한 영역의 오류와 재시도는 다른 영역의 저장된 기록·상태를 유지하고, 재시도 중에도 전체 화면 spinner로 되돌리지 않는다.
- `/record`의 상태·개수는 응답 시점의 값이며 전체 목록 snapshot token이 아니다. `PROCESSING`에서 polling한 Job·Session 상태가 바뀌거나 `transcript.version.updated`·`job.updated`·`session.updated`를 받으면 `/record`를 다시 조회해 새 개수·상태·경로를 반영한다.
- Session은 종료 확정 직후 `PROCESSING`이며 새 Audio·질문·반응·Answer 입력을 제공하지 않는다. 실시간 Audio WS와 partial은 끝나지만 active publisher 브라우저의 녹음 upload는 별도 전송으로 계속될 수 있다.
- `PROCESSING`에는 개인 `REVIEW` Chat을 제공하지 않는다. 진입 시 삭제된 LIVE Summary·Chat·Message·Evidence와 Job의 선택 ID·cache를 모두 비우며, 직접 조회 `404`를 반복 재시도하지 않는다.
- 녹음 upload는 UI에서 준비(`preparing`), 진행 중(`in-progress`), 중단(`interrupted`), 재개 중(`resuming`), 완료(`completed`), 실패(`failed`)를 구분하고 완료 시 원본 저장 완료를 함께 알린다. 이 이름은 화면 상태이며 API resource·enum을 먼저 확정하지 않는다.
- upload가 `completed`가 된 뒤 영구 Transcript를 `FINALIZING`으로 전환하고 `RECORDING_TRANSCRIPTION` Job으로 전체 녹음 HQ STT를 시작한다.
- resumable protocol·checksum·만료·최대 크기와 fragment manifest는 표시하지 않는다. 서버가 신뢰할 수 있는 전체·전송량을 제공할 때만 progress를 계산하고 그렇지 않으면 불확정 진행 상태를 사용한다.
- Transcript 후처리는 `전체 녹음 재처리 → 문장 분할·시작/종료 시각 정렬 → 녹음 위치 매핑 → Segment 저장 → canonical 전환` 단계로 표시한다. 전체 재처리·Segment 저장·시간/녹음 위치 매핑·canonical 전환이 모두 성공해야 `FINALIZED`다.
- 정상 처리가 끝났지만 Segment가 0개이면 `EMPTY`, HQ STT 또는 canonical 전환이 실패하면 `FAILED`로 표시한다. `FINALIZED`는 100% 정확도를 보장하는 품질 표기가 아니라 기술적 완료 상태다.
- canonical 전환 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증한다. Answer 재매핑은 별도 단계에서 `PENDING`, `SUCCEEDED`, `FAILED`를 표시하고, 전체 범위를 연결한 `SUCCEEDED`에만 새 Segment 범위 이동을 제공한다. 허용 오차와 부분 일치 세부 상태는 화면이 임의로 만들지 않는다.
- 완료된 음성 Answer는 LIVE에서 `NOT_STARTED`, PROCESSING에서 재매핑과 source 선택이 끝나기 전에는 `job_id=null`인 `WAITING_SOURCE`로 표시한다. source가 terminal이 되면 coordinator가 `visibility=SHARED`, `blocks_session_completion=true`인 `ANSWER_ORGANIZATION` Job을 Answer마다 자동 생성한다. Job 생성 시 재매핑 성공 HQ 범위를 우선하고, 재매핑이 없거나 실패하면 Answer의 immutable 원본 LIVE 범위를 선택해 재시도에도 고정한다. 원본 범위는 정리 결과로 덮어쓰지 않는다.
- 독립 작업은 `SESSION_POSTPROCESSING`, `RECORDING_TRANSCRIPTION`, 음성 Answer별 `ANSWER_ORGANIZATION`, `FINAL_SUMMARY`, `clustering_mode=FINAL`인 `QUESTION_CLUSTERING`이다. 화면은 Transcript·Answer 재매핑·Answer별 AI 정리·Summary·clustering의 부분 실패를 하나의 “전체 실패”로 합치지 않는다.
- HQ source gate가 아직 결정되지 않았으면 FINAL Summary는 `summary_status=PENDING`, `summary=null`, `job_id=null`인 정상 대기 상태로 표시한다. 최신 HQ `RECORDING FINALIZED` Segment가 1개 이상이 되는 transaction에서 coordinator가 FINAL Summary Job을 함께 생성하므로, 이 eligible 상태에 Job이 없으면 `data-error`다. `PROCESSING`에서는 attempt와 무관하게 Job `PENDING|RUNNING`이 정상이고 `COMPLETED`에서는 실패 뒤 명시적 재시도인 `attempt > 1` active Job만 정상이다.
- RECORDING이 `EMPTY`이면 Job 없는 `NOT_APPLICABLE/NO_FINAL_TRANSCRIPT`, source 부재·실패·10분 deadline이면 Job 없는 `FAILED/SUMMARY_SOURCE_UNAVAILABLE`로 표시한다. data-error에는 내부 원문 대신 “요약 상태를 확인할 수 없습니다. 잠시 후 다시 시도해 주세요.”를 사용한다.
- `PROCESSING` 전환 시 실행 중이던 LIVE 클러스터링은 종료하고 늦은 결과 commit을 받지 않는다. FINAL Job은 Transcript 완료에 의존하지 않고, 종료 시점 질문 sequence 이하의 학생 실제 질문 전체와 종료 시각까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 input snapshot으로 사용한다. 학생 질문 상한은 모든 attempt에서 유지한다. FINAL Job이 `FAILED`한 뒤 교수자가 명시적으로 재시도할 때만 `base_revision`과 Answer 시각 상한을 현재 값으로 다시 캡처한다.
- Job은 API 상태 `PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`와 attempt를 표시한다. 화면의 `retrying`은 별도 API enum이 아니라 `attempt > 1`인 `PENDING|RUNNING`을 묶은 파생 상태다. percent가 제공될 때만 progress를 표시한다.
- 교수자만 FAILED·retryable 공유 Job을 같은 `job_id`, `attempt + 1 → PENDING`으로 재시도한다. 음성 Answer 정리도 Answer별 같은 규칙을 사용하고, 이미 `SUCCEEDED`한 정리는 MVP에서 다시 생성하지 않는다. 학생은 교수자 재시도 안내만 본다.
- Worker heartbeat는 15초 주기이며 60초 동안 신호가 없으면 중단으로 표시한다. HQ STT를 포함한 모든 후처리 Job은 최대 5분, Session 전체 `PROCESSING`은 최대 10분을 적용한다. 10분 제한에서 아직 생성·시작하지 못한 필수 후속 작업도 재시도 가능한 `FAILED` Job으로 표시한다.
- Course owner는 Session `PROCESSING` 중에도 class 제목을 수정할 수 있지만 class·Material 삭제와 Material 업로드는 할 수 없다. 화면에는 PDF 선택·삭제 control을 렌더링하지 않고, stale 요청의 `SESSION_STATE_CONFLICT`에는 “class 기록 정리 중에는 강의자료를 추가하거나 삭제할 수 없습니다.”를 표시한다.
- 녹음 upload·HQ STT·Answer 재매핑·Answer별 AI 정리·Summary·clustering 실패가 PDF·저장된 live Segment·원본 질문·Answer를 가리지 않는다. AI 정리가 실패해도 해당 Answer의 교수자 text와 immutable 원본 LIVE 음성 범위를 계속 제공한다. HQ STT 실패 시 실시간 기본 조회에 사용하던 LIVE 포인터는 보존하지만 이를 완료 기록의 final source로 인정할지는 미정이다. 실패 경로와 계속 열람 가능한 기록을 함께 안내한다.
- 최종 Cluster는 `is_final=true`인 현재 확정본을 `ordinal ASC, id ASC`로 표시한다. 중앙 AI 대표질문과 typed child를 미리보기로 제공하고, 마지막 FINAL Job의 mode·상태·attempt와 `finalized_at`을 함께 표시한다. Answer로 보존된 과거 대표질문 외의 membership 이력은 제공하지 않는다.
- 완료는 클라이언트 Job count, 녹음 upload 상태나 HQ STT 시작 여부를 조합해 추정하지 않는다. 음성 Answer별 `ANSWER_ORGANIZATION`을 포함한 완료 차단 Job이 모두 `SUCCEEDED` 또는 `FAILED`가 되면 부분 실패가 있어도 서버가 Session을 `COMPLETED`로 바꾸며, `session.updated=COMPLETED` 또는 REST Session이 진실이다.
- 실패한 AI 기능을 완료 화면에서 재시도해도 Session을 다시 `PROCESSING`으로 돌리지 않는다. 늦게 도착한 이전 Worker 결과는 화면에 반영하지 않고 최신 Job·attempt를 다시 조회한다.
- Session, 녹음 upload, 영구 Transcript, Answer 재매핑, Answer별 AI 정리와 기존 후처리의 UI 축을 독립적으로 둔다. 현재 `class-processing.html`에는 이 상태들이 없으며 PR10에서 정적 상태를 추가한다.
- 900px 이하에서 작업과 저장 기록을 한 열로 배치하되 저장 기록을 spinner 아래 숨기지 않는다. terminal 변화는 polite, 실패는 alert, retry 이름에는 작업명을 포함한다.

Session `completed_at`, `blocks_session_completion`, 공개 Job 상태와 retry 가능 여부는 API 계약을 그대로 사용한다. Answer AI 정리 결과의 구조·최대 길이·모델·prompt·품질 기준과 성공 결과의 KnowledgeChunk 재색인 방식은 미정이다. 그 밖의 미정 사항은 HQ 실패 시 보존된 LIVE 포인터의 final source 인정 여부, Answer 재매핑 허용 오차·부분 실패 상태, upload protocol·checksum·만료·최대 크기, fragment manifest와 upload 실패 뒤 새 upload 재시작 정책이다.

## 8. 완료 class 기록

### 8.1 완료 기록 공통 녹음 playback·Transcript seek

- 교수자와 학생 완료 화면은 같은 녹음 player 구조와 접근 거부 상태를 제공한다. 이 구조는 두 역할 모두에게 playback을 허용한다는 뜻이 아니며 정확한 역할별 접근 정책은 미정이다.
- playback은 `checking`, `ready`, `loading`, `playing`, `paused`, `ended`, `unavailable`, `forbidden`, `expired`, `error` 상태를 제공한다. 재생 시작과 Transcript 문장 seek 요청마다 서버에서 현재 권한을 다시 확인한다.
- 전체 Course 접근이 거부되면 기존 화면 규칙대로 기록 전체를 숨긴다. 녹음만 허용되지 않는 경우에는 player control을 비활성화하고 `forbidden`을 표시하되 PDF·Transcript·질문 등 허용된 기록은 유지한다.
- 보호된 녹음 source를 내부 storage key·물리 경로나 장기 URL로 DOM에 넣지 않는다. 인증 만료와 접근 거부 뒤에는 기존 source를 재사용하지 않는다.
- canonical Transcript 문장은 click과 키보드로 활성화할 수 있게 하고, 권한 확인 뒤 표시 시각을 파싱한 추정값이 아니라 서버가 제공한 녹음 위치로 seek한다. `idle`, `seeking`, `active`, `seek-error`를 player 상태와 독립적으로 알린다.
- Transcript cursor 페이지는 `segments[]`와 `gaps[]`를 나눠 받고 두 배열의 항목 합계는 요청 `limit`을 넘지 않는다. 화면은 같은 version의 두 배열을 시작 시각의 안정적인 순서로 merge해 한 흐름에 함께 표시한다. gap은 재생 위치 control을 제공하지 않고 `[05:20~05:35] 연결 문제로 기록되지 않은 구간입니다.`처럼 누락 범위와 원인을 읽을 수 있게 표시한다.
- 영구 Transcript 상태는 실시간 연결·STT 상태와 별개이며 완료 기록에서는 다음과 같이 표현한다.

| Transcript 상태 | Transcript 영역                                                                      | Summary 영역                                                                       |
| --------------- | ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| `FINALIZED`     | source와 함께 Segment·gap을 시간 순서대로 표시하고 Segment에 seek 제공               | 최신 HQ `RECORDING FINALIZED`만 Summary·Job 표시; LIVE만 있으면 source unavailable |
| `EMPTY`         | “확정된 강의 문장이 없습니다.”를 표시하고 final gap이 있으면 시간 순서대로 함께 표시 | `NOT_APPLICABLE`·`NO_FINAL_TRANSCRIPT`, “요약할 강의 내용이 없습니다.”             |
| `FAILED`        | “Transcript 처리에 실패했습니다.”와 재시도 가능 여부 표시                            | `SUMMARY_SOURCE_UNAVAILABLE`, “Transcript 처리 문제로 요약을 만들지 못했습니다.”   |

- `FINALIZING`인데 Session이 아직 `PROCESSING`이면 7.1절로 안내한다. `COMPLETED` 응답에서 계속 `FINALIZING`이면 클라이언트가 완료를 추정하지 않고 데이터 불일치 상태로 표시한다.
- `FINALIZED`는 전체 녹음 재처리, Segment 저장, 시간·녹음 위치 매핑과 canonical 전환의 성공을 뜻하며 100% 정확도 보장 문구를 사용하지 않는다.
- FINAL Summary는 최신 HQ `RECORDING FINALIZED` Segment가 있을 때만 표시한다. Recording source 자체가 없으면 즉시, Recording은 있지만 HQ 실패·무결과이면 terminal 시점에 `SUMMARY_SOURCE_UNAVAILABLE`을 표시하고 보존된 LIVE 포인터를 자동 final source로 사용하지 않는다.
- `READY`의 `summary=null`·Job 없음과 `LIVE`에서 요청 전의 같은 조합은 정상이다. `PROCESSING`에서 HQ source gate가 정해지지 않은 `PENDING`도 `summary=null`, `job_id=null`이 정상이며 7.1절을 따른다.
- RECORDING `EMPTY`는 Job 없이 `NOT_APPLICABLE/NO_FINAL_TRANSCRIPT`, source 부재·실패·deadline은 Job 없이 `FAILED/SUMMARY_SOURCE_UNAVAILABLE`이다. 반대로 `SUCCEEDED` FINAL Summary Job인데 결과가 없거나 `COMPLETED`의 eligible `FINALIZED` source에 Job이 없으면 `DATA_INTEGRITY_ERROR`다.
- `COMPLETED`에서 최초 `attempt=1` FINAL Summary Job이 `PENDING|RUNNING`으로 남은 경우도 `DATA_INTEGRITY_ERROR`다. 이 상태를 retry 가능한 모델 실패로 가장하지 않고 “요약 상태를 확인할 수 없습니다. 잠시 후 다시 시도해 주세요.”를 표시하며 provider 원문·누락된 내부 식별자는 노출하지 않는다.
- 완료된 음성 Answer는 교수자 작성 text, 별도 label의 AI 정리 결과와 음성 원본을 서로 합치지 않고 표시한다. 교수자 text가 있으면 기본 답변으로 우선 배치하고, AI 정리는 “AI로 정리한 답변”으로 구분한다. 이동·재생에는 성공한 HQ 재매핑 범위를 우선 사용하되 사용할 수 없으면 immutable 원본 LIVE 범위를 사용하며 원본 범위 접근은 항상 유지한다.
- Answer 하나의 AI 정리 실패는 다른 Answer와 기록 영역을 가리지 않는다. 학생은 실패 상태와 원본 음성을 볼 수 있고, Course 교수자에게만 같은 `ANSWER_ORGANIZATION` Job의 재시도를 제공한다. `SUCCEEDED` 결과의 다시 생성 control은 MVP에 제공하지 않는다.
- 녹음 download control은 MVP에 제공하지 않는다. canonical 전환 뒤 Answer 시간 범위 재매핑 결과를 사용하되, 허용 오차나 부분 실패 상태를 화면이 임의로 추정하지 않는다.
- player와 문장 seek는 키보드 이름·focus·현재 문장 표시를 제공하고 비동기 권한 확인은 polite, 접근 거부와 seek 실패는 alert로 알린다.
- 현재 교수자·학생 완료 Prototype에는 player·문장 seek, mixed gap, Transcript final·empty·failed, compact manifest·cursor 추가 로딩·Evidence 이동 상태가 없다. PR10에서 정적 상태만 추가하고 실제 audio source나 네트워크 권한 확인은 실행하지 않는다.

미정 사항: 저장 녹음 codec·container, 녹음 동의, 역할별 접근, 보관 기간과 삭제 정책. 이 결정 전에는 default 허용 역할이나 download 가능 여부를 화면 fixture로 확정하지 않는다.

#### 완료 기록 manifest·cursor·Evidence 공통 규칙

- 화면 진입 시 `/record`를 먼저 요청해 기록 요약·처리 상태·영역별 개수·권한 검사 공개 조회 경로를 받는다. `/record`에 Material·Transcript·질문·Answer·Cluster·공유 Job 전체 배열이 있다고 가정하지 않는다.
- manifest의 count가 0인 영역은 즉시 빈 상태로 표시하고, 항목이 있는 영역은 화면 순서와 viewport에 따라 공개 조회 경로를 독립적으로 점진 로딩한다. 개수는 초기 안내와 완료 여부에 쓰고 실제 항목은 각 목록 응답을 진실로 사용한다.
- Transcript·질문·Answer·Cluster 목록은 각각 불투명 cursor와 고정된 정렬을 사용한다. 화면 기본 page size는 20이고 요청 가능 최대는 100이며, 각 Cluster의 child 목록도 같은 방식으로 따로 연다.
- Transcript는 하나의 `next_cursor`를 공유하는 `segments[]`·`gaps[]`를 페이지별로 수신한다. 페이지의 두 배열 항목 합계가 `limit`을 넘지 않는 계약을 전제로, 로딩된 모든 페이지를 같은 version·시작 시각 순서로 merge해 Segment·gap 혼합 타임라인을 만든다.
- 첫 페이지 실패는 해당 영역만 `error`로 표시하고, 추가 페이지 실패는 이미 로딩한 항목을 유지한 채 “다음 항목을 불러오지 못했습니다”와 다음 cursor 재시도를 표시한다. 재시도는 같은 공개 경로·cursor를 사용하고 다른 영역을 새로고침하지 않는다.
- 불러오기 동작은 명시적인 “더 보기” control을 기본으로 제공하고 불투명한 자동 로딩만에 의존하지 않는다. 요청 중 control의 로딩·비활성 상태와 새로 추가된 항목 수를 polite로 알린다.
- Cluster generation 교체 중 기존 cursor의 만료·재시작 정책과 큰 마인드맵의 preload page 수·자동 축소·탐색 방식은 미정이다. 이 정책이 확정되기 전에 클라이언트가 cursor를 임의로 재구성하지 않는다.
- Evidence `source_kind`는 `MATERIAL`, `TRANSCRIPT_SEGMENT`, `STUDENT_QUESTION`, `AI_REPRESENTATIVE_QUESTION`, `ANSWER`이다. 사용자용 `label`을 동작 이름으로 보여 주고, 선택 시 API가 반환한 안정적인 공개 `link`를 열어 현재 권한을 다시 검사한 뒤 각각 자료 페이지, Transcript Segment, 학생 질문, AI 대표질문, Answer로 이동한다.
- Evidence `link`는 `/record` 배열 index나 pagination cursor를 포함하거나 그 순서로 클라이언트가 만들지 않는다. DOM·URL·오류 문구에 내부 `knowledge_chunk_id`, storage key·path, provider 원문 URL을 노출하지 않는다.
- Evidence 이동 실패는 저장된 AI 답변을 삭제하거나 Chat 전체를 오류로 바꾸지 않고 해당 근거만 재시도할 수 있게 한다. Material detach·원본 삭제 또는 AI 대표질문 폐기 후에는 안전한 `label` snapshot을 유지하고 `link=null`인 이동 control을 비활성화한다. 폐기된 대표질문은 새 검색에서 제외하고 마지막 Evidence를 삭제하는 transaction에서 함께 정리한다. Material Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이다.

#### 교수자·학생 공통 개인 REVIEW Chat

- 개인 `REVIEW` Chat은 Session이 `COMPLETED`일 때만 교수자·학생에게 같은 UI와 기능으로 제공한다. `PROCESSING`에서는 form과 기존 REVIEW 대화를 제공하지 않고 정리 중 화면을 표시한다. 교수자 전용 기록 관리 control은 공통 Chat UI 밖에 추가한다.
- USER Message와 그 Message를 입력으로 고정한 `CHAT_RESPONSE` Job·outbox를 `202` 응답 전에 하나의 transaction으로 반드시 저장한 뒤 요청자가 Job을 polling한다. 성공하면 저장된 최종 assistant Message와 Evidence를 REST로 다시 조회하며, 생성 중 assistant 부분 문장은 저장·표시·공유하지 않는다.
- 한 Chat에는 진행 중인 `CHAT_RESPONSE` Job을 하나만 허용한다. 실패 재시도는 같은 `job_id`의 `attempt`를 증가시켜 `PENDING`으로 되돌리고 동일 USER Message를 사용하며 기존 USER Message와 성공 결과는 유지한다.
- USER `content`는 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 1자 이상 2,000자 이하다. 공백뿐이거나 초과한 입력은 자르지 않고 `422 VALIDATION_ERROR`로 거부하며, 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 사용한다. textarea 원문은 그대로 두고 정규화 결과의 글자 수와 오류를 표시한다. assistant Message에는 이 길이 제한을 적용하지 않는다.
- Chat은 `empty|pending|complete|no-evidence|failed`를 제공한다. Evidence 유형·label·link·이동 실패는 앞 절의 공통 규칙을 따르며 근거 없음도 성공한 최종 결과로 표시한다.
- 개인 Chat Message·Job·생성 상태·결과는 shared Session WebSocket에 broadcast하지 않는다. token delta, SSE, streaming HTTP와 private WebSocket 없이 requester-only Job polling과 REST 저장 결과만 사용한다.
- LIVE→PROCESSING 정리 대상은 LIVE 개인 AI에 한정하며 FINAL Summary와 `COMPLETED` REVIEW Chat을 삭제하거나 서로 승격하지 않는다.

### 8.2 `ENDED_CLASS_PAGE_PROF` · 교수자 완료 기록

- `COMPLETED` Session에서 `/record`의 기록 요약·상태·영역별 개수·공개 조회 경로를 먼저 받고 attached materials, canonical Transcript, FINAL Summary, questions, answers, Cluster와 shared jobs를 영역별로 점진 로딩한다. 학생과 같은 개인 `REVIEW` Chat 및 별도 권한 확인을 거친 녹음 player를 함께 제공하고 `view=loading|normal|error|forbidden|expired|processing`을 사용한다.
- manifest 전체 요청이 실패하면 `view=error`에서 재시도하고, 성공한 뒤에는 8.1절 공통 규칙에 따라 영역별 `loading|empty|normal|loading-more|page-error|error`를 서로 독립적으로 표시한다.
- Header에는 제목, `lecture_date`, 실제 `started_at`과 `completed_at`을 표시한다. owner는 제목만 수정할 수 있고 상태 시각은 읽기 전용이며 class 삭제를 확인한 뒤 Course 화면으로 이동할 수 있다.
- owner에게 복수 PDF의 서버 표시 이름·크기·쪽 수·처리 상태, 권한 확인된 원본 열람, active `N/10`, PDF 추가와 Material 삭제를 제공한다. 파일 제한·suffix·상태·오류 문구는 5.1절을 따른다.
- `READY` Material만 복습 AI의 새 근거가 된다. `UPLOADED`, `PROCESSING`은 처리 중으로, `FAILED`는 “업로드 실패 · 사용할 수 없음”으로 표시하며 실패 행에는 열람 동작을 제공하지 않는다.
- Material 삭제가 성공하면 목록·열람·새 AI 검색에서 즉시 제거하고 background cleanup을 기다리지 않는다. 현재 `class-ended-professor.html` Prototype의 업로드·삭제 없는 읽기 전용 자료 영역은 후속 동기화 대상이다.
- Transcript가 `FINALIZED`이면 cursor 페이지의 `segments[]`·`gaps[]`를 시작 시각 순서로 merge해 점진 표시하고 playback 권한이 확인된 Segment에는 녹음 위치 seek control을 제공한다. `EMPTY`·`FAILED` 표현은 8.1절을 따른다. 음성 Answer는 target 하나, immutable 원본 LIVE 범위, 선택적 HQ 재매핑 범위, 교수자 text와 별도 AI 정리를 표시하고 text-only Answer는 target 하나와 교수자 text를 표시하며 여러 질문을 한 Answer로 묶지 않는다.
- FINAL 질문 영역은 질문·Answer·Cluster를 서로 독립적인 cursor 목록으로 조회한다. `is_final=true` Cluster를 `ordinal ASC, id ASC`로 나열하고, 각 Cluster의 typed child는 해당 members 경로의 cursor로 펼친다. 현재 AI 대표질문을 중앙에, 학생 질문과 답변된 과거 AI 대표질문을 child로 표시하고 마지막 FINAL Job의 mode·상태·attempt와 `finalized_at`을 제공한다.
- Answer 범위 선택 시 해당 sequence에 focus하고 highlight한다. 현재 Cluster에 다시 join해 문구를 바꾸지 않고 공개 응답의 `target_text_snapshot`으로 교수자가 답변한 당시 target 문구를 표시한다.
- 새 질문·반응과 음성 Answer 캡처는 제공하지 않는다. Course 교수자는 미답변 학생 질문 또는 AI 대표질문 하나에 text-only Answer를 만들고, 기존 `COMPLETED` 음성·text-only Answer의 교수자 text를 추가·수정할 수 있다. 대표질문 Answer는 child의 답변 상태를 바꾸지 않는다.
- 개인 REVIEW Chat은 8.1절의 교수자·학생 공통 polling·입력 검증·Evidence 규칙을 그대로 사용한다. 교수자 Chat을 학생 Chat과 다른 보관·전송 방식으로 분기하지 않는다.
- 영역 상태는 playback과 seek의 8.1절 상태, materials `normal|multiple|empty|uploading|processing|failed|deleting|error`, transcript `finalized|empty|failed|error`, summary `pending|normal|not-applicable|source-unavailable|failed|retrying|data-error`, questions `normal|empty|cluster-failed|cluster-retrying|error`, answers `normal|organization-waiting-source|organization-pending|organization-running|organization-succeeded|organization-failed|organization-retrying|organization-data-error|text-editing|text-saving|text-error`, jobs `success|partial-failure|retrying`이다. `cluster-retrying`은 `mode=FINAL`, `attempt > 1`, `PENDING|RUNNING`인 마지막 Job의 화면 파생 상태이고 기존 저장 질문은 유지한다. `organization-retrying`도 `attempt > 1`인 `PENDING|RUNNING`의 파생 상태다. `organization-data-error`는 Answer를 유지한 채 “답변 정리 상태를 확인할 수 없습니다.”로 표시하고 내부 오류 원문과 잘못된 retry control을 숨긴다.
- Summary의 `summary=null`·Job 없음은 8.1절의 source·Session 상태와 함께 해석한다. `EMPTY`와 source unavailable은 정상 terminal 표현이고, eligible source의 Job 누락 또는 SUCCEEDED Job의 결과 누락은 `data-error`로 표시한다. 녹음 playback·HQ STT·Answer 재매핑·Summary·clustering 실패가 PDF·열람 가능한 기존 기록·원본 질문·Answer를 가리지 않는다.
- 교수자의 추가 쓰기 동작은 class 제목 수정·삭제, Material 추가·삭제, text-only Answer 등록·기존 Answer의 교수자 text 추가·수정과 FAILED+retryable shared Job 재시도다. 개인 REVIEW Chat은 학생과 같은 공통 동작이다. 실패한 Answer AI 정리는 같은 `job_id`의 attempt를 증가시키고 Session은 COMPLETED를 유지하며, 성공한 Answer 정리와 FINAL Summary 다시 생성은 제공하지 않는다.
- 완료 화면은 shared WS가 필수가 아니며 REST가 진실이다. 900px 이하 한 열에서도 PDF→녹음 player→Summary→Transcript→Q&A→Jobs→개인 REVIEW Chat 순서를 유지한다.

미정·후속: generation 교체 중 cursor 만료·재시작과 큰 마인드맵의 preload page 수·자동 축소·탐색 방식, Answer text 최대 길이·`COMPLETED` Answer 삭제 정책, 성공한 FINAL 뒤 text Answer 추가에 따른 자동 재클러스터링, Answer AI 정리 결과의 구조·최대 길이·모델·prompt·품질 기준과 KnowledgeChunk 재색인, HQ 실패 시 보존된 LIVE 포인터의 final source 인정 여부와 Answer 재매핑 허용 오차·부분 실패 상태를 동기화한다.

### 8.3 `ENDED_CLASS_PAGE_STUD` · 학생 완료 기록과 복습 AI

- `COMPLETED` Session에서 `/record`의 기록 요약·상태·영역별 개수·공개 조회 경로를 먼저 받고 attached PDF, canonical Transcript, FINAL Summary, 최종 질문 마인드맵, target별 교수자 Answer와 shared Job을 영역별로 점진 로딩한다. 별도 권한 확인을 거친 녹음 player를 함께 제공하고 Session이 아직 `PROCESSING`이면 기록 정리 중 화면으로 안내한다.
- manifest 성공 후 영역별 `loading|empty|normal|loading-more|page-error|error`를 독립적으로 표시하고, 한 영역의 첫 페이지·추가 페이지 재시도가 다른 영역의 로딩된 기록을 초기화하지 않는다.
- Header에는 제목, `lecture_date`, 실제 `started_at`과 `completed_at`을 읽기 전용으로 표시한다. 학생에게는 제목 수정과 class 삭제 control을 제공하지 않는다.
- 완료 기록은 읽기 전용이다. 학생 DOM에는 Material 업로드·삭제, shared AIJob 재시도, 새 질문 작성, 반응 추가·취소, Answer 시작 control을 렌더링하지 않는다.
- attached PDF의 표시와 부분 실패 격리 원칙은 교수자 화면과 같지만 학생에게 관리 control은 없다. 복습 AI는 `READY` Material만 새 근거로 사용하고 `FAILED`는 사용할 수 없는 업로드 실패로 표시한다.
- 녹음 player·문장 seek·Summary·Transcript·질문·Answer의 표현과 부분 실패 격리 원칙은 교수자 화면과 같다. 녹음 접근이 거부돼도 허용된 다른 기록을 유지한다. Answer는 `target_text_snapshot`으로 선택 당시 target 문구를 표시하고, 교수자 text가 있으면 우선 표시하며 AI 정리 결과는 별도 label로 구분한다. 음성 Answer의 immutable 원본 LIVE 범위는 항상 제공하고 성공한 HQ 재매핑 범위가 있으면 이동·재생에 우선 사용한다.
- Transcript가 `FINALIZED`이면 cursor 페이지의 `segments[]`·`gaps[]`를 시작 시각 순서로 merge해 점진 표시한다. 녹음 권한이 확인된 Segment 선택은 녹음 위치로 seek하고, Answer range 선택은 재매핑된 Segment 구간으로 이동해 focus와 highlight를 제공한다. `EMPTY`·`FAILED`와 Summary 표현은 8.1절을 따른다.
- 최종 질문 영역은 질문·Answer·Cluster 목록을 각 cursor로 나눠 조회한다. `is_final=true` Cluster를 `ordinal ASC, id ASC`로 나열하고 AI 대표질문을 중앙에, 학생 질문과 답변된 과거 대표질문을 members cursor로 펼치는 typed child로 표시한다. 마지막 FINAL Job의 mode·상태·attempt와 `finalized_at`을 제공하고 일반 과거 membership 이력은 표시하지 않는다. 종료된 class에서는 질문·반응을 새로 만들 수 없다.
- 개인 복습 Chat은 `COMPLETED`에서만 `REVIEW` 모드로 제공하며 8.1절의 교수자·학생 공통 polling·입력 검증·Evidence 규칙을 그대로 사용한다. `PROCESSING`에서는 Chat form을 제공하지 않는다.
- 답변 근거가 없을 때는 기록 범위 밖임을 명시하고 근거를 꾸며내지 않는다. Evidence의 내부 `knowledge_chunk_id`, storage key·path와 provider 원문 URL은 DOM·이동 경로·오류 문구에 노출하지 않는다. 공개 `link` 열기 실패는 AI 답변을 유지한 채 해당 Evidence만 재시도한다.
- Material detach·원본 삭제 또는 AI 대표질문 폐기 후에는 기존 AI Evidence 행의 안전한 `label` snapshot을 계속 표시하고 `link=null`로 이동 control을 비활성화한다. 폐기된 대표질문 tombstone은 새 검색에서 제외하고 마지막 Evidence를 삭제하는 transaction에서 함께 정리한다. Material Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이며 detached Material과 KnowledgeChunk는 새 검색에 사용하지 않는다.
- 개인 Chat Message·Job·생성 상태·결과는 shared Session WebSocket에 broadcast하지 않는다. MVP 프로토타입은 token delta, SSE, streaming HTTP와 private WebSocket 없이 polling과 REST 저장 결과만 표시한다.
- 화면 전체 상태는 `view=loading|normal|error|forbidden|expired|wrong-role|processing`이다. Course 학생 역할과 Session 접근 권한을 서버에서 다시 검사하며 접근 실패 시 완료 기록과 Chat form을 숨긴다.
- 영역 상태는 playback과 seek의 8.1절 상태, materials `multiple|normal|empty|failed|error`, transcript `finalized|empty|failed|error`, summary `pending|normal|not-applicable|source-unavailable|failed|data-error`, questions `normal|empty|cluster-failed|cluster-retrying|error`, answers `normal|organization-waiting-source|organization-pending|organization-running|organization-succeeded|organization-failed|organization-retrying|organization-data-error|error`, jobs `success|partial-failure|retrying`이며 서로 독립적으로 복구할 수 있다. 학생은 control 없이 FINAL과 Answer organization 재시도의 `PENDING|RUNNING` 상태와 attempt를 관찰하고 기존 저장 결과를 계속 본다. `organization-data-error`와 Summary `data-error`에서는 저장 기록을 유지한 채 안전한 오류 문구만 본다.
- 900px 이하에서는 한 열로 바꾸고 375px에서도 Header·완료 상태 → PDF → 녹음 player → Summary → Transcript → 질문·Answer → 개인 복습 Chat 순서를 유지한다. Chat 결과 영역은 `role="log"`, 비동기 상태는 polite, 입력 오류와 작업 실패는 alert로 알린다.

미정·후속: Material detach·원본 삭제 후 기존 Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책, generation 교체 중 cursor 만료·재시작과 큰 마인드맵의 preload page 수·자동 축소·탐색 방식, Answer text 최대 길이·`COMPLETED` Answer 삭제 정책, 성공한 FINAL 뒤 text Answer 추가에 따른 자동 재클러스터링, Answer AI 정리 결과의 구조·최대 길이·모델·prompt·품질 기준과 KnowledgeChunk 재색인, HQ 실패 시 보존된 LIVE 포인터의 final source 인정 여부, Answer 재매핑 허용 오차·부분 실패 상태를 API·DB 명세와 동기화한다.
