# GOAL: God Of All Lectures 기획안

## 1. 프로젝트 개요

**GOAL: God Of All Lectures**는 오프라인 수업을 위한 실시간 학습 보조 및 복습 서비스이다. 수업 중에는 스트리밍 음성 인식(STT)으로 생성되는 실시간 강의 기록과 AI 요약을 통해 놓친 내용을 따라가고, 익명으로 질문할 수 있다. 수업 후에는 강의 자료, 강의 기록 및 질문·답변을 기반으로 AI 튜터와 복습할 수 있다.

### 목표 사용자

- 대학교 교수, 강사 등 다수의 학습자를 가르치는 교수자
- 오프라인 수업에 참여하는 학생

### 계정과 Course 역할

- 로그인 계정에는 전역 교수자·학생 역할이 없다. 모든 인증 사용자는 Course를 생성하거나 참여 코드로 기존 Course에 참여할 수 있다.
- Course를 생성한 사용자는 해당 Course의 유일한 교수자이자 owner가 되고, 코드로 참여한 사용자는 해당 Course의 학생이 된다.
- Course당 교수자는 한 명이며 교수자 추가와 owner 이전은 제공하지 않는다.

## 2. 해결하려는 문제

### 학생

- 공개적으로 질문하기 부담스럽다.
- 수업 중 놓친 내용을 바로 따라가기 어렵다.
- 복습할 때 교수자가 실제로 설명한 내용을 찾기 어렵다.
- 다른 학생들이 어떤 부분을 어려워했는지 알기 어렵다.

### 교수자

- 학생들이 어느 지점에서 막혔는지 실시간으로 파악하기 어렵다.
- 비슷한 질문이 반복되어 수업 흐름이 끊긴다.
- 수업이 끝난 뒤 질문과 답변을 별도로 정리하기 번거롭다.

## 3. 핵심 해결 방식

인증 사용자는 한 학기 단위의 수업방(Course Room)을 만들고 해당 Course의 유일한 교수자가 되며, 매 수업마다 강의 세션(Lecture Session)을 생성한다. 수업 중 교수자의 음성을 스트리밍 STT로 처리해 임시 인식 결과를 즉시 전달하고 발화 단위 live Segment를 저장한다. 수업 종료 후에는 저장한 전체 녹음을 HQ STT로 다시 처리해 문장별 시작·종료 시각과 녹음 재생 위치가 연결된 canonical Transcript를 만든다. 각 세션에는 강의자료, 강의 기록, 질문, 답변 및 AI 정리 결과가 축적된다.

학생은 수업 중 익명 질문과 실시간 기록을 활용한다. 교수자와 학생은 같은 개인 AI 영역에서 수업 중 요약·Chat을 사용하고, 수업 완료 후에는 해당 세션의 자료와 질문·답변을 근거로 AI와 대화하며 복습한다. 교수자에게만 필요한 수업·자료·답변 관리 기능은 이 공통 화면에 추가 control로 제공한다.

```text
수업방 (한 학기)
├── 03/05 강의 세션
│   ├── 강의자료 PDF
│   ├── 강의 기록(Transcript)
│   ├── 학생 질문 및 반응
│   ├── 교수자 답변
│   ├── AI 요약
│   └── 복습용 AI 채팅
├── 03/12 강의 세션
└── 03/19 강의 세션
```

## 4. 주요 기능

### 수업 전

**인증 사용자 공통**

- 한 학기 Course 생성 또는 참여 코드로 기존 Course 참여
- Course 생성 시 유일한 교수자 owner, 코드 참여 시 학생 역할 획득

**Course 교수자**

- 학생 참여 코드 확인·회전
- 날짜별 강의 세션 생성
- `READY` class에 선택적으로 강의자료 PDF 업로드·삭제
- 수업 시작

**Course 학생**

- 현재 강의 세션 상태 확인과 입장

### 수업 중

**교수자**

- 동일한 교수자 마이크 입력을 실시간 STT 전송과 브라우저 로컬 녹음으로 함께 처리하고 각 경로 상태 확인
- 첫 `audio.start` 성공 탭에서만 음성을 전송하고 중복 publisher 탭에서는 수업 조회 유지
- 학생의 익명 질문 실시간 확인
- AI가 유사 질문을 묶고 중앙에 대표질문을 두는 질문 마인드맵 확인
- ‘나도 궁금해요’ 반응이 많은 질문 우선 확인
- 선택한 학생 질문 또는 AI 대표질문 하나에 음성으로 답변
- `LIVE` class의 강의자료 PDF 추가·삭제
- 학생과 같은 개인 AI 영역에서 현재 강의 Summary·Chat 사용

**학생**

- 스트리밍 STT로 생성되는 실시간 강의 기록 확인
- 익명 질문 작성
- 다른 질문에 ‘나도 궁금해요’ 반응
- 현재 연결된 `READY` 강의자료와 현재까지의 수업 기록을 바탕으로 AI에게 질문
- “방금 내용 요약해줘”, “놓친 부분을 설명해줘” 등의 따라잡기 요청

개인 AI Summary·Chat은 Course 역할과 무관하게 같은 UI와 기능을 사용하며 결과는 요청자에게만 보인다. 교수자의 publisher·강의 종료·자료 관리·Answer control만 공통 화면에 추가한다.

### 수업 후

- 해당 날짜의 강의 기록 페이지 자동 생성
- 연결된 강의자료, 수업 기록, 질문 및 교수자 답변 저장
- AI가 강의 핵심 내용과 질문·답변을 요약
- 수업 중 완료된 각 음성 Answer를 후처리에서 AI가 자동 정리하고, 원래 음성 범위·교수자 작성 text와 구분해 저장·표시
- 교수자와 학생에게 해당 강의 기록을 기반으로 한 동일한 `REVIEW` AI 채팅 제공
- 교수자가 `COMPLETED` class의 미답변 학생 질문에만 새 text-only Answer를 추가하고, 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` Answer의 text를 추가·수정
- 같은 날짜에 여러 class가 있으면 각 class의 실제 시작 시각으로 기록을 구분
- 교수자가 `COMPLETED` class의 강의자료 PDF를 추가·삭제
- 종료한 교수자 탭의 로컬 녹음을 재개 가능한 방식으로 업로드해 원본 녹음으로 저장하고 완료 후 HQ STT 실행
- HQ STT로 전체 녹음을 다시 처리하고 문장 분할·시작/종료 시각·녹음 위치를 정렬한 뒤 canonical Transcript로 전환
- HQ Transcript를 canonical로 전환한 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증하고 최종 Summary 수행
- Transcript 후처리와 독립적으로, 종료 시점까지의 학생 실제 질문과 답변이 완료된 AI 대표질문을 현재 중앙 여부와 무관하게 처음부터 최종 클러스터링한다. 성공 결과에는 각 대상 입력이 정확히 한 번 포함되고 분류할 수 없는 입력은 명시적인 `기타` Cluster에 배치된다.
- 재생 권한을 다시 확인한 뒤 저장된 강의 음성을 듣고 canonical Transcript 문장으로 해당 시점 이동
- `/record`는 기록 요약·처리 상태·영역별 개수·공개 조회 경로만 주는 compact manifest로 사용하고, Material·Transcript·질문·Answer·Cluster·공유 Job 목록은 해당 경로에서 필요한 영역부터 나눠 조회
- Transcript·질문·Answer·Cluster는 각각 독립적인 cursor pagination으로 점진 로딩하고, AI Evidence는 정확히 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`인 `source_kind`·안전한 `label`·안정적인 공개 `link`만 표시

## 5. 사용 시나리오

1. 인증 사용자가 Course를 만들어 유일한 교수자 owner가 되고 학생에게 참여 코드를 공유한다.
2. 교수자가 날짜별 강의 세션을 생성하고 필요하면 강의자료 PDF를 업로드한다. Course에 `READY`, `LIVE`, `PROCESSING` class가 있으면 다음 class를 만들지 않는다.
3. 학생이 수업 코드로 참여하고 오늘의 세션에 입장한다.
4. 교수자는 PDF가 없어도 수업을 시작할 수 있다. 연결된 PDF 중 처리 중인 자료가 있으면 완료·실패 또는 삭제를 기다리며, 시작하면 같은 마이크 입력이 실시간 STT와 브라우저 로컬 녹음으로 분기된다.
5. 교수자와 학생은 같은 실시간 강의 기록·개인 AI 영역에서 요약과 Chat을 사용해 현재 맥락을 확인한다. 각 개인 AI 결과는 요청자에게만 보인다.
6. 수업 중 학생은 익명 질문을 작성하거나 기존 질문에 반응한다.
7. 질문이 저장될 때마다 시스템이 클러스터링 필요 상태를 기록한다. active·retry 예약이 모두 없으면 fresh Job을 시작하고, 실행 중이거나 같은 실패 행의 재시도가 예약돼 있으면 새 질문을 watermark에 합쳐 다음 실행에서 한꺼번에 처리한다.
8. 교수자는 실시간 질문 마인드맵의 AI 대표질문과 각 학생 질문을 별도로 선택해 음성으로 답변한다. 대표질문에 답해도 하위 질문은 자동으로 답변 완료가 되지 않는다.
9. 수업 종료가 확정되면 Session은 즉시 `PROCESSING`이 되어 새 입력을 막고, 로컬 녹음의 재개 가능한 업로드와 원본 저장 완료 뒤 전체 녹음 기반 HQ STT를 시작한다.
10. HQ STT는 문장을 나누고 각 문장의 시작·종료 시각과 녹음 위치를 정렬해 Segment를 저장하고 canonical로 전환한다. 그 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증한다.
11. 후처리 coordinator는 LIVE에서 완료된 각 음성 Answer마다 공유 `ANSWER_ORGANIZATION` Job을 자동 생성한다. 성공한 HQ 재매핑 범위를 우선 사용하되 사용할 수 없으면 선택 당시부터 보존한 원본 LIVE 범위로 정리하며, Answer 하나의 실패가 다른 Answer 정리를 막지 않는다.
12. `PROCESSING`에서는 남아 있는 LIVE 클러스터링 결과를 반영하지 않고, 종료 시점 질문 sequence 이하의 학생 실제 질문 전체와 종료 시각까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 처음부터 최종 클러스터링한다. 학생 질문 상한은 같은 Job의 모든 attempt에서 고정하고 미답변 AI 대표질문은 제외한다. 성공한 FINAL 결과에는 각 대상 입력이 정확히 한 번 포함되며 분류할 수 없는 입력은 명시적인 `기타` Cluster에 둔다. FINAL Job이 `FAILED`한 뒤 교수자가 명시적으로 재시도하면 재시도 시점까지 새로 답변된 AI 대표질문을 포함하도록 Answer 시각 상한만 다시 캡처한다.
13. 재생 권한이 확인된 사용자는 저장된 강의 음성을 듣고 canonical Transcript 문장을 선택해 해당 시점으로 이동하며, 현재 연결된 자료·강의 기록·질문과 답변으로 복습한다. 교수자와 학생 모두 `COMPLETED` class에서 동일한 개인 `REVIEW` Chat을 사용할 수 있다. 교수자는 공통 화면의 추가 control로 미답변 학생 질문에만 새 text-only Answer를 등록한다. 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` Answer에는 text를 추가·수정할 수 있지만, FINAL·REVIEW 전용 또는 미답변 `ACTIVE` AI 대표질문에는 새 text-only Answer를 만들지 않는다. 교수자 text가 있으면 이를 우선 표시하고 AI 정리 결과는 별도 label로 구분한다.

## 6. 1차 MVP 범위

### 필수 기능

1. 인증 사용자의 Course 생성과 유일 교수자 owner 등록
2. 인증 사용자의 참여 코드 입력과 Course 학생 등록
3. 날짜별 강의 세션 생성과 Course별 단일 active class 제약
4. `READY`·`LIVE`·`COMPLETED` class의 강의자료 PDF 추가·삭제와 처리 상태 표시
5. 교수자 음성의 스트리밍 전송 및 실시간 STT 처리
6. STT 임시(partial) 결과의 실시간 표시와 확정(final) Transcript 저장
7. 교수자·학생 공통의 최근 Transcript 기반 개인 LIVE Summary·Chat
8. 익명 질문 작성 및 실시간 반영
9. 질문에 ‘나도 궁금해요’ 반응
10. 질문 저장 연동 자동 LIVE 클러스터링·coalescing과 AI 대표질문 생성
11. 학생 질문·AI 대표질문별 교수자 음성 Answer 기록과 취소
12. 수업 종료 후 강의 기록 페이지 생성
13. 교수자·학생 공통의 `COMPLETED` 기록 기반 개인 `REVIEW` Chat
14. Course owner의 참여 코드 회전과 Course 삭제
15. 모든 class 상태의 제목 수정과 상태별 class 삭제
16. 동일 마이크 입력의 실시간 STT 전송과 브라우저 로컬 녹음 이중 경로
17. 첫 `audio.start` 성공 탭의 단일 publisher와 동일 publisher 재연결·재개
18. 종료 후 로컬 녹음의 재개 가능한 업로드·원본 저장과 HQ STT 처리
19. 권한 재확인 기반 저장 음성 playback
20. Transcript 문장 선택을 통한 녹음 시점 이동
21. 수업 전체 녹음의 HQ STT 재처리와 문장 시작·종료 시각·녹음 위치 정렬
22. 영구 Transcript 상태와 version·canonical 전환
23. canonical Transcript 기반 기존 Answer 시간 범위·AI 검색 데이터 재연결
24. Segment와 누락 시간 구간(Transcript gap)을 시간 순서대로 함께 표시
25. 수업 종료 후 각 대상 입력을 정확히 한 번 포함하고 미분류 입력을 `기타` Cluster에 두는 FINAL 클러스터링과 최종 마인드맵 생성
26. `COMPLETED` class에서 미답변 학생 질문에만 교수자 text-only Answer 등록, 학생 질문 또는 Answer로 보존된 AI 대표질문의 기존 `COMPLETED` Answer text 추가·수정
27. 수업 중 완료된 음성 Answer별 자동 AI 정리, 원본 음성 범위 보존과 Answer별 실패·재시도
28. compact `/record` manifest와 Transcript·질문·Answer·Cluster의 독립 cursor pagination
29. 정확히 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`인 `source_kind`를 사용하고 내부 KnowledgeChunk·storage key·Segment ID를 노출하지 않는 AI Evidence 표시·공개 리소스 이동
30. 개인 AI의 polling-only 실행·요청자 전용 조회와 LIVE 종료 시 데이터 제거. 정리 전 polling/resource·멱등 replay 응답 의미는 미정
31. LIVE·REVIEW Chat USER 입력의 trim·Unicode NFC 후 Unicode code point 1~2,000자 검증

### Course·class 관리 규칙

- Course에는 유일한 교수자 owner 한 명만 있으며 owner 이전은 제공하지 않는다.
- Course는 종료·보관 상태 없이 owner가 삭제한다. active class가 있을 때의 삭제 허용 여부와 삭제된 Course 복구 정책은 미정이다.
- Course당 `READY`, `LIVE`, `PROCESSING` class의 합계는 최대 하나다. 이 active class 하나를 `current_session`으로 표시하고, `COMPLETED`가 된 뒤에만 다음 class를 생성한다.
- 같은 `lecture_date`의 class를 여러 개 만들 수 있다. 완료 기록은 `started_at` 내림차순으로 정렬하고 실제 시작 시각을 함께 표시한다.
- class 제목은 모든 상태에서 수정할 수 있다. 빈 제목은 서버가 `Course 제목 · YYYY.MM.DD HH:mm` 형식으로 바꾼다. 날짜는 `lecture_date`, 시각은 Session `created_at`을 기본 timezone `Asia/Seoul`로 표시한 값이며 이후 상태가 바뀌어도 다시 계산하지 않는다. `lecture_date`, `started_at`, `ended_at`, `completed_at`은 생성·전이 후 사용자가 수정하지 않는다.
- `READY`와 `COMPLETED` class는 owner가 삭제할 수 있다. `LIVE`는 먼저 종료하고 `PROCESSING`이 끝나 `COMPLETED`가 된 뒤 삭제하며, `PROCESSING`에서는 완료를 기다린다.
- 참여 코드는 앞뒤 공백을 제거하고 대문자로 정규화한 `[A-Z]{6}`이며 구분자를 허용하지 않는다. 기본 만료는 없고 owner만 회전할 수 있으며, 회전 즉시 기존 코드를 무효화하고 회전 감사·이력을 보관하거나 제공하지 않는다.

### 강의자료 관리 규칙

- PDF가 0개여도 class를 시작할 수 있다. 연결된 Material 중 `PROCESSING`이 하나라도 있으면 시작할 수 없고, 자료가 없거나 `READY`, `UPLOADED`, `FAILED`만 있으면 시작할 수 있다.
- `UPLOADED`와 `FAILED` Material은 class 시작을 막지 않지만 AI 참고 자료로 사용하지 않는다. AI는 현재 연결된 `READY` Material만 참고한다.
- Course owner는 Session이 `READY`, `LIVE`, `COMPLETED`일 때 PDF를 업로드하거나 삭제할 수 있고, Session `PROCESSING`에서는 두 동작을 할 수 없다.
- 한 Session에 연결된 active PDF는 최대 10개이며 파일 하나의 최대 크기는 decimal `100,000,000 bytes`다. 동일한 내용의 PDF를 다시 업로드할 수 있다.
- 표시 이름이 겹치면 확장자를 제외한 basename 뒤에 ` (1)`, ` (2)`를 안정적으로 붙인다. 삭제 후 기존 표시 이름을 재번호화하지 않는다.
- 삭제가 확정되면 자료는 즉시 목록·열람·새 AI 검색에서 분리된다. 실제 object와 Knowledge 자료는 백그라운드에서 정리하며 내부 storage key와 정리 상태는 사용자에게 노출하지 않는다.
- 연결된 `UPLOADED`, `PROCESSING`, `READY` 자료는 권한을 다시 확인한 뒤 원본 PDF를 열 수 있다. `FAILED` 자료는 사용할 수 있는 업로드 자료로 보이지 않게 실패를 명확히 표시하고 원본 열람을 제공하지 않는다. Material detach 전에 생성한 AI Evidence는 안전한 `label` snapshot을 유지하되 `link=null`로 원문 이동을 제공하지 않는다. 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이다.

### 질문·클러스터링·Answer 규칙

- 학생 질문과 질문 초안은 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 각각 1~300자, 1~500자다. 공백뿐이거나 제한을 넘으면 잘라 저장하지 않고 `422 VALIDATION_ERROR`로 거부하며, 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 제공한다.
- AI 질문 문장 작성 도움 초안은 같은 기준의 1자 이상 500자 이하다. 500자 초과는 잘라서 전송하지 않고 입력 오류로 거부한다. 요청은 AIJob을 만들지 않고 동기 `200` 제안을 반환하며, 제안은 300자 이하의 질문으로 정리한다. 초안과 제안을 자동으로 공개 질문으로 저장하지 않는다.
- 질문 commit은 Session의 클러스터링 필요 watermark를 증가시킨다. Session당 `PENDING` 또는 `RUNNING`인 클러스터링 Job은 합쳐서 최대 하나다.
- `active_job_id`와 `retry_job_id`가 모두 없을 때만 시스템이 새 Job을 자동 생성한다. 실행 중 들어온 질문은 별도 Job을 즉시 만들지 않고 watermark에 coalescing하며, 현재 Job이 terminal이 된 뒤 누적된 새 질문을 다음 Job 하나에서 처리한다.
- `active_job_id=null`, `retry_job_id`가 있는 실패 Job은 scheduler가 같은 행의 `attempt + 1`로 다시 실행할 예약이다. 이 예약이 해제되기 전에는 새 질문이 들어와도 fresh Job을 만들지 않고 watermark만 갱신한다.
- LIVE Job은 새 질문만 기존 Cluster에 배치하거나 새 Cluster를 만든다. 기존 질문은 다른 Cluster로 옮기지 않고, 새 질문이 배치된 Cluster의 중앙 AI 대표질문만 새 immutable 문장으로 생성해 교체한다. Cluster 배치와 대표질문 생성이 모두 끝나야 해당 Job이 완료된다.
- 같은 semantic Cluster의 공개 ID는 LIVE generation이 바뀌어도 유지하고 새 Cluster만 새 ID를 받는다. 모든 입력을 다시 배치하는 FINAL 결과는 새 Cluster ID를 사용한다.
- AI 대표질문의 공개 응답과 화면 provenance에는 해당 문장이 만들어진 generation을 나타내는 `created_in_generation`을 포함한다. 교체된 과거 대표질문에 `CAPTURING` 또는 `COMPLETED` Answer가 있으면 새 대표질문의 Cluster 자식으로 보존한다. Answer가 없으면 공개 마인드맵·단건 조회·새 AI 검색에서 즉시 폐기한다. 기존 Evidence가 참조하지 않으면 hard delete하고, 참조 중이면 Evidence provenance만 지키는 내부 tombstone으로 남겨 `label`만 표시하고 `link=null`로 이동을 막는다. 마지막 Evidence를 삭제하는 transaction에서 tombstone도 원자적으로 정리한다. 캡처 중인 대표질문은 완료 또는 취소 결과가 확정될 때까지 유지한다.
- 마인드맵은 AI 대표질문을 Cluster 중앙에, 학생 실제 질문과 보존된 과거 대표질문을 typed child로 표시한다. Cluster member 공개 discriminator는 `source_kind=STUDENT_QUESTION|AI_REPRESENTATIVE`를 정확히 사용한다. Cluster는 `ordinal ASC`, 동률이면 `id ASC`로 안정적으로 표시하고 마지막 클러스터링 Job의 유형·상태·attempt를 함께 식별한다.
- Answer는 학생 질문 또는 AI 대표질문 하나만 target으로 갖는다. 대표질문에 답해도 Cluster child는 자동으로 `ANSWERED`가 되지 않으며, 각 target은 취소되지 않은 Answer를 최대 하나만 갖는다.
- LIVE에서 교수자는 target 하나를 선택해 음성 Answer 캡처를 시작하고, 완료하면 해당 Transcript 범위와 연결한다. LIVE에서는 text Answer 작성이나 음성 Answer의 교수자 text 편집을 제공하지 않는다. 캡처를 취소하면 생성 중이던 Answer를 삭제하고 `CANCELLED` 기록을 보관·노출하지 않는다.
- 완료된 음성 Answer는 LIVE에서는 `NOT_STARTED`, PROCESSING에서 HQ source·재매핑 선택이 끝나기 전에는 Job ID 없는 `WAITING_SOURCE`로 표시한다. source가 terminal이 되면 후처리 coordinator가 각 Answer마다 `visibility=SHARED`, `blocks_session_completion=true`인 `ANSWER_ORGANIZATION` Job을 하나씩 자동 생성한다. Answer별로 독립 실행하고, 성공·실패 terminal 상태가 되어야 해당 Job의 완료 차단이 끝난다.
- AI 정리 Job을 생성할 때 재매핑이 성공했으면 HQ 범위를 우선하고, 재매핑이 없거나 실패했으면 Answer 생성 때 고정해 둔 원본 LIVE Transcript 범위를 선택해 immutable 입력으로 저장한다. 후처리와 canonical 전환 뒤에도 원본 LIVE 범위는 지우거나 덮어쓰지 않는다.
- AI 정리 결과는 교수자가 작성한 text와 별도 데이터로 저장한다. 완료 기록에서는 교수자 text가 있으면 이를 기본 답변으로 우선 표시하고, AI 결과는 “AI로 정리한 답변”처럼 별도 label로 구분하며 원본 음성 범위는 항상 다시 열 수 있게 한다.
- Answer 하나의 AI 정리 실패는 다른 Answer와 저장 기록을 가리지 않는다. Course 교수자는 실패한 같은 Job 행을 `attempt + 1`로 재시도할 수 있고 Session은 terminal 실패를 포함해 완료되며, 이미 `SUCCEEDED`한 정리 결과는 MVP에서 다시 생성하지 않는다.
- 필수 정리 Job 또는 성공 결과 원장이 누락된 경우 해당 Answer를 숨기지 않고 `DATA_INTEGRITY_ERROR`로 분리 표시한다. 이는 AI 생성 실패와 다른 서버 데이터 오류이며 사용자에게 내부 원문을 노출하지 않는다.
- `PROCESSING`으로 전이하면 실행 중이던 LIVE 클러스터링과 retry 예약을 Session·attempt fence로 막고 늦은 결과도 commit하지 않는다. 이 LIVE Job을 공개 terminal에서 `CANCELLED|SUPERSEDED`로 표현할지, 재시도 불가능한 `FAILED`로 표현할지는 미정이다. FINAL Job은 종료 시점 질문 sequence 이하의 학생 실제 질문 전체와 종료 시각까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 입력으로 삼아 처음부터 재배치한다. 성공 결과는 eligible 입력마다 member를 정확히 하나만 만들고 분류 불가능한 입력을 명시적인 `기타` Cluster에 둔다. 성공한 FINAL Job은 입력 snapshot을 고정한다. FINAL Job이 `FAILED`한 뒤 명시적으로 재시도할 때만 재시도 시점까지 새로 답변된 AI 대표질문을 포함하도록 Answer 시각 상한을 다시 캡처한다. 완료 뒤 추가한 text Answer로 성공한 FINAL 마인드맵을 다시 만드는 정책은 미정이다.
- FINAL 대상 학생 질문과 답변된 AI 대표질문이 모두 0건일 때 Job을 생략할지, 빈 성공 원장을 만들지와 `final_generation` 표현은 미정 사항으로 남긴다.
- `COMPLETED` class에서 교수자는 미답변 `STUDENT_QUESTION`에만 새 text-only Answer를 만든다. 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` 음성·text-only Answer에는 text를 추가·수정할 수 있다. FINAL·REVIEW 전용 또는 미답변 `ACTIVE` AI 대표질문에는 새 text-only Answer를 만들지 않는다. text 최대 길이와 삭제 정책은 미정이다.
- Cluster 목록과 child 목록은 기본 20·최대 100개의 cursor pagination API로 조회한다. 정렬은 고정하지만 generation 교체 중 기존 cursor 만료·재시작 정책과 큰 마인드맵의 preload page 수·점진 loading·자동 축소·탐색 UX는 후속 계약으로 남긴다.
- AI Answer 정리 결과의 외부 object shape는 API 계약을 사용하며, `content` 형식·최대 길이·모델·prompt·품질 기준과 성공 결과를 KnowledgeChunk에 재색인하는 방식은 미정이다.

### 녹음·HQ STT·playback 규칙

- 교수자 마이크 입력 하나를 MVP v1 실시간 WebSocket용 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk와 브라우저 로컬 녹음 경로로 동시에 분기한다.
- 실시간 전송과 로컬 녹음은 독립 상태로 관리한다. 한 경로의 실패가 다른 경로와 질문·이미 저장된 final Transcript를 자동 실패 처리하지 않는다.
- Session에서 처음 `audio.start`에 성공한 교수자 탭만 active publisher가 된다. 두 번째 탭의 publisher 시도는 음성 전송만 거부하고 Transcript·질문 등 수업 조회는 유지한다.
- active publisher는 같은 `client_stream_id`로 재연결하고 수락된 위치부터 resume할 수 있다. active publisher 탭을 닫거나 이탈하려 할 때 녹음·전송 중단 가능성을 경고한다.
- 종료 요청이 확정되면 Session은 즉시 `PROCESSING`으로 전환해 새 실시간 입력을 막는다. 교수자 브라우저의 로컬 녹음은 재개 가능한 업로드로 전달해 원본 녹음으로 저장하고, 업로드 완료 뒤 HQ STT 후속 처리를 시작한다.
- playback은 요청 시점의 권한을 서버에서 다시 확인한 뒤 제공한다. 권한이 확인된 사용자는 Transcript 문장을 선택해 서버가 제공한 녹음 위치로 이동할 수 있다.
- 실시간 화면의 연결·STT 표시 상태와 별개로 각 영구 Transcript version은 `FINALIZING`, `FINALIZED`, `FAILED`, `EMPTY` 중 하나의 상태를 가진다. Session 시작 뒤 live final Segment를 담는 `LIVE` version과 녹음 upload 완료 뒤 전체 녹음을 재처리하는 `RECORDING` version을 구분한다. 정상 처리됐지만 Segment가 0개이면 `EMPTY`, 처리에 실패하면 `FAILED`가 된다.
- `LIVE` version은 수업과 종료 drain 동안 `FINALIZING`이고 drain이 끝나 Segment가 있으면 `FINALIZED`, 없으면 `EMPTY`가 된다. class 시작 때 실시간 기본 조회용 canonical 포인터가 이 version을 가리키지만, `LIVE` version의 `FINALIZED`만으로 고품질 Final Transcript가 완성된 것은 아니다. 이후 `RECORDING` version 성공 결과가 포인터를 교체하며, HQ 실패 시 보존된 LIVE 포인터를 완료 기록의 final source로 인정할지는 미정이다.
- 사용자에게 고품질 Final Transcript 완료로 표시하는 조건은 전체 녹음 재처리, Segment 저장, 문장별 시작·종료 시각과 녹음 위치 매핑, `RECORDING` version의 `FINALIZED`와 canonical 전환이 모두 성공한 상태다. STT 내용이 100% 정확하다는 뜻은 아니며 교수자의 Transcript 수정 기능은 MVP 이후로 둔다.
- canonical 전환 뒤 기존 Answer의 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증한다. Answer 재매핑은 `PENDING`, `SUCCEEDED`, `FAILED`를 구분하고 전체 범위를 연결한 경우에만 새 Segment 범위를 사용한다. 재매핑 허용 오차와 일부 범위만 연결됐을 때의 세부 상태는 미정이다.
- Transcript 조회는 문장 Segment와 만들 수 없었던 시간 구간인 gap을 시간 순서대로 함께 제공한다. gap은 녹음 또는 STT 처리 일부가 유실되어 Transcript를 만들 수 없는 구간이며, 화면에서는 예를 들어 `[05:20~05:35] 연결 문제로 기록되지 않은 구간입니다.`로 표시한다.

### 완료 기록 조회·Evidence 규칙

- `/record`는 화면 초기화용 compact manifest다. Session·Recording·Transcript·Summary·후처리의 요약 및 상태, 영역별 개수, 권한 검사를 거치는 공개 조회 경로만 반환하고 Material·Transcript·질문·Answer·Cluster·공유 Job 전체 배열을 포함하지 않는다.
- manifest의 상태·개수는 응답 시점의 값이지 전체 목록을 고정하는 snapshot token이 아니다. `PROCESSING`에서 Job·Transcript·후처리 상태나 개수가 바뀌면 `/record`를 다시 조회해 manifest를 갱신한다.
- 화면은 manifest를 먼저 받은 뒤 표시할 영역의 공개 경로를 이용해 각 목록을 독립적으로 점진 로딩한다. 한 영역의 로딩·실패·재시도가 이미 열린 다른 영역을 초기화하거나 가리지 않는다.
- Transcript·질문·Answer·Cluster 목록은 각각 고정된 정렬과 불투명 cursor를 사용하며 `limit`은 기본 20, 최대 100이다. 현재 페이지를 유지한 채 다음 cursor 요청만 재시도하고, Cluster generation 교체 중 기존 cursor의 만료·재시작 세부 정책은 미정으로 남긴다.
- Transcript 한 페이지는 같은 version의 시간축을 나타내는 `segments[]`와 `gaps[]`를 나눠 받되 두 배열의 항목 합계는 요청 `limit`을 넘지 않는다. 화면은 두 배열을 시작 시각의 안정적인 순서로 merge해 하나의 `next_cursor`를 사용하는 타임라인으로 표시한다.
- AI Evidence의 공개 `source_kind`는 정확히 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`이다. `QUESTION`은 학생 질문과 AI 대표질문을 모두 포괄하고 사용자용 `label`과 공개 link 대상이 둘을 구분한다. `TRANSCRIPT` link는 Segment ID가 아니라 Session·Transcript version과 안정적인 sequence·시간 범위를 locator로 사용해 해당 문장 또는 구간으로 이동한다. 모든 `link`는 현재 권한을 다시 검사하며 `/record` 배열 위치·pagination cursor에 의존하지 않는다. 내부 `knowledge_chunk_id`·storage key·물리 경로는 응답과 화면에 노출하지 않는다.
- 근거가 표시되면 `label`과 이동 동작을 함께 제공하고, 이동 실패는 Chat 결과를 지우지 않는 독립 오류로 표시한다. detached Material 또는 폐기된 AI 대표질문을 가리키는 기존 Evidence는 안전한 `label` snapshot을 유지하고 `link=null`로 이동 동작을 비활성화한다. 폐기된 대표질문은 새 검색에서 제외하고 마지막 Evidence를 삭제하는 transaction에서 함께 정리한다. Material Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이다.

### 개인 AI Summary·Chat 규칙

- Course 교수자와 학생은 역할 구분 없이 같은 개인 AI UI를 사용한다. `LIVE`에서는 개인 Summary·Chat을, `COMPLETED`에서는 개인 `REVIEW` Chat을 제공한다. `PROCESSING`에서는 `REVIEW` Chat을 제공하지 않고 기록 정리 상태만 표시한다.
- 개인 AI 요청은 `202 Accepted`와 요청자 전용 AIJob을 받은 뒤 Job을 polling하고, 성공하면 저장된 최종 Summary 또는 assistant Message를 REST로 다시 조회한다. token delta, SSE, streaming HTTP와 private WebSocket은 제공하지 않고 shared Session WebSocket에는 개인 Job 상태·결과를 보내지 않는다.
- Chat 요청을 받으면 새로고침 복구를 위해 USER Message와 그 Message를 입력으로 고정한 `CHAT_RESPONSE` Job·outbox를 `202 Accepted` 응답 전에 하나의 transaction으로 반드시 저장한다. 같은 Job을 재시도해도 이 USER Message를 그대로 사용한다. 생성 중인 assistant 부분 문장은 저장·공유하지 않고, `SUCCEEDED`한 최종 assistant Message·Evidence와 성공한 Summary 결과만 저장한다.
- LIVE Summary·Chat·Message·Evidence와 이에 연결된 요청자 전용 Job은 Session이 `LIVE`에서 `PROCESSING`으로 전이할 때 모두 사라지며 FINAL Summary와 `COMPLETED`의 `REVIEW` Chat에는 영향을 주지 않는다. 정리 전에 받은 polling·resource 식별자의 이후 응답과 같은 `Idempotency-Key` replay 응답 의미는 미정이므로 `404`나 이전 성공 응답 재생을 고정하지 않는다.
- LIVE 화면은 `session.updated=PROCESSING`을 받으면 선택한 개인 Job·Chat 식별자와 화면 cache를 즉시 비우고 기록 정리 화면으로 이동한다. 클라이언트에 남은 예전 응답을 복원하거나 완료 기록으로 승격하지 않는다.
- LIVE·REVIEW Chat의 USER `content`는 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 1자 이상 2,000자 이하다. 공백뿐이거나 초과한 입력은 자르지 않고 `422 VALIDATION_ERROR`로 거부하며, 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 제공한다. 화면은 입력 원문을 유지하고 정규화 결과의 글자 수와 오류를 표시한다. 이 제한은 assistant Message 길이에 적용하지 않는다.

### Transcript 기반 Summary 규칙

- `READY`와 Summary 요청 전 `LIVE`의 정상 무요약 상태는 `summary_status=NOT_STARTED`, `summary_reason=null`, `summary=null`, `job_id=null`이다.
- `LIVE`에서 저장된 live final Segment가 0개이면 Summary Job을 만들지 않고 `409 SUMMARY_TRANSCRIPT_NOT_READY`와 “아직 확정된 강의 내용이 없습니다. 잠시 후 다시 시도해 주세요.”를 표시한다.
- `PROCESSING`에서 HQ source gate가 아직 끝나지 않았으면 `summary_status=PENDING`, `summary_reason=null`, `summary=null`, `job_id=null`이 정상이다. 최신 HQ `RECORDING` version이 `FINALIZED`이고 Segment가 1개 이상이 되는 transaction에서 coordinator는 FINAL Summary Job을 원자적으로 생성한다. 이 조건을 충족했는데 Job이 없으면 `DATA_INTEGRITY_ERROR`다.
- RECORDING 처리가 정상 종료됐지만 Segment가 0개인 `EMPTY`이면 Job 없이 `summary_status=NOT_APPLICABLE`, `summary_reason=NO_FINAL_TRANSCRIPT`로 기록하고 “요약할 강의 내용이 없습니다.”를 표시한다.
- Recording source 자체가 없으면 즉시, Recording은 있지만 HQ source가 `FAILED`이거나 10분 제한까지 결과가 없으면 Job 없이 `summary_status=FAILED`, `summary_reason=SUMMARY_SOURCE_UNAVAILABLE`로 처리하고 “Transcript 처리 문제로 요약을 만들지 못했습니다.”를 표시한다.
- `PROCESSING`의 FINAL Summary Job은 최초 실행이나 실패 뒤 재시도 attempt와 무관하게 `PENDING|RUNNING`일 수 있다. `COMPLETED`의 active Job은 실패 뒤 명시적으로 재시도한 `attempt > 1`이거나, 성공한 HQ 재시도 뒤 `attempt > 1`인 `SESSION_POSTPROCESSING` coordinator가 새 source에 생성한 FINAL Summary일 때 정상이다. 최초 종료 후처리의 coordinator `attempt=1`에서 만든 FINAL Summary가 `attempt=1` active로 남은 경우는 `DATA_INTEGRITY_ERROR`다.
- `SUCCEEDED` FINAL Summary Job에 저장 결과가 없거나, `COMPLETED`인데 Segment가 1개 이상인 `FINALIZED` source에 FINAL Summary Job이 없으면 `DATA_INTEGRITY_ERROR`다. 화면은 내부 원인을 노출하지 않고 “요약 상태를 확인할 수 없습니다. 잠시 후 다시 시도해 주세요.”라고 안내한다.
- 보존된 LIVE 포인터를 final source로 인정할지 결정되기 전에는 이를 자동 FINAL Summary에 사용하지 않는다.

### 후처리 종료 보장 규칙

- Worker heartbeat는 15초마다 lease를 갱신하고 60초 동안 갱신이 없으면 중단으로 판단한다. 이 heartbeat·lease 값은 고정한다. 일반 후처리 Job의 기본 실행 상한은 5분이지만 HQ STT `RECORDING_TRANSCRIPTION`은 이 기본값에서 제외하며 HQ 개별 Job timeout은 미정이다. Session 전체 `PROCESSING` 제한은 `ended_at`부터 10분으로 유지한다.
- 제한을 넘긴 Job은 `FAILED`로 종료한다. 10분 제한에서 아직 시작하지 못한 필수 후속 작업도 재시도 가능한 `FAILED` 항목으로 기록한 뒤, Session 완료를 막는 모든 Job이 `SUCCEEDED` 또는 `FAILED`가 되면 실패가 있어도 Session을 `COMPLETED`로 전환한다. 늦게 도착한 이전 Worker 결과는 저장하지 않는다.
- LIVE에서 완료된 음성 Answer별 `ANSWER_ORGANIZATION`도 완료 차단 공유 Job으로 포함한다. Answer 하나가 실패해도 `FAILED` terminal로 종료해 다른 Answer와 Session 완료를 막지 않으며, Course 교수자가 완료 기록에서 같은 행을 `attempt + 1`로 재시도해도 Session을 다시 `PROCESSING`으로 되돌리지 않는다.
- `COMPLETED`에서 실패한 HQ STT를 재시도해 새 RECORDING version이 성공하면 같은 `SESSION_POSTPROCESSING` coordinator 행을 `attempt + 1`로 다시 queue한다. Session은 `COMPLETED`를 유지한 채 Answer mapping과 canonical Knowledge link를 다시 만들고 eligible하면 `FINAL_SUMMARY`를 생성한다. 기존 `ANSWER_ORGANIZATION` Job·결과는 새 HQ 범위로 조용히 재생성하거나 rebound하지 않는다.
- 실패한 AI 기능은 완료 기록에서 따로 표시하고 나중에 재시도하되 Session을 다시 `PROCESSING`으로 되돌리지 않는다. 저장된 PDF·Transcript·질문·답변과 원본 LIVE 음성 범위는 실패와 관계없이 열람 가능한 범위에서 제공한다.

### 후처리·녹음 미정 사항

- HQ STT `RECORDING_TRANSCRIPTION` 개별 Job timeout
- `LIVE → PROCESSING`에서 fence된 LIVE clustering의 공개 terminal 표현: `CANCELLED|SUPERSEDED` 또는 `retryable=false`인 `FAILED`
- 종료 때 사라진 LIVE Summary·Chat·Message·Evidence·Job의 기존 polling/resource 요청 응답과 같은 `Idempotency-Key` replay 의미
- 저장 녹음의 codec·container와 브라우저 local storage 방식
- resumable upload protocol·checksum·만료·최대 크기와 fragment manifest
- publisher lease·재획득·takeover 정책
- `DEGRADED`, stop timeout·강제 종료, audio replay와 live gap 생성 기준
- 녹음 동의, 역할별 접근, 보관 기간과 삭제 정책
- HQ STT 실패 시 보존된 LIVE 포인터를 완료 기록의 final source로 인정할지 여부
- Answer 시간 범위 재매핑 허용 오차와 부분 실패 상태

### 후순위 기능

- 강의 음성 원본 다운로드
- 교수자 음성과 학생 음성 자동 분리
- 수업 전체 자동 챕터링
- 한 학기 전체 기록 통합 검색 및 AI 채팅
- 교수자의 canonical Transcript 수정

## 7. MVP 핵심 가치

1차 MVP는 화자 분리보다 **낮은 지연의 스트리밍 STT, 브라우저 녹음 저장과 HQ STT, 수업 중 AI 요약, 익명 질문 수집 및 수업 기록·음성 기반 복습**의 효용을 검증하는 데 집중한다.

- 교수자와 학생은 같은 개인 AI UI의 실시간 Transcript·Summary·Chat으로 현재 강의 맥락을 빠르게 확인할 수 있다.
- 학생은 부담 없이 질문하고 다른 학생의 어려움도 함께 확인할 수 있다.
- 교수자는 수업 흐름을 크게 끊지 않고 질문의 우선순위를 파악할 수 있다.
- 수업이 끝나면 녹음, HQ STT 결과, 질문 및 답변이 축적되고 권한이 확인된 사용자는 문장과 음성을 함께 복습할 수 있다.
