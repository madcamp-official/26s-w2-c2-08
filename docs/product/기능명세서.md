# GOAL: God Of All Lectures 기능 명세서

## 1. 문서 목적

본 문서는 GOAL 서비스가 사용자에게 제공해야 하는 기능과 동작 규칙을 정의한다. 구현 기술과 내부 구조는 [기술명세서](../architecture/기술명세서.md)에서 별도로 다룬다.

GOAL은 강의 중 발생하는 음성, 질문과 답변을 실시간으로 기록하고, 수업 종료 후 이를 AI 기반 복습 자료로 정리하는 강의 보조 서비스이다.

## 2. 계정과 Course 역할

| 구분          | 역할                                                                                                                                                                                      |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 인증 사용자   | 계정 전역 역할 없이 Course를 생성하거나 참여 코드로 Course에 참여한다.                                                                                                                    |
| Course 교수자 | Course를 생성한 유일한 owner로서 class와 참여 코드를 관리하고 학생 질문에 답변한다. 학생과 같은 개인 LIVE Summary·Chat 및 완료 REVIEW Chat을 사용하며, Course당 한 명이고 이전할 수 없다. |
| Course 학생   | 참여 코드로 해당 Course에 참여해 강의 기록·익명 질문을 이용하고 교수자와 같은 개인 LIVE Summary·Chat 및 완료 REVIEW Chat을 사용한다.                                                      |

### 3.2 수업 중

#### Course 멤버 공통

| ID        | 기능                 | 요구사항                                                                                                                                                                                                  |
| --------- | -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| LIVE-C-01 | 개인 LIVE Summary    | Course 교수자와 학생이 같은 UI에서 현재까지 저장된 live final Transcript를 개인 요약으로 요청하고 요청자 전용 결과를 확인할 수 있다. 요청 전에는 `summary_status=NOT_STARTED`, `summary_reason=null`이다. |
| LIVE-C-02 | 개인 LIVE Chat       | Course 교수자와 학생이 같은 UI에서 현재 연결된 `READY` PDF와 현재 Transcript를 우선 근거로 개인 AI와 대화할 수 있다. 답변을 직접 뒷받침하는 강의 근거가 있으면 이를 밝히고 Evidence를 표시하며, 근거가 없거나 불충분하면 일반 지식 기반 답변임을 구분해 표시한다. 교수자 전용 수업 control은 공통 UI에 추가한다. |
| LIVE-C-03 | 개인 AI polling 복구 | 요청자는 `202 + AIJob` 뒤 Job을 polling하고 성공한 저장 결과를 REST로 조회한다. shared Session WebSocket, token delta, SSE와 private WebSocket은 사용하지 않는다.                                         |

#### Course 교수자

| ID        | 기능                   | 요구사항                                                                                                                                                                                              |
| --------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| LIVE-T-01 | 실시간 Transcript 확인 | 강의 음성이 변환된 텍스트를 시간 순서대로 확인할 수 있다.                                                                                                                                             |
| LIVE-T-02 | 익명 질문 확인         | 학생이 작성한 질문을 작성자 이름 없이 실시간으로 확인할 수 있다.                                                                                                                                      |
| LIVE-T-03 | 질문 마인드맵 확인     | `created_in_generation` provenance가 있는 AI 대표질문을 중앙에 두고 학생 질문·보존된 과거 대표질문이 `source_kind=STUDENT_QUESTION\|AI_REPRESENTATIVE` child로 연결된 Cluster를 확인할 수 있다.       |
| LIVE-T-04 | 인기 질문 확인         | ‘나도 궁금해요’ 반응이 많은 질문을 우선순위로 정렬할 수 있다.                                                                                                                                         |
| LIVE-T-05 | 답변 target 선택       | 미답변 학생 질문 또는 AI 대표질문 하나를 선택할 수 있다. Cluster 전체를 하나의 Answer target으로 삼지 않는다.                                                                                         |
| LIVE-T-06 | 음성 답변 연결         | 선택 이후 교수자가 답변한 Transcript 구간을 해당 target 하나의 Answer로 연결할 수 있다.                                                                                                               |
| LIVE-T-07 | 답변 완료·취소         | 음성 Answer를 완료하거나 취소할 수 있다. 취소한 Answer는 삭제하고 `CANCELLED` 기록을 노출하지 않는다.                                                                                                 |
| LIVE-T-08 | 강의 종료              | 강의를 종료하고 강의 기록 생성 작업을 시작할 수 있다.                                                                                                                                                 |
| LIVE-T-09 | 실시간 음성 이중 처리  | 같은 마이크 입력을 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk의 실시간 WebSocket 전송과 브라우저 로컬 녹음으로 함께 처리하고 두 경로 상태를 독립적으로 확인할 수 있다.                                   |
| LIVE-T-10 | 단일 audio publisher   | Session에서 첫 `audio.start`에 성공한 교수자 탭만 음성을 전송한다. 두 번째 publisher 탭은 전송만 거부하고 수업 조회는 유지하며, active publisher는 같은 `client_stream_id`로 재연결·resume할 수 있다. |

### 3.3 수업 후

| ID                | 기능                         | 요구사항                                                                                                                                                                                                                                                                                        |
| ----------------- | ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| POST-01           | 강의 기록 자동 생성          | 수업 종료 후 해당 날짜의 강의 기록 페이지가 자동으로 생성된다. `/record`는 영역별 전체 배열이 아닌 요약·상태·개수·조회 경로 manifest를 제공한다.                                                                                                                                                |
| POST-02           | canonical Transcript 조회    | 교수자와 학생은 같은 cursor 페이지에서 나뉘 `segments[]`와 `gaps[]`를 시간 순서대로 merge한 canonical Transcript를 확인할 수 있다.                                                                                                                                                              |
| POST-03           | AI 강의 요약 조회            | 강의 핵심 내용과 주요 개념을 정리한 결과를 확인할 수 있다.                                                                                                                                                                                                                                      |
| POST-04           | 질문·답변 정리               | 학생 질문과 AI 대표질문에 연결된 교수자 Answer를 target별로 확인할 수 있다. 교수자 text가 있으면 이를 우선 표시하고, AI 정리 결과와 원본 음성 범위는 서로 구분한다. 대표질문 Answer는 child 질문의 답변 완료를 의미하지 않는다.                                                                 |
| POST-05           | FINAL 질문 마인드맵          | 종료 시점 학생 실제 질문 전체와 그때까지 Answer가 완료된 AI 대표질문을 현재 중앙 여부와 무관하게 처음부터 재클러스터링하고, eligible 입력을 정확히 한 번씩 포함하며 미분류 입력을 `기타` Cluster에 둔 최종 마인드맵을 확인할 수 있다.                                                           |
| POST-06           | 개인 REVIEW Chat             | Course 교수자와 학생이 `COMPLETED` Session의 같은 UI에서 현재 연결된 `READY` PDF, final Transcript와 Q&A를 기반으로 개인 AI와 복습할 수 있다. `PROCESSING`에서는 제공하지 않는다.                                                                                                               |
| POST-07           | 이전 강의 목록               | 같은 날짜의 class도 실제 시작 시각으로 구분해 `started_at` 내림차순으로 확인할 수 있다.                                                                                                                                                                                                         |
| POST-08           | 강의자료 다시 보기           | 해당 세션에 현재 연결된 열람 가능한 PDF를 다시 볼 수 있다.                                                                                                                                                                                                                                      |
| POST-09           | 녹음 원본 저장과 HQ STT      | Session 종료 후 브라우저 로컬 녹음을 재개 가능한 방식으로 업로드해 원본 녹음으로 저장하고, 업로드가 완료되면 전체 녹음을 HQ STT로 다시 처리한다.                                                                                                                                                |
| POST-10           | 저장 음성 playback           | 저장 녹음의 접근 권한을 재생 요청 시 다시 확인하고 허용된 사용자에게 playback을 제공한다. 정확한 허용 역할은 미정이다.                                                                                                                                                                          |
| POST-11           | Transcript 문장 seek         | playback 권한이 확인된 사용자는 Transcript 문장을 선택해 서버가 제공한 해당 녹음 위치로 이동할 수 있다.                                                                                                                                                                                         |
| POST-12           | Transcript 시간 정렬         | HQ STT 결과를 문장으로 나누고 각 Segment의 시작·종료 시각과 녹음 재생 위치를 정렬해 저장한다.                                                                                                                                                                                                   |
| POST-13           | Transcript canonical 전환    | 영구 Transcript의 version과 `FINALIZING`·`FINALIZED`·`FAILED`·`EMPTY` 상태를 제공하고 성공한 HQ 결과를 canonical로 전환한다.                                                                                                                                                                    |
| POST-14           | 후처리 데이터 재연결         | canonical 전환 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 HQ Segment에 다시 연결해 검증하고, 성공한 HQ Segment를 최종 Summary의 원본으로 사용한다. FINAL 질문 클러스터링은 Transcript 재연결과 독립적으로 질문 target snapshot을 사용한다.                                                     |
| POST-15           | 수업 후 텍스트 답변          | Course 교수자는 `COMPLETED` class의 미답변 `STUDENT_QUESTION`에만 새 text-only Answer를 등록하고, 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` Answer에 text를 추가·수정할 수 있다.                                                                            |
| POST-16           | 음성 Answer 자동 AI 정리     | 후처리 coordinator는 LIVE에서 완료된 각 음성 Answer마다 공유 완료 차단 `ANSWER_ORGANIZATION` Job을 자동 생성한다. 성공한 HQ 재매핑 범위를 우선 사용하고 없거나 실패하면 immutable 원본 LIVE 범위를 사용하며, 결과는 교수자 text와 별도로 저장한다.                                              |
| POST-17           | 완료 기록 점진 조회          | Material·Transcript·질문·Answer·Cluster·공유 Job은 `/record`의 공개 조회 경로로 영역별 독립 로딩한다. Transcript·질문·Answer·Cluster 목록은 기본 20·최대 100의 안정적인 cursor pagination을 사용한다.                                                                                           |
| POST-18           | AI Evidence 확인             | AI 결과에서 정확히 `MATERIAL\|TRANSCRIPT\|QUESTION\|ANSWER`인 `source_kind`·사용자용 `label`을 확인하고 권한을 다시 검사하는 안정적인 공개 `link`로 해당 기록에 이동할 수 있다. `QUESTION`은 학생·AI 대표질문을 포괄하고 `TRANSCRIPT`는 Session·version·안정적인 sequence/시간 범위를 사용한다. |
| COURSE-ARCHIVE-01 | Course workspace shell       | Course 멤버는 정확히 네 archive navigation과 항상 존재하는 `LIVE CLASS` 영역, 완료 class cursor 목록을 같은 shell에서 사용한다.                                                                                                                                                                 |
| COURSE-ARCHIVE-02 | Course PDF archive           | Course 멤버는 모든 class에 현재 연결된 PDF를 class 정보와 함께 cursor로 조회하고, 사용할 수 있는 원문을 열거나 개별 다운로드할 수 있다. detached·FAILED 원문과 내부 storage 정보는 제공하지 않는다.                                                                                             |
| COURSE-ARCHIVE-03 | Course Transcript archive    | Course 멤버는 `LIVE`·`PROCESSING`·`COMPLETED` class의 공개 Transcript 상태를 cursor로 조회하고, 선택 class의 canonical 또는 live final timeline을 기존 안전한 cursor API로 점진 로딩할 수 있다.                                                                                                 |
| COURSE-ARCHIVE-04 | Course FINAL Summary archive | Course 멤버는 `PROCESSING`·`COMPLETED` class의 공용 FINAL Summary와 `PENDING\|AVAILABLE\|FAILED\|NOT_APPLICABLE\|DATA_INTEGRITY_ERROR` 상태를 class별로 확인할 수 있다. 개인 LIVE 결과는 포함하지 않는다.                                                                                       |
| COURSE-ARCHIVE-05 | Course Q&A archive           | Course 멤버는 모든 class의 미답변 학생 질문, 답변된 학생 질문과 Answer가 존재하는 AI 대표질문 target snapshot을 작성자 식별정보 없이 cursor로 확인하고 해당 class 기록으로 이동할 수 있다. 교수자 관리 control은 class 기록에만 둔다.                                                           |

### 3.4 Course·class 관리

| ID       | 기능            | 요구사항                                                                                                                                                                                                                |
| -------- | --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| MGT-T-01 | 참여 코드 회전  | Course owner는 기존 코드를 즉시 무효화하고 새 `[A-Z]{6}` 코드를 발급할 수 있다. 회전 감사·이력은 보관하거나 제공하지 않는다.                                                                                            |
| MGT-T-02 | Course 삭제     | 유일한 Course owner는 `READY`·`LIVE`·`PROCESSING` class가 없을 때만 Course를 삭제할 수 있다. Course 종료·보관·owner 이전·복구는 제공하지 않고, 성공 뒤 접근·참여 코드를 즉시 차단하며 private object를 비동기 정리한다. |
| MGT-T-03 | class 제목 수정 | Course owner는 `READY`, `LIVE`, `PROCESSING`, `COMPLETED` class의 제목을 수정할 수 있다. 빈 제목은 서버 자동 제목으로 바뀐다.                                                                                           |
| MGT-T-04 | class 삭제      | Course owner는 `READY`와 `COMPLETED` class를 삭제할 수 있다. `LIVE`는 종료한 뒤 `PROCESSING` 완료를 기다려 삭제하고, `PROCESSING`에서는 바로 삭제하지 않는다.                                                           |
| MGT-T-05 | 강의자료 관리   | Course owner는 Session `READY`·`LIVE`·`COMPLETED`에서 PDF를 추가·삭제하고 상태를 확인할 수 있다. Session `PROCESSING`에서는 추가·삭제할 수 없다.                                                                        |

## 5. 주요 동작 규칙

### 수업방과 세션

- 로그인 계정은 전역 역할을 갖지 않으며, 모든 인증 사용자는 Course를 생성하거나 참여 코드로 Course에 참여할 수 있다.
- Course당 교수자 owner는 생성자 한 명이며 추가·이전할 수 없다. Course owner만 class를 생성하고 시작·종료하며 Course와 class 관리 동작을 수행한다.
- 참여 코드는 입력의 앞뒤 공백을 제거하고 대문자로 정규한 `[A-Z]{6}`이다. 구분자를 허용하지 않고 자동 만료되지 않으며, owner만 회전할 수 있다.
- 참여 코드를 회전하면 기존 코드는 즉시 무효화하고 회전 감사·이력을 보관·표시하지 않는다.
- Course에는 `READY`, `LIVE`, `PROCESSING` 상태를 통틀어 active class가 최대 하나만 있다. Course의 `current_session`은 이 단일 active class 또는 `null`이다.
- active class가 `COMPLETED`가 된 뒤에만 다음 class를 생성할 수 있다. 동시 생성 경쟁은 하나만 성공해야 한다.
- 같은 `lecture_date`의 class를 여러 개 만들 수 있다. 완료 class 목록은 `started_at` 내림차순으로 정렬하고 날짜와 실제 시작 시각을 함께 표시한다.
- class 제목은 모든 상태에서 수정할 수 있고, trim 후 비어 있으면 서버가 `Course 제목 · YYYY.MM.DD HH:mm` 형식의 자동 제목을 지정한다. 날짜는 `lecture_date`, 시각은 생성 시 기록한 `created_at`을 기본 timezone `Asia/Seoul`로 표시한 값이며 상태 전이 뒤에도 바꾸지 않는다.
- `lecture_date`, `started_at`, `ended_at`, `completed_at`은 사용자가 수정할 수 없다.
- Course는 종료·보관 상태를 갖지 않고 owner가 삭제한다. `READY`, `LIVE`, `PROCESSING` class가 하나라도 있으면 삭제를 거부하며, `COMPLETED` class만 남은 Course는 삭제할 수 있다. 삭제 성공 뒤에는 복구를 제공하지 않고 모든 Course 접근·참여 코드를 즉시 차단한다. 물리 파일과 기록 정리는 비동기로 재시도한다.
- `READY`와 `COMPLETED` class는 owner가 삭제할 수 있다. `LIVE`는 먼저 종료하고 `PROCESSING` 완료 후 삭제하며 `PROCESSING`은 완료까지 삭제하지 않는다.
- 종료된 세션에는 새 질문이나 반응을 등록할 수 없다.
- 세션의 사용자 표시 상태는 `시작 전`, `진행 중`, `정리 중`, `완료`로 구분한다.

### 강의자료

- 한 Session에는 삭제되지 않고 연결된 active PDF를 최대 10개까지 둘 수 있다. `UPLOADED`, `PROCESSING`, `READY`, `FAILED` 상태의 연결된 Material은 모두 이 제한에 포함한다.
- PDF 하나의 최대 크기는 decimal `100,000,000 bytes`다. 동일 content의 PDF를 다시 업로드할 수 있으며 content 중복을 오류로 처리하지 않는다.
- 같은 표시 이름이 있으면 확장자를 제외한 basename 뒤에 ` (1)`, ` (2)`를 업로드 시점에 안정적으로 붙인다. 예를 들어 `강의자료.pdf`, `강의자료 (1).pdf`, `강의자료 (2).pdf`로 표시하며, 중간 자료를 삭제해도 남은 이름을 재번호화하지 않는다.
- Course owner는 Session이 `READY`, `LIVE`, `COMPLETED`일 때 PDF를 업로드하거나 삭제할 수 있다. Session `PROCESSING`에서는 업로드와 삭제를 모두 거부한다.
- class 시작 시 현재 연결된 Material 중 `PROCESSING`이 하나라도 있으면 시작을 거부한다. 연결된 자료가 없거나 `READY`, `UPLOADED`, `FAILED`만 있으면 시작할 수 있다.
- AI 참고 자료는 현재 연결된 `READY` Material로 한정한다. `UPLOADED`, `PROCESSING`, `FAILED`와 삭제된 Material의 Knowledge 자료는 새 AI 검색에 사용하지 않는다.
- 연결된 `UPLOADED`, `PROCESSING`, `READY` Material은 권한을 다시 확인한 뒤 원본 PDF를 열 수 있다. `FAILED` Material은 원본 열람·사용 가능한 업로드 자료 수·AI 참고 자료로 표현하지 않고, 업로드에 실패해 사용할 수 없음을 명확히 표시한다. 재시도 가능 여부와 삭제 제공 여부는 Job과 Session 상태 규칙을 따른다.
- 삭제 transaction이 확정되면 Material은 목록·열람과 새 AI 검색에서 즉시 분리된다. object와 Knowledge 자료의 물리 정리는 백그라운드에서 진행하며 정리 실패가 삭제를 되돌리거나 자료를 다시 노출하지 않는다.
- 중복 업로드·삭제 요청의 재실행은 공통 멱등성 규칙을 따른다. 상태 또는 version 경합으로 삭제를 확정하지 못하면 최신 목록을 조회한 뒤 다시 시도한다.
- Material detach 전에 생성된 AI Evidence는 안전한 `label` snapshot을 유지하고 `link=null`로 원문 이동을 제공하지 않는다. 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이다.

### 녹음·HQ STT·playback

- 교수자 마이크 입력은 MVP v1 실시간 WebSocket용 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk와 브라우저 로컬 녹음으로 동시에 분기한다. 실시간 wire format과 저장 녹음의 codec·container는 같은 결정이 아니다.
- 실시간 전송과 로컬 녹음은 독립 상태로 표시하고 한 경로 실패를 다른 경로의 실패로 간주하지 않는다. 남아 있는 경로와 질문·final Transcript는 계속 사용할 수 있어야 한다.
- Session에서 첫 `audio.start`에 성공한 교수자 탭만 active publisher가 된다. 이후 publisher 시도는 음성 전송만 거부하며 해당 탭의 Transcript·질문 등 읽기 기능은 유지한다.
- active publisher의 연결이 끊기면 같은 `client_stream_id`로 재연결하고 서버가 수락한 위치부터 resume할 수 있다. active publisher 탭을 닫거나 이탈하려 할 때 녹음과 실시간 전송이 중단될 수 있음을 경고한다.
- 종료 확인 즉시 client는 종료 HTTP와 Audio 중단·MediaRecorder flush/pause를 병행한다. `202`이면 Session은 이미 `PROCESSING`이며 로컬 녹음을 최종 마감해 재개 가능한 upload로 전달한다. 실패·응답 유실은 Session REST로 조정하고 `LIVE`가 확인된 경우에만 같은 MediaRecorder instance를 재개한다. upload 완료 뒤 HQ STT를 시작한다.
- 녹음 upload UI는 녹음 중, 마무리 중, 진행 중, 완료, 네트워크 실패, 만료와 로컬 저장 실패를 구분한다. 브라우저는 MediaRecorder Blob을 `localStorage`가 아닌 IndexedDB fragment에 저장하고, 서버가 확인한 연속 offset에 완전히 포함된 fragment만 삭제한다. 프로토콜은 SHA-256, 8 MiB chunk, 24시간 만료와 100,000,000 byte 전체 상한을 사용하며 물리 fragment manifest는 외부 API에 공개하지 않는다.
- 같은 기기의 audio publisher·로컬 녹음 lifecycle·녹음 upload는 목적별 session-scoped Web Lock으로 중복 탭을 직렬화한다. 새 page는 이전 MediaRecorder instance의 미완결 container에 조각을 이어 붙이지 않으며, `stop`·마지막 fragment commit 뒤의 `finalizationReady`가 없는 중단 원본은 정상 HQ 녹음으로 승격하지 않는다. upload lock 소유 탭이 사라지면 다른 PROCESSING 탭이 저장된 idempotency key와 offset으로 자동 인계한다.
- 미동의·zero-byte·로컬 저장 실패·완결성 실패를 브라우저에서 즉시 알더라도 이를 서버 Recording `FAILED`로 알릴 endpoint는 현재 없다. 이 경우 후처리 source gate가 최대 10분 watchdog을 기다리는 것은 구현 완료가 아니라 integration gap이다.
- playback은 요청할 때마다 서버가 권한을 다시 확인한다. 접근이 거부되면 녹음 control을 비활성화하되 허용된 다른 완료 기록까지 실패로 처리하지 않는다.
- Transcript 문장 seek는 표시 시각을 추정하지 않고 서버가 제공한 녹음 위치를 사용한다.
- Session 시작 뒤 live final Segment 원장인 `source=LIVE` Transcript version을 두고, 녹음 upload가 완료되면 `source=RECORDING`, `FINALIZING`인 다음 version으로 전체 녹음을 HQ STT로 다시 처리한다. 문장 분할, Segment 시작·종료 시각과 녹음 위치 정렬, Segment 저장과 canonical 전환이 모두 성공하면 `FINALIZED`가 된다.
- `LIVE` version은 수업과 종료 drain 동안 `FINALIZING`이고 drain이 끝나 Segment가 있으면 `FINALIZED`, 없으면 `EMPTY`가 된다. class 시작 때 실시간 기본 조회용 canonical 포인터가 이 version을 가리키지만 이 상태만으로 고품질 Final Transcript가 완성된 것은 아니다. HQ `RECORDING` version이 실패했을 때 보존된 LIVE 포인터를 완료 기록의 final source로 인정할지는 미정이다.
- 정상 처리가 끝났지만 Segment가 0개이면 `EMPTY`, HQ STT 또는 canonical 전환이 실패하면 `FAILED`다. version의 영구 상태 `FINALIZING`·`FINALIZED`·`FAILED`·`EMPTY`는 `LIVE` 화면의 WebSocket 연결·실시간 STT 표시 상태와 별도 원장으로 관리한다.
- 사용자에게 고품질 Final Transcript 완료로 표시하는 조건은 전체 녹음 재처리, Segment 저장, 시간·녹음 위치 매핑, `RECORDING` version의 `FINALIZED`와 canonical 전환이 모두 성공한 상태다. Transcript 문장이 100% 정확하다는 보장은 아니며 교수자의 canonical Transcript 수정은 MVP 이후 기능이다.
- canonical 전환 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증한다. Answer 재매핑은 `PENDING`, `SUCCEEDED`, `FAILED`를 구분하고 전체 범위를 연결한 경우에만 새 Segment 범위를 사용한다. 재매핑 허용 오차와 일부 범위만 연결됐을 때의 세부 상태는 미정이다.
- Transcript 응답은 한 cursor 페이지의 문장 `segments[]`와 누락 시간 구간 `gaps[]`를 나눠 제공한다. 두 배열의 항목 합계는 요청 `limit`을 넘지 않고, 화면은 같은 Transcript version의 두 배열을 시작 시각의 안정적인 순서로 merge해 하나의 시간축으로 표시한다. gap은 녹음 또는 STT 처리 일부가 유실되어 Transcript를 만들 수 없는 구간이다.
- 저장 녹음 backend는 `audio/webm`·`audio/mp4`, 100,000,000 bytes, 24시간 resumable upload를 제공한다. upload는 첫 publisher 교수자만, playback은 현재 Course 멤버만 가능하고 download endpoint는 없다. 브라우저는 녹음 동의 뒤 IndexedDB에만 audio fragment를 저장한다. `COMPLETED` 시점부터 30일이 지나면 녹음 playback을 차단하고 비동기 물리 삭제하며, Course owner는 완료 수업의 녹음을 더 일찍 삭제할 수 있다. 삭제는 복구하지 않는다.

### 익명 질문·클러스터링과 반응

- 학생의 이름과 계정 정보는 교수자와 다른 학생에게 표시하지 않는다.
- 서비스는 중복 반응 방지 등을 위해 질문 작성자와 반응 사용자를 내부적으로 식별할 수 있다.
- 공개 질문과 질문 초안은 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 각각 1~~300자, 1~~500자다. 공백뿐이거나 제한을 넘으면 묵시적으로 잘라 저장하지 않고 `422 VALIDATION_ERROR`로 거부한다. 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 제공한다.
- 질문 초안은 같은 집계 기준으로 1자 이상 500자 이하이며, 초과 입력은 자르지 않고 오류로 거부한다. AI 질문 문장 작성 도움은 AIJob 없이 동기 `200`으로 300자 이하 후보를 반환한다. 초안·후보 선택만으로 공개 질문을 자동 저장하지 않는다.
- 한 학생은 같은 질문에 ‘나도 궁금해요’를 한 번만 추가할 수 있으며 취소할 수 있다.
- 질문 commit마다 Session의 pending watermark를 갱신한다. Session당 `PENDING`·`RUNNING` 클러스터링 Job은 합쳐서 최대 하나이며 `active_job_id`와 `retry_job_id`가 모두 없을 때만 시스템이 fresh Job을 생성한다.
- `active_job_id=null`, `retry_job_id`가 있는 실패 Job은 scheduler가 같은 행을 `attempt + 1`로 재실행할 예약이다. 예약 중 새 질문은 watermark에만 합치고 fresh Job을 만들지 않는다.
- Job 실행 중 저장된 질문은 pending으로 coalescing한다. 현재 Job이 terminal이 되면 누적된 질문 전체를 다음 Job 하나에서 자동 처리한다.
- LIVE 클러스터링은 새 질문만 기존 Cluster에 배치하거나 새 Cluster를 만든다. 기존 질문은 다른 Cluster로 이동시키지 않는다.
- 새 질문이 들어간 Cluster만 새 immutable AI 대표질문을 생성해 교체한다. 질문 배치와 대표질문 생성이 모두 끝나야 클러스터링 Job이 완료된다.
- 같은 semantic Cluster의 공개 ID는 LIVE generation 사이에 계승하고 새 Cluster만 새 ID를 받는다. FINAL full rebuild는 새 Cluster ID를 할당한다.
- AI 대표질문의 공개 응답과 화면 provenance에는 해당 문장이 만들어진 generation인 `created_in_generation`을 포함한다. 교체된 대표질문에 `CAPTURING` 또는 `COMPLETED` Answer가 있으면 해당 Cluster의 typed child로 보존한다. Answer가 없으면 공개 마인드맵·단건 조회·새 AI 검색에서 즉시 폐기한다. 기존 Evidence 참조가 없으면 hard delete하고, 참조 중이면 provenance 전용 내부 tombstone으로 남겨 Evidence의 `label`만 유지하고 `link=null`로 이동을 막는다. 마지막 Evidence를 삭제하는 transaction에서 tombstone도 원자적으로 정리한다. 캡처 중이면 완료·취소가 확정될 때까지 보존한다.
- 마인드맵은 현재 AI 대표질문을 중앙에, 학생 질문과 보존된 과거 AI 대표질문을 타입이 구분된 child로 보여준다. Cluster member 공개 discriminator는 정확히 `source_kind=STUDENT_QUESTION|AI_REPRESENTATIVE`이고, Cluster 정렬은 `ordinal ASC, id ASC`이며 마지막 클러스터링 Job의 mode·상태·attempt를 식별할 수 있어야 한다.
- Cluster 목록과 child 목록은 기본 20·최대 100개의 cursor pagination API를 사용한다. 정렬은 고정하지만 generation 교체 중 기존 cursor 만료·재시작 정책과 큰 마인드맵의 preload page 수·점진 loading·자동 축소·탐색 방식은 후속 계약으로 남긴다.
- 클러스터링 이후에도 학생 원본 질문을 각각 확인할 수 있어야 한다.
- 클러스터링에 실패해도 질문 작성, 조회와 답변 기능은 계속 사용할 수 있어야 한다.

### Transcript와 답변

- 아직 확정되지 않은 partial, DB에 저장된 live final Segment와 수업 후 canonical Transcript를 화면에서 구분한다. live final은 실시간 발화 확정이며 영구 Transcript `FINALIZED`와 같은 뜻이 아니다.
- Answer는 학생 질문 또는 AI 대표질문 하나만 target으로 갖는다. 하나의 target에는 취소되지 않은 Answer가 최대 하나다.
- 교수자가 LIVE에서 target을 선택한 시점부터 완료한 시점까지의 확정 Transcript를 음성 Answer 후보로 연결한다.
- 교수자는 자동 연결 결과를 확인한 후 답변 완료 상태로 변경할 수 있다.
- 대표질문에 Answer를 완료해도 소속 Cluster의 child 질문은 자동으로 `ANSWERED`가 되지 않는다.
- LIVE 캡처를 취소하면 해당 Answer를 삭제한다. 학생 질문과 현재 `ACTIVE` 대표질문은 미답변 상태로 복귀한다. 이미 `PRESERVED`인 과거 대표질문은 membership과 공개 화면에서 즉시 제거하며, Evidence 참조가 없으면 hard delete하고 참조 중이면 provenance 전용 내부 tombstone으로 전환한다. `CANCELLED` Answer 기록은 보관하거나 사용자에게 노출하지 않는다.
- LIVE에서는 text-only Answer 생성과 음성 Answer의 교수자 text 편집을 제공하지 않는다.
- `COMPLETED` class에서 교수자는 미답변 `STUDENT_QUESTION`에만 새 text-only Answer를 등록한다. 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` 음성·text-only Answer에는 text를 추가·수정할 수 있다. FINAL·REVIEW 전용 또는 미답변 `ACTIVE` AI 대표질문에는 새 text-only Answer를 만들지 않는다. text는 trim·Unicode NFC 정규화 뒤 1~2,000자이며 수정은 `expected_version`을 요구한다. 충돌 시 현재 version·text를 안내하되 사용자가 쓴 로컬 초안은 유지한다. text-only Answer 철회는 행 삭제와 학생 질문 `OPEN` 복귀이고, 음성 Answer 철회는 text만 제거한다.
- 완료된 음성 Answer는 LIVE에서 `NOT_STARTED`, PROCESSING에서 HQ source·재매핑 선택이 끝나기 전에는 Job ID 없는 `WAITING_SOURCE`다. source가 terminal이 되면 후처리 coordinator가 각 Answer마다 `visibility=SHARED`, `blocks_session_completion=true`인 `ANSWER_ORGANIZATION` Job을 하나씩 자동 생성하며 사용자가 직접 생성하지 않는다.
- Answer 정리 Job을 생성할 때 재매핑이 성공했으면 HQ 범위를 우선하고, 재매핑이 없거나 실패했으면 Answer에 보존된 immutable 원본 LIVE Transcript 범위를 선택해 이후 재시도에도 고정한다. canonical 전환 뒤에도 원본 범위를 제거하거나 덮어쓰지 않는다.
- AI 정리 결과는 교수자 text와 별도 데이터로 저장·응답한다. 교수자 text가 있으면 이를 기본 답변으로 우선 표시하고 AI 결과는 별도 label로 표시하며, 두 결과와 관계없이 원본 음성 범위를 항상 확인할 수 있게 한다. 공개 응답도 교수자 text·AI 정리·원본 LIVE 범위·선택적 HQ 재매핑 범위를 하나의 값으로 합치지 않는다.
- Answer별 Job은 서로 독립적이다. 하나가 `FAILED`여도 다른 Answer 정리와 기록을 가리지 않고 terminal 실패로 Session 완료에 참여한다. Course 교수자는 같은 Job 행을 `attempt + 1`로 재시도할 수 있으며 성공한 정리는 MVP에서 다시 생성하지 않는다.
- 적용 대상의 필수 Job 또는 성공 결과 원장이 누락되면 Answer 자체는 계속 표시하고 AI 정리 상태만 `DATA_INTEGRITY_ERROR`로 구분한다. 내부 오류 원문은 노출하지 않는다.
- AI 정리 결과의 외부 object shape는 API 계약을 사용하며, `content` 형식·최대 길이·모델·prompt·품질 기준과 성공 결과의 KnowledgeChunk 재색인 방식은 미정이다.
- FINAL clustering 성공 결과에는 종료 attempt의 eligible 입력이 각각 정확히 한 번 포함되어야 하며, 분류할 수 없는 입력은 명시적인 `기타` Cluster에 배치한다. 대상 입력이 0건일 때 Job 생략·빈 성공 원장·`final_generation` 표현은 미정이다.
- canonical 전환 뒤 기존 Answer 시간 범위를 새 Segment에 재매핑한다. `PENDING`, `SUCCEEDED`, `FAILED`를 구분하고 전체 범위를 연결한 경우에만 새 Segment 범위로 이동한다. 재매핑이 일부 실패해도 해당 실패를 다른 저장 기록과 분리해 표시하며, 정확한 허용 오차와 부분 일치 상태는 미정이다.
- canonical Transcript에는 문장 Segment와 Transcript를 만들 수 없었던 gap을 시간 순서대로 함께 표시한다. gap 예시는 `[05:20~05:35] 연결 문제로 기록되지 않은 구간입니다.`이다.

### 완료 기록 조회와 Evidence

- `GET /sessions/{id}/record`는 완료 기록 화면을 초기화하는 compact manifest다. Session·Recording·Transcript·Summary·후처리의 요약 및 상태, Material·Transcript·질문·Answer·Cluster·공유 Job의 개수와 권한 검사를 거치는 공개 조회 경로만 반환하고 각 영역의 전체 배열을 embed하지 않는다.
- manifest의 상태·개수는 응답 시점의 값이며 목록 전체를 잠그는 snapshot token이 아니다. `PROCESSING`에서 Job·Transcript·후처리 상태나 개수가 바뀌면 `/record`를 다시 조회해 새 manifest를 반영한다.
- 화면은 manifest의 개수로 빈 영역을 구분하고, 표시할 영역의 공개 경로를 따라 Material·Transcript·질문·Answer·Cluster·공유 Job을 독립적으로 점진 로딩한다. 영역별 `loading|empty|normal|error`와 재시도를 두고, 재시도는 다른 영역의 이미 로딩된 항목을 초기화하지 않는다.
- Transcript·질문·Answer·Cluster 목록은 각각 고정된 정렬과 불투명 cursor를 사용하며 `limit`은 기본 20, 최대 100이다. 페이지 추가 로딩이 실패하면 이미 표시한 항목을 유지한 채 같은 다음 cursor를 재시도한다. Cluster generation 교체 중 기존 cursor 만료·재시작 정책은 미정이다.
- AI Evidence의 공개 `source_kind`는 정확히 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`이다. `QUESTION`은 학생 질문과 AI 대표질문을 모두 포괄하며 사용자용 `label`과 공개 link 대상이 둘을 구분한다. `TRANSCRIPT` link는 Segment ID가 아니라 Session·Transcript version과 안정적인 sequence·시간 범위를 locator로 사용해 해당 문장 또는 구간으로 이동한다. `link`는 `/record` 배열 위치·pagination cursor에 의존하지 않는다.
- 클라이언트는 내부 `knowledge_chunk_id`, storage key·path, 원본 provider URL을 표시하거나 이동 경로로 조합하지 않는다. Evidence `link` 열기가 실패해도 저장된 AI 결과는 유지하고 근거 이동만 독립 오류로 안내한다.
- Material detach·원본 삭제 또는 AI 대표질문 폐기 후 기존 Evidence는 안전한 `label` snapshot을 유지하고 `link=null`로 이동을 비활성화한다. 폐기된 대표질문은 새 검색에서 제외하고 마지막 Evidence를 삭제하는 transaction에서 함께 정리한다. Material Evidence의 정확한 보관 기간·FK snapshot 전환·hard delete·`410 Gone` 정책은 미정이다.

### Course workspace archive

- Course 화면은 정확히 `PDF 자료`, `Transcript`, `AI 요약`, `질의응답` 네 archive navigation과 class rail을 제공한다. class rail의 `LIVE CLASS` 영역은 `current_session` 유무와 무관하게 항상 존재하며 `READY`, `LIVE`, `PROCESSING`, 없음 상태를 실제 값대로 구분한다.
- 완료 class rail과 각 archive collection은 불투명하고 Course·resource·filter scope에 묶인 cursor를 사용한다. 다른 Course나 archive에 cursor를 재사용하면 `400 INVALID_CURSOR`로 거부한다. 추가 page 실패 때 이미 표시한 항목은 유지한다.
- 완료 class는 제품 기준인 `started_at DESC, id DESC`로 정렬한다. active class는 `current_session`으로 별도 고정하므로 완료 목록 cursor에 포함하지 않는다.
- PDF archive는 Session 상태와 무관하게 현재 attached Material이 하나 이상인 class를 최근 class 순서로 묶어 반환한다. `UPLOADED`, `PROCESSING`, `READY`는 권한 확인 원문 열기·개별 다운로드를 제공하고 `FAILED`는 원문 동작을 제공하지 않는다. bulk zip과 detached 자료는 범위 밖이다.
- Transcript archive는 `LIVE`, `PROCESSING`, `COMPLETED` class index에 전체 timeline 배열을 embed하지 않는다. 선택 class의 공개 `timeline_url`을 따라 Segment·Gap cursor를 로딩하며 canonical이 있으면 canonical을, 없으면 공개 가능한 최신 LIVE final version을 사용하고 처리·terminal 상태를 명시한다.
- Summary archive는 `PROCESSING`, `COMPLETED` class의 공용 `FINAL`만 반환한다. `summary_type=LIVE`, `visibility=REQUESTER_ONLY`, 개인 Chat·Message·Job은 SQL 조회와 public schema 모두에서 제외한다.
- Q&A archive의 한 행은 `STUDENT_QUESTION` 또는 `AI_REPRESENTATIVE_QUESTION` target이다. 학생 질문은 미답변이면 `answer=null`, 답변됐으면 공개 Answer를 포함한다. AI 대표질문은 완료 Answer가 존재할 때만 immutable `target_text_snapshot`과 함께 포함한다. 작성자 식별정보, `CAPTURING`·취소 Answer와 폐기된 대표질문은 제외한다.
- Q&A archive는 신규 Answer·수정·철회·공용 Job retry를 제공하지 않고 기존 class 상세·완료 기록으로 이동한다.

### AI 기능

- Course 교수자와 학생은 같은 역할 중립 UI에서 개인 AI를 사용한다. `LIVE`에서는 Summary·Chat, `COMPLETED`에서는 `REVIEW` Chat을 제공하고, `PROCESSING`에서는 `REVIEW` Chat을 제공하지 않는다. publisher·수업 종료·자료·Answer 같은 교수자 전용 control만 공통 UI에 추가한다.
- 실시간 요약은 요청 시점까지의 Transcript와 해당 세션에 현재 연결된 `READY` PDF를 기준으로 생성한다.
- 개인 AI는 요청자 전용 `202 Accepted + AIJob` 뒤 Job polling과 성공한 저장 결과의 REST 조회만 사용한다. Chat 요청은 USER Message와 그 Message를 입력으로 고정한 `CHAT_RESPONSE` Job·outbox를 `202` 응답 전에 하나의 transaction으로 반드시 저장하고, 같은 Job 재시도도 동일 USER Message를 사용한다. 생성 중 assistant 부분 문장은 저장·공유하지 않고, 성공한 최종 assistant Message·Evidence와 Summary 결과만 저장한다.
- 개인 Job·Message·결과는 shared Session WebSocket에 broadcast하지 않으며 token delta, SSE, streaming HTTP와 private WebSocket을 제공하지 않는다.
- LIVE·REVIEW Chat USER `content`는 앞뒤 공백 제거와 Unicode NFC 정규화 뒤 Unicode code point 기준 1자 이상 2,000자 이하다. 공백뿐이거나 초과한 입력은 자르지 않고 `422 VALIDATION_ERROR`로 거부하며, 최대 길이 위반은 `field`, `reason=MAX_LENGTH_EXCEEDED`, `max_length`, `actual_length`를 제공한다. 클라이언트는 입력 원문을 유지하고 정규화 결과의 글자 수와 오류를 표시한다. assistant Message는 이 길이 제한 대상이 아니다.
- Session이 `LIVE`에서 `PROCESSING`으로 전이하는 transaction에서 모든 LIVE Summary·Chat·Message·Evidence와 관련 요청자 전용 Job을 제거한다. FINAL Summary와 `COMPLETED` REVIEW Chat에는 영향을 주지 않는다. 정리 전에 받은 polling·resource 식별자의 이후 응답과 같은 `Idempotency-Key` replay 의미는 미정이므로 고정 `404`나 이전 성공 응답 재생을 계약하지 않는다.
- 클라이언트는 `session.updated=PROCESSING`을 받으면 선택한 LIVE Job·Chat ID와 cache를 지우고 기록 정리 화면으로 이동한다. 이전 개인 결과를 완료 기록으로 승격하지 않는다.
- `READY`와 Summary 요청 전 `LIVE`의 정상 무요약 상태는 `summary_status=NOT_STARTED`, `summary_reason=null`, `summary=null`, `job_id=null`이다.
- `LIVE`에서 저장된 live final Segment가 0개이면 Summary Job을 만들지 않고 `409 SUMMARY_TRANSCRIPT_NOT_READY`와 “아직 확정된 강의 내용이 없습니다. 잠시 후 다시 시도해 주세요.”를 반환한다.
- `PROCESSING`에서 HQ source gate가 결정되지 않은 동안은 Job 없이 `summary_status=PENDING`, `summary_reason=null`, `summary=null`이 정상이다. 최신 HQ `RECORDING` version이 `FINALIZED`이고 Segment가 1개 이상이 되는 transaction에서 coordinator가 FINAL Summary Job을 원자적으로 생성하며, eligible source에 Job이 없으면 `DATA_INTEGRITY_ERROR`다.
- RECORDING Transcript가 정상적으로 `EMPTY`가 되면 Job 없이 `summary_status=NOT_APPLICABLE`, `summary_reason=NO_FINAL_TRANSCRIPT`와 “요약할 강의 내용이 없습니다.”를 제공한다.
- Recording source 자체가 없으면 즉시, Recording은 있지만 HQ source가 `FAILED`이거나 10분 제한까지 결과가 없으면 Job 없이 `summary_status=FAILED`, `summary_reason=SUMMARY_SOURCE_UNAVAILABLE`와 “Transcript 처리 문제로 요약을 만들지 못했습니다.”를 제공한다.
- `PROCESSING`의 FINAL Summary Job은 최초 실행이나 실패 뒤 재시도 attempt와 무관하게 `PENDING|RUNNING`일 수 있다. `COMPLETED`의 active Job은 실패 뒤 명시적으로 재시도한 `attempt > 1`이거나, 성공한 HQ 재시도 뒤 `attempt > 1`인 `SESSION_POSTPROCESSING` coordinator가 새 source에 생성한 FINAL Summary일 때 정상이다. 최초 종료 후처리의 coordinator `attempt=1`에서 만든 FINAL Summary가 `attempt=1` active로 남으면 `DATA_INTEGRITY_ERROR`다.
- `SUCCEEDED` FINAL Summary Job에 저장 결과가 없거나 `COMPLETED`의 eligible source에 FINAL Summary Job이 없으면 `DATA_INTEGRITY_ERROR`다. 사용자에게는 내부 원문 대신 “요약 상태를 확인할 수 없습니다. 잠시 후 다시 시도해 주세요.”를 제공한다.
- 보존된 LIVE 포인터는 final source 정책이 확정되기 전까지 자동 FINAL Summary에 사용하지 않는다.
- `REVIEW` Chat은 `COMPLETED`인 선택 Session에 현재 연결된 `READY` 자료와 final 기록만 기본 답변 범위로 사용한다.
- 강의 기록에서 근거를 찾지 못하면 확인할 수 없는 내용임을 사용자에게 알린다.
- AI 작업 중에는 진행 상태를 표시한다. 실패한 LIVE incremental clustering은 시스템 scheduler가 같은 Job 행을 자동 재시도하고, 교수자에게 허용된 실패 공유 Job은 완료 기록에서 같은 Job 행을 명시적으로 재시도할 수 있게 한다.

### 수업 종료

- 강의 종료가 확정되면 Session을 즉시 `PROCESSING`으로 표시하고 새 실시간 입력을 차단한다.
- 같은 전이에서 요청자별 LIVE Summary·Chat·Message·Evidence와 관련 Job을 모두 삭제한다. 클라이언트는 `session.updated=PROCESSING`을 받으면 선택한 개인 AI 식별자와 cache를 지우며, FINAL Summary·REVIEW Chat 데이터는 이 정리 대상에 포함하지 않는다.
- `PROCESSING` 전이 순간부터 실행 중이거나 retry 예약된 LIVE 클러스터링을 Session·attempt fence로 막고 늦은 결과도 commit하지 않으며 더 이상 재실행하지 않는다. 이 Job의 공개 terminal 표현을 `CANCELLED|SUPERSEDED`로 둘지, `retryable=false`인 `FAILED`로 둘지는 미정이다.
- FINAL 클러스터링은 종료 시점 질문 sequence 이하의 학생 실제 질문 전체와 종료 시각까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 입력으로 삼아 처음부터 재배치한다. 성공 결과에는 eligible 입력이 각각 정확히 한 번 포함되고 분류할 수 없는 입력은 명시적인 `기타` Cluster에 들어간다. 학생 질문 watermark는 모든 attempt에서 유지하고 미답변 AI 대표질문은 제외한다. FINAL Job이 `FAILED`한 뒤 교수자가 명시적으로 재시도할 때는 `base_revision`과 Answer 시각 상한을 현재 값으로 다시 캡처하며, 성공한 attempt의 snapshot은 이후 자동 변경하지 않는다.
- 로컬 녹음 upload와 원본 저장 완료 후 전체 녹음 HQ STT를 시작한다. 화면은 upload, HQ STT, 문장·시간 정렬, 기존 Answer 시간 범위 재매핑 검증, canonical 전환과 최종 AI 작업을 서로 구분해 표시한다.
- 후처리 coordinator는 LIVE에서 완료된 음성 Answer마다 공유 완료 차단 `ANSWER_ORGANIZATION` Job을 생성한다. HQ 재매핑이 성공하면 그 범위를, 그렇지 않으면 immutable 원본 LIVE 범위를 사용하며 Answer별 결과·실패를 독립적으로 기록한다.
- Worker heartbeat는 15초마다 lease를 갱신하고 60초 동안 갱신이 없으면 중단으로 판단한다. 이 heartbeat·lease 값은 고정한다. 일반 후처리 Job의 기본 실행 상한은 5분이지만 HQ STT `RECORDING_TRANSCRIPTION`은 제외하며 HQ 개별 Job timeout은 미정이다. Session 전체 `PROCESSING`은 `ended_at`부터 최대 10분이고, Session 제한을 넘긴 작업과 아직 시작하지 못한 필수 후속 작업은 재시도 가능한 `FAILED`로 기록한다.
- Session 완료를 막는 모든 Job이 `SUCCEEDED` 또는 `FAILED`가 되면 실패가 있어도 Session을 `COMPLETED`로 바꾼다. 늦게 도착한 이전 Worker 결과는 저장하지 않으며 실패한 AI 기능을 나중에 재시도해도 Session을 다시 `PROCESSING`으로 돌리지 않는다.
- 실패한 `ANSWER_ORGANIZATION`은 Course 교수가 완료 화면에서 같은 행의 `attempt + 1`로 재시도한다. 이미 성공한 정리는 다시 생성하지 않고, 재시도 중에도 교수자 text와 원본 음성 범위를 계속 제공한다.
- `COMPLETED`에서 실패한 HQ STT를 재시도해 새 RECORDING version이 성공하면 기존 `SESSION_POSTPROCESSING` coordinator 행을 `attempt + 1`로 다시 queue한다. Session은 `COMPLETED`를 유지한 채 Answer mapping과 canonical Knowledge link를 재구축하고 eligible하면 `FINAL_SUMMARY`를 생성한다. 기존 `ANSWER_ORGANIZATION` Job·결과는 새 HQ 범위로 조용히 재생성하거나 rebound하지 않는다.
- 일부 AI 작업이 실패하더라도 현재 연결된 열람 가능한 PDF, Transcript와 Q&A는 먼저 열람할 수 있어야 한다.
- 후처리가 완료되면 강의 기록 페이지에서 요약, 질문·답변과 복습 채팅을 제공한다.

### 명시적 미정 사항

- HQ STT `RECORDING_TRANSCRIPTION` 개별 Job timeout
- `LIVE → PROCESSING`에서 fence된 LIVE clustering의 공개 terminal 표현: `CANCELLED|SUPERSEDED` 또는 `retryable=false`인 `FAILED`
- 종료 때 제거된 LIVE Summary·Chat·Message·Evidence·Job의 기존 polling/resource 요청 응답과 같은 `Idempotency-Key` replay 의미

## 6. MVP 완료 조건

1. 인증 사용자가 Course를 생성해 유일한 교수자 owner가 되고, 다른 인증 사용자가 참여 코드로 해당 Course의 학생이 될 수 있다.
2. Course owner가 active class가 없을 때 class를 만들고 PDF를 업로드할 수 있다.
3. Transcript, 질문, 반응과 답변 상태가 새로고침 없이 참여자 화면에 반영된다.
4. 학생이 익명 질문을 작성하고 다른 질문에 한 번만 반응할 수 있다.
5. 질문 commit마다 pending watermark가 갱신되고 Session당 active clustering Job 하나에서 실행 중 새 질문을 coalescing할 수 있다.
6. LIVE에서 새 질문만 배치하고 기존 질문을 옮기지 않으며, 영향받은 Cluster의 immutable AI 대표질문만 재생성할 수 있다.
7. 교수자가 학생 질문 또는 AI 대표질문 하나를 선택해 음성 Answer Transcript를 연결하고, 취소하면 Answer 기록을 남기지 않을 수 있다.
8. 교수자와 학생이 같은 역할 중립 UI에서 요청자 전용 LIVE Summary·Chat을 사용하고, 교수자 전용 수업 control은 공통 UI에 추가할 수 있다.
9. 강의 종료 후 실행 중이던 LIVE 클러스터링 결과를 반영하지 않고, 종료 시점 학생 실제 질문 전체와 그때까지 `COMPLETED` Answer가 있는 AI 대표질문을 현재 중앙 여부와 무관하게 처음부터 FINAL 클러스터링하며 eligible 입력을 정확히 한 번씩 포함하고 미분류 입력을 `기타` Cluster에 둘 수 있다.
10. 최종 마인드맵을 `ordinal ASC, id ASC`로 표시하고 member `source_kind=STUDENT_QUESTION|AI_REPRESENTATIVE`, AI 대표질문의 `created_in_generation`과 마지막 클러스터링 Job의 mode·상태·attempt를 식별할 수 있다.
11. 교수자가 `COMPLETED` class의 미답변 학생 질문에만 새 text-only Answer를 등록하고, 학생 질문 또는 Answer로 보존된 AI 대표질문을 target으로 하는 기존 `COMPLETED` Answer의 text를 추가·수정할 수 있다.
12. LIVE에서 완료된 각 음성 Answer마다 공유 완료 차단 `ANSWER_ORGANIZATION` Job이 자동 생성되고, HQ 재매핑 범위 우선·원본 LIVE 범위 fallback으로 독립 처리될 수 있다.
13. 교수자 text·AI 정리 결과·원본 음성 범위를 구분해 제공하고, Course 교수가 실패한 Answer 정리를 같은 Job 행의 `attempt + 1`로 재시도할 수 있으며 성공 결과는 다시 생성하지 않는다.
14. 공개 질문 300자·초안 500자 제한을 trim·Unicode NFC 후 Unicode code point로 검증하고, 공백뿐이거나 초과한 입력을 자르지 않고 구조화된 `422 VALIDATION_ERROR`로 거부할 수 있다.
15. 강의 종료 후 Transcript, 요약과 질문·답변이 포함된 기록 페이지가 생성된다.
16. 교수자와 학생이 `COMPLETED` class에서 같은 개인 `REVIEW` Chat으로 해당 강의 기록을 근거로 복습할 수 있고, `PROCESSING`에서는 Chat을 시작할 수 없다.
17. 권한이 없는 사용자는 수업 자료와 기록에 접근할 수 없다.
18. 실패한 AI 작업의 상태를 확인하고 다시 실행할 수 있다.
19. Course owner가 참여 코드를 회전하면 기존 코드는 즉시 무효화되고 새 `[A-Z]{6}` 코드만 참여에 사용할 수 있다.
20. Course에는 active class가 최대 하나만 존재하고 `current_session`은 해당 class 또는 `null`이며, 완료 후에만 다음 class를 생성할 수 있다.
21. 같은 `lecture_date`의 완료 class를 실제 시작 시각과 함께 `started_at` 내림차순으로 구분해 확인할 수 있다.
22. Course owner가 모든 상태의 class 제목을 수정할 수 있고, 빈 제목은 자동 제목으로 바뀌며 날짜와 상태 시각은 수정할 수 없다.
23. Course owner가 `READY`·`COMPLETED` class를 삭제할 수 있고, `LIVE`·`PROCESSING`에서는 허용된 종료·완료 절차를 거친 뒤 삭제할 수 있다.
24. Course owner가 Session `READY`·`LIVE`·`COMPLETED`에서 최대 10개의 PDF를 추가·삭제할 수 있고, 각 파일은 decimal `100,000,000 bytes`를 넘지 않는다.
25. PDF가 없거나 `READY`·`UPLOADED`·`FAILED`만 있으면 class를 시작할 수 있고, 연결된 `PROCESSING` Material이 하나라도 있으면 시작할 수 없다.
26. Material 삭제가 확정되면 목록·열람·새 AI 검색에서 즉시 사라지고 백그라운드 정리 실패에도 다시 노출되지 않는다.
27. 동일 content 재업로드와 안정적인 표시 이름 suffix를 지원하며, `FAILED` 자료는 사용할 수 없는 업로드 실패로 명확히 표시한다.
28. 동일한 교수자 마이크 입력이 고정 실시간 wire format과 브라우저 로컬 녹음으로 동시에 분기되고 두 경로의 실패가 격리된다.
29. 첫 `audio.start` 성공 탭만 publisher가 되며 두 번째 publisher는 전송만 거부되고, active publisher는 같은 `client_stream_id`로 재연결·resume할 수 있다.
30. 종료 확정 즉시 Session이 `PROCESSING`으로 전환해 새 입력을 막고, 재개 가능한 녹음 upload와 원본 저장 완료 뒤 HQ STT를 시작한다.
31. 저장 녹음 playback 요청마다 권한을 다시 확인하고 접근 거부 상태를 다른 완료 기록과 분리해 표시한다.
32. playback 권한이 확인된 사용자가 Transcript 문장을 선택해 서버가 제공한 녹음 위치로 이동할 수 있다.
33. 전체 녹음을 HQ STT로 재처리하고 문장 Segment의 시작·종료 시각과 녹음 위치를 정렬해 저장할 수 있다.
34. 실시간 연결·STT 상태와 별개인 영구 Transcript 상태 `FINALIZING`·`FINALIZED`·`FAILED`·`EMPTY`를 복구할 수 있다.
35. 전체 녹음 재처리, Segment 저장, 시간·녹음 위치 매핑, `RECORDING` version의 `FINALIZED`와 canonical 전환이 성공하면 고품질 Final Transcript 완료로 표시한다. 이 상태를 100% 정확도 보장으로 표현하지 않는다.
36. canonical 전환 뒤 기존 Answer 시간 범위와 AI 검색 데이터를 새 Segment에 다시 연결해 검증하며, Segment와 gap을 같은 시간축으로 조회할 수 있다.
37. READY·요청 전 LIVE의 `NOT_STARTED`·`summary_reason=null`, PROCESSING source 대기의 Job 없는 `PENDING`, live final 0건, 정상 `EMPTY`, source unavailable, eligible source의 원자적 Job 생성과 결과·Job 누락 `DATA_INTEGRITY_ERROR`를 구분한다.
38. 15초 heartbeat·60초 lease, HQ를 제외한 일반 후처리 기본 5분과 Session 10분 제한을 적용하고, 완료 차단 작업이 모두 terminal이면 부분 실패가 있어도 Session을 완료해 다음 class를 막지 않는다. HQ 개별 timeout은 미정으로 남긴다.
39. `/record`에서 요약·상태·개수·조회 경로만 받고 Transcript·질문·Answer·Cluster를 각각 기본 20·최대 100의 cursor page로 끝까지 조회할 수 있다.
40. AI Evidence에서 정확히 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`인 `source_kind`·사용자용 `label`·안정적인 공개 `link`만 사용하고, Transcript link는 Session·version·안정적인 sequence/시간 범위를 사용하며 내부 chunk·storage·Segment ID·배열 index·cursor를 노출하지 않는다.
41. 개인 AI 요청을 `202 + AIJob → requester polling → 성공한 저장 결과 REST 조회`로 처리하고 생성 중 부분 결과나 개인 상태를 shared Session WebSocket으로 보내지 않는다.
42. Session이 `LIVE`에서 `PROCESSING`으로 전이하면 LIVE 개인 AI 데이터와 Job이 모두 사라지고 클라이언트 선택·cache도 지워진다. 이전 polling/resource 응답과 멱등 replay 의미는 미정으로 남긴다.
43. LIVE·REVIEW Chat USER 입력을 trim·Unicode NFC 후 Unicode code point 1~2,000자로 검증하며 원문을 자르지 않고 오류 뒤에도 입력 원문과 정규화 결과의 글자 수를 유지한다.
44. Course 화면에 정확히 네 archive navigation과 항상 존재하는 `LIVE CLASS` 영역을 제공하고, active class가 없거나 `READY`·`PROCESSING`이어도 실제 상태를 오인 없이 표시할 수 있다.
45. Course 멤버가 모든 class의 attached PDF를 cursor로 끝까지 조회하고, 기존 inline 열람을 깨지 않으면서 `attachment` 다운로드할 수 있다.
46. Course 멤버가 class별 Transcript와 공용 FINAL Summary를 cursor로 끝까지 조회하며 개인 LIVE Summary·Chat 결과가 archive에 노출되지 않는다.
47. Course 멤버가 작성자 식별정보 없이 모든 class의 미답변·답변 질문과 Answer target snapshot을 cursor로 조회하고, 관리가 필요하면 해당 class 기록으로 이동할 수 있다.
