현재 저장소는 `STORAGE_ROOT` 아래 temporary/final namespace를 분리하는 개발용 filesystem `Storage` adapter와 in-memory/failure test adapter를 제공한다. PDF Material은 이 adapter를 통해 업로드·본문 열람·분리까지 연결되지만, 녹음 원본 handler, resumable upload, playback, 삭제 scheduler·outbox consumer와 외부 Object Storage 연동은 구현되지 않았다. 녹음 원본 저장·playback은 MVP 목표이며 97 GiB KCloud 루트 디스크를 장기 원장으로 사용하지 않는다. 운영 MVP에는 외부 Object Storage 또는 동등한 내구성·용량 경계를 마련해야 하고, 제품·provider·전환 시점은 미정이다. Redis와 별도 벡터 데이터베이스는 실제 부하가 확인될 때 검토한다.

위 표는 현재 저장소의 구성 상태이고, 아래의 Course·class·녹음·멱등성·AIJob 규칙은 구현이 따라야 할 MVP 목표 계약이다. schema spine, 인증 API·service, Course·membership·참여 코드 API, Session lifecycle, Material PDF Worker, 익명 Question·Reaction API와 공통 AIJob kernel·polling·retry가 구현됐다. LIVE 학생은 `POST /sessions/{id}/question-drafts`로 500자 초안을 동기 adapter에 보내 300자 이하 후보를 받을 수 있으며, 이 경로는 Question·초안·AIJob·event를 저장하지 않는다. 현재 application은 deterministic fake를 사용하므로 실제 AI 품질·provider 운영 배포가 완료된 상태는 아니다. LIVE Question clustering은 captured watermark·revision·attempt/run-token fence, coalescing·same-row retry, immutable generation 저장·REST 조회와 local standalone poller까지 구현됐고, 실제 AI provider와 운영 배포는 아직 목표 계약이다. Session event WebSocket의 ticket·outbox·replay, Question invalidation과 audio publisher·PCM ACK·live final Segment·Gap, LIVE Answer 캡처·완료·취소와 COMPLETED text Answer의 versioned 수정·철회, `KNOWLEDGE_INDEXING` Job의 Material·canonical Transcript·text Answer 색인과 내부 scoped retrieval/Evidence projection도 구현됐다. Answer HQ 재매핑·AI 정리 worker, Summary·Chat API와 그 밖의 기능별 producer·worker는 아직 목표 계약이다.

### 처리 경계

- API 서버는 요청 검증, 권한 확인, 상태 변경과 실시간 전파를 담당한다.
- 오래 걸리는 PDF·STT·LLM 작업은 API 요청 처리와 분리한다.
- 데이터베이스를 최종 상태의 기준으로 사용하고 WebSocket은 변경 알림 용도로 사용한다.
- event ticket은 60초·1회용이며 원문 대신 HMAC hash만 저장한다. upgrade에서 Course membership을
  다시 확인하고, 재연결 cursor는 서명된 Outbox event ID로 검증한다. 현재 replay는 500개 이하의
  retained public event만 best-effort로 전송하며, 불가능하면 `resync.required` 뒤 REST query를
  다시 읽는다.
- Outbox publisher는 현재 FastAPI process-local hub에서 동작한다. 따라서 다중 API process
  fan-out과 Redis/NATS 같은 cross-process transport는 운영 확장 전 미정이며, 현재도 REST resync가
  정합성 복구 경로다.
- AI 기능이 실패해도 수업방, 질문과 저장된 기록 기능은 계속 동작해야 한다.
- 학생 질문 작성 도움은 예외적으로 짧은 동기 LLM adapter 호출이다. Course 학생·LIVE Session·500자 정규화 초안을 검증한 뒤 5초 deadline으로 실행하며, 후보는 300자 이하로 검증한다. AIJob·Outbox·Question을 만들지 않고 provider 오류는 안전한 `503`으로 바꾼다.
- 실시간 PCM 전송과 브라우저 로컬 녹음은 하나의 microphone source에서 분기하되 독립 상태로 관리한다. 한 경로의 실패가 다른 경로와 질문·이미 저장된 final Transcript를 자동 실패 처리하지 않는다.

## 4. 데이터 구조 원칙

상세 스키마는 별도 DB 문서에서 정의하되 다음 데이터 영역을 기준으로 설계한다.

| 영역 | 핵심 엔티티 |
|---|---|
| 사용자·권한 | User, CourseMember |
| 수업 | Course, LectureSession, LectureMaterial |
| 강의 기록 | SessionRecording, RecordingUpload, TranscriptVersion, TranscriptSegment, TranscriptGap, LectureSummary |
| 질문·답변 | Question, QuestionReaction, QuestionClusteringState, QuestionCluster, QuestionClusterMember, AIRepresentativeQuestion, Answer, AnswerTranscriptMapping, AnswerOrganization |
| AI 채팅 | ChatSession, ChatMessage, ChatMessageEvidence |
| 비동기 처리 | AIJob, KnowledgeChunk |

### 제약조건 원칙

- 모든 하위 데이터는 `course_id` 또는 `session_id`를 통해 소속 범위를 확인할 수 있어야 한다.
- Course 생성자는 해당 Course의 불변 owner이자 유일한 `PROFESSOR`다. 교수자 추가와 owner 이전은 제공하지 않으며 DB가 owner와 교수자 membership의 정확히 한 개 불변식을 검증한다.
- 참여 코드는 trim·대문자 정규화 후 `[A-Z]{6}`이고 자동 만료하지 않는다. 조회용 HMAC과 표시용 암호문을 분리하며 owner 회전이 commit되면 이전 코드는 즉시 무효화하고 이력을 남기지 않는다.
- Course에는 `READY`, `LIVE`, `PROCESSING` 상태를 합쳐 active class가 최대 하나다. partial UNIQUE를 최종 동시성 방어선으로 사용한다.
- 같은 날짜 class는 순차적으로 여러 개 허용하고 완료 목록은 `lecture_date`, 실제 `started_at`, `id` 내림차순으로 안정적으로 정렬한다.
- class 제목은 모든 상태에서 수정할 수 있다. 날짜와 이미 기록된 시작·종료·완료 시각은 변경하지 않는다.
- `detached_at IS NULL`인 Material만 class에 연결된 자료다. Session당 최대 10개, 파일당 decimal `100000000` bytes를 DB constraint·trigger와 서비스 transaction이 함께 검증한다.
- 업로드 원본명과 생성 뒤 바꾸지 않는 `display_name`을 분리한다. 표시 이름 충돌은 확장자 앞 ` (1)`, ` (2)` suffix로 해결하고 같은 내용의 재업로드는 허용한다.
- 질문 반응은 `(question_id, user_id)` 조합을 고유하게 만들어 중복을 방지한다.
- 질문 등록 transaction은 Question과 `QuestionClusteringState.requested_sequence` 증가를 함께 commit한다. Session당 `PENDING`·`RUNNING` `QUESTION_CLUSTERING` Job은 최대 하나이며 `applied_sequence` 이하의 질문은 다시 적용하지 않는다. 아직 적용하지 못한 captured 구간을 소유한 retryable `FAILED` LIVE Job은 하나의 `retry-reserved Job`으로 취급한다. 이는 새 AIJob status가 아니라 공개 `clustering_state.retry_job_id`로 식별하는 논리 예약이다.
- Cluster member는 내부적으로 학생 질문 또는 AI 대표 질문 FK 하나만 선택하고, 공개 discriminator는 `source_kind=STUDENT_QUESTION|AI_REPRESENTATIVE`를 사용한다. AI 대표질문은 Cluster title 문자열로 덮어쓰지 않고 immutable text·Job·attempt·`created_in_generation`을 가진 `AIRepresentativeQuestion` 리소스로 보존한다. 현재 대표·답변 보존·Evidence tombstone은 `lifecycle_status=ACTIVE|PRESERVED|DISCARDED`, 답변 여부는 `status=OPEN|SELECTED|ANSWERED`로 독립 관리하되 `DISCARDED`는 내부 상태라 공개 대표질문 조회에서 제외한다.
- Answer는 target kind가 `STUDENT_QUESTION` 또는 `AI_REPRESENTATIVE_QUESTION`이고 해당 FK 하나만 가진다. target당 삭제되지 않은 Answer는 최대 하나이며, 대표질문의 Answer가 하위 학생 질문을 답변 완료로 전이시키지 않는다.
- Answer 수동 text는 trim·Unicode NFC 정규화 후 1~2,000자다. `COMPLETED` text-only 철회는 Answer를 hard delete하고 학생 질문을 `OPEN`으로 복귀시키며, voice-backed Answer 철회는 text만 제거한다. version이 다른 수정은 현재 값과 함께 `409 ANSWER_VERSION_CONFLICT`로 거부한다.
- 완료된 음성 Answer마다 논리 `ANSWER_ORGANIZATION` Job과 성공 결과 `AnswerOrganization`은 각각 최대 하나다. AI 정리문은 교수자 작성 `Answer.text_content`와 분리하고, 둘 중 어느 쪽도 다른 Worker 결과가 덮어쓰지 않는다.
- Session당 외부에 노출하는 논리 SessionRecording은 최대 하나다. 첫 `audio.start`의 `client_stream_id` claim과 `CAPTURING` 전이는 원자적으로 처리하고 원문 stream ID는 저장하지 않는다.
- SessionRecording은 물리 object 개수와 무관한 논리 aggregate다. 단일 파일인지 fragment·manifest 집합인지와 DB·스토리지의 물리 cardinality는 미정이며 외부 계약에 노출하지 않는다.
- Transcript는 Session별 증가하는 `TranscriptVersion`으로 분리한다. source는 `LIVE`, `RECORDING`, 영구 상태는 `FINALIZING`, `FINALIZED`, `FAILED`, `EMPTY`이며 Segment와 Gap은 반드시 한 version에 귀속한다.
- Session의 `canonical_transcript_version_id`가 기본 조회와 AI 검색의 유일한 기준 version이다. HQ Worker가 조립 중인 비canonical Segment·Gap은 이 포인터가 전환되기 전까지 공개 조회에서 제외한다. canonical 전환 후 새 version용 KnowledgeChunk 재연결이 끝나기 전까지 이전 Transcript Chunk를 대신 검색하지 않는다.
- Segment는 class 시간축의 `start_ms`, `end_ms`와 별도로 녹음 seek용 nullable `recording_start_ms`, `recording_end_ms`를 저장한다. 클라이언트는 두 시간축을 임의 환산하지 않는다.
- Gap의 `start_ms`, `end_ms`는 class(Session) timeline의 누락 구간만 뜻한다. Gap에는 녹음 playback offset을 두지 않으며 seek에는 Segment의 `recording_start_ms`, `recording_end_ms`만 사용한다.
- 임베딩 검색에는 반드시 수업방 및 세션 범위 필터를 적용한다. Material source는 같은 SQL에서 `processing_status = 'READY' AND detached_at IS NULL`, Transcript source는 `source_transcript_version_id = canonical_transcript_version_id`, AI 대표질문 source는 `lifecycle_status IN ('ACTIVE', 'PRESERVED')`도 강제해 `DISCARDED` Chunk를 제외한다.
- Chat Evidence 원장은 내부 `knowledge_chunk_id`와 생성 시점의 안전한 `label_snapshot`을 저장한다. 공개 응답은 typed source를 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`로 축약하고 `label_snapshot`을 `label`로, 내부 ID 대신 안전한 상대 REST `link`로 계산한다. Transcript link는 Session·version·stable sequence 또는 time range를 사용한다.
- 개인 AI는 Course member 공통 기능이다. `PROFESSOR`, `STUDENT` 모두 `LIVE` Summary·Chat과 `COMPLETED`의 `REVIEW` Chat을 같은 방식으로 사용하며 `READY`, `PROCESSING`에서는 개인 Chat을 제공하지 않는다.
- Chat `USER` Message는 생성 Job·attempt와 model·prompt metadata가 모두 없고 trim·Unicode NFC 정규화 뒤 Unicode code point 1~2,000자다. `ASSISTANT` Message는 생성 Job·attempt가 필수이며 2,000자 상한은 적용하지 않는다. Assistant의 `model_name`, `prompt_version`은 현재 nullable 계약을 유지한다.
- 개인 AI는 요청자 전용 Job polling과 저장 결과 REST 조회만 사용한다. USER Message는 Worker claim 전에 저장하고, 생성 중 Assistant delta는 저장·공유하지 않으며 성공한 최종 Summary 또는 Assistant Message·Evidence만 결과로 commit한다.
- `CHAT_RESPONSE` Job은 Chat뿐 아니라 요청 transaction에서 저장한 immutable `target_user_message_id`를 typed FK로 가진다. 같은 Chat의 USER role·owner·Session을 검증하고 retry에서도 이 target을 바꾸지 않아 이후 turn의 history를 실패 turn 입력으로 오인하지 않는다.
- `LIVE → PROCESSING`은 LIVE Summary·Chat·Message·Evidence와 관련 requester-only Job을 원자 삭제한다. purge-scoped 멱등성 record의 조기 삭제와 종료 뒤 polling·재요청 응답은 미정이며 FINAL Summary와 REVIEW Chat은 유지한다.

### 상태 표현

```text
LectureSession: READY → LIVE → PROCESSING → COMPLETED
Recording:      CAPTURING → UPLOAD_PENDING → UPLOADING → UPLOADED | FAILED
RecordingUpload: ACTIVE → COMPLETED | EXPIRED | FAILED
TranscriptVersion: FINALIZING → FINALIZED | EMPTY | FAILED
Question:       OPEN → SELECTED → ANSWERED
AIRepresentativeQuestion lifecycle: ACTIVE → PRESERVED | DISCARDED; PRESERVED → DISCARDED
Answer(VOICE, LIVE): CAPTURING → COMPLETED | hard delete on cancel
Answer(TEXT, COMPLETED Session): create COMPLETED for unanswered student question | supplement an existing completed Answer
AIJob:          PENDING → RUNNING → SUCCEEDED | FAILED | CANCELLED | SUPERSEDED
```

Course 자체에는 종료·보관 상태를 두지 않는다. 상태 전이는 서버에서 검증하며 잘못된 순서의 요청은 거부한다. 재시도가 필요한 AI 작업은 같은 Job 행의 attempt를 증가시키고 현재 attempt·run token을 확인해 중복·늦은 결과를 차단한다.

## 5. API 설계 원칙

- REST API는 수업방, 세션, 자료, 녹음, 질문, 반응, 답변, 요약과 채팅 리소스를 중심으로 구성한다.
- 인증은 Google OAuth/OIDC 또는 이메일·비밀번호 인증 뒤 발급하는 HttpOnly server Session Cookie를 사용한다. 외부 authorization code나 access token을 application session으로 직접 사용하지 않는다.
- Google callback은 state·nonce·PKCE와 일회성 OAuth transaction을 검증하고, 이메일 가입·로그인도 같은 session 발급 경로에서 기존 Cookie를 새 token으로 회전한다. Session은 발급부터 7일 절대 만료이며 일반 요청으로 수명을 연장하지 않는다.
- 이메일은 NFKC·trim·casefold 정규화 뒤 활성 User 사이에서 유일하다. 비밀번호는 versioned scrypt salt hash만 저장하며 이메일 확인·reset·Google/email account linking과 로그인 rate limit provider는 아직 제공하지 않는다.
- 상태 변경 HTTP 요청은 환경별 정확한 Frontend origin allowlist를 필수로 검사한다. Origin 누락·`null`·불일치는 Course 권한 검사 전에 거부한다.
- 인증된 사용자 ID를 요청 본문에서 신뢰하지 않고 인증 컨텍스트에서 가져온다.
- 모든 리소스 접근 시 수업방 소유자 또는 참여 학생 여부를 검사한다.
- owner 전용 Course 참여 코드 회전·Course 삭제·class 관리 요청은 생성자와 현재 `PROFESSOR` membership이 모두 일치하는지 확인한다.
- 질문 반응 추가·취소와 강의 종료 요청은 반복 호출에 안전하게 설계한다.
- 실제 질문은 앞뒤 공백 trim 뒤 Unicode NFC로 정규화한 값의 code point 기준 1~300자, AI 작성 도움 초안은 같은 기준 1~500자로 검증한다. 초과 입력은 잘라 저장하지 않고 `422 VALIDATION_ERROR`로 거부한다. 작성 도움은 정규화한 초안을 300자 이하 NFC 질문 후보로 다듬어 `200` 동기 응답하며 초안·후보를 저장하거나 AIJob을 만들지 않는다.
- Chat USER Message도 trim 뒤 Unicode NFC로 정규화한 값의 code point 1~2,000자를 검증하며 빈 값과 초과 입력은 자르지 않고 `422 VALIDATION_ERROR`로 거부한다. 정규화한 값만 저장하고 이 제한은 Assistant Message에 적용하지 않는다.
- 일반 멱등성 terminal 응답은 terminal 전이 시각부터 정확히 24시간 같은 status와 body를 재사용한다. LIVE Summary·LIVE Chat 생성·Message와 관련 Job retry는 `purge_on_session_end=true`로 표시한다. Session 종료 뒤 원 리소스·Job은 삭제하지만 기존 polling·단건 조회와 멱등 재요청을 어떤 응답으로 끝낼지, 관련 멱등성 행을 조기 삭제할지는 미정이다. FINAL·REVIEW 요청과 retry는 purge 대상이 아니다.
- `/api/v1/sessions/{session_id}/record`는 Session·후처리 상태, 리소스별 count와 전용 조회 path만 반환하는 manifest다. Material·Transcript·Question·Cluster·Answer·AIJob 본문 배열을 embed하지 않는다.
- Material·Transcript·Question·Answer·Cluster·공용 AIJob은 전용 cursor API로 조회한다. Transcript는 version과 `(start_ms, SEGMENT 우선, id)`를 고정한 union page 하나를 만든 뒤 같은 page를 `segments[]`, `gaps[]`로 나누며 두 배열에 별도 cursor를 두지 않는다.
- 기록 manifest의 Question path는 종료 뒤 불변인 `(created_at DESC, id DESC)`를 쓰도록 `sort=RECENT`를 고정한다. 반응 수가 바뀌는 `POPULAR` 정렬은 이 완료 기록 cursor의 기본 경로로 사용하지 않는다.
- cursor는 scope·filter와 unique tie-breaker를 포함한 불투명 값이다. canonical 전환 뒤에도 Transcript cursor는 최초 version을 유지하고, Cluster cursor 안정성은 고정 generation 안에서만 보장한다. generation 교체 중 만료·재시작 계약은 미정이다.
- 오류 응답은 코드, 사용자 메시지와 요청 ID를 일관된 형식으로 반환한다.
- PDF 다운로드는 권한 확인을 거친 API 또는 만료 시간이 짧은 서명 URL을 사용한다.
- 녹음 metadata·upload·playback 요청은 매 요청에서 현재 인증과 Course 접근 권한을 다시 확인한다. storage key, 서버 경로, fragment key와 manifest는 응답·공유 event·로그에 노출하지 않는다.
- Evidence `link`는 현재 권한을 다시 확인하는 API root 기준 상대 경로다. `/record` 배열 위치, pagination cursor, 내부 KnowledgeChunk ID, storage key와 signed storage URL에 의존하지 않으며 안전한 공개 source 경로가 없으면 `null`이다. `DISCARDED` 대표질문 Evidence도 `source_kind=QUESTION`과 저장 label은 유지하고 link만 `null`이다.

정확한 경로, 요청·응답 스키마와 오류 코드는 API 문서에서 관리하며 구현 코드의 OpenAPI 출력을 최종 계약으로 사용한다.

### Course·class lifecycle

- `current_session`은 해당 Course의 유일한 `READY`, `LIVE`, `PROCESSING` class이며 없으면 `null`이다. active class가 `COMPLETED`가 된 뒤에만 다음 class를 생성한다.
- 현재 PR-09 구현은 Session CRUD·`READY → LIVE → PROCESSING`과 내부 worker용 `PROCESSING → COMPLETED` 전이, LIVE TranscriptVersion 생성, blocking `SESSION_POSTPROCESSING` coordinator 저장까지 제공한다. Recording/HQ STT·coordinator 실행과 downstream Job 생성은 후속 구현이며, 화면은 Job 개수로 완료를 추측하지 않고 Session `status`만 표시한다.
- class 생성 경쟁은 하나만 성공하고 나머지는 `ACTIVE_SESSION_EXISTS`로 응답한다. 허용되지 않은 상태 전이·삭제는 `SESSION_STATE_CONFLICT`로 구분한다.
- 빈 class 제목은 `Course 제목 · YYYY.MM.DD HH:mm` 서버 자동 제목으로 치환한다. 날짜는 `lecture_date`, 시각은 생성 시 기록한 `created_at`의 기본 timezone `Asia/Seoul` 표시값이며 이후 상태 전이에도 고정한다. 사용자는 `lecture_date`, `started_at`, `ended_at`, `completed_at`을 수정할 수 없다.
- class 시작은 Session과 연결된 Material을 잠근 뒤 `PROCESSING`이 하나라도 있으면 `MATERIAL_PROCESSING_ACTIVE`로 거부한다. PDF가 0개이거나 연결된 Material이 `READY`, `UPLOADED`, `FAILED`뿐이면 시작할 수 있다.
- owner는 `READY`, `COMPLETED` class만 aggregate 단위로 삭제할 수 있다. `LIVE`는 종료하고 `PROCESSING`은 완료될 때까지 기다린다.
- owner는 Course를 삭제할 수 있지만 active class가 있을 때의 허용 여부와 삭제 후 복구 유예는 아직 미정이다. 구현은 이 두 정책을 임의로 확정하지 않는다.

## 6. 실시간 통신

클라이언트는 강의 세션 단위로 서로 다른 두 WebSocket에 연결한다. Session event WS는 Course 멤버의 공용 변경 알림을 전달하고, audio WS는 Course `PROFESSOR`의 실시간 PCM과 ack·flow control·resume만 담당한다. 두 채널은 별도 ticket·권한·연결 상태를 가지며 event WS로 audio frame이나 recording upload chunk를 보내지 않고 audio WS로 공용 Session event를 broadcast하지 않는다.

| 이벤트 | 용도 |
|---|---|
| `transcript.partial` | 아직 확정되지 않은 실시간 STT 결과 전파 |
| `transcript.final` | 저장된 확정 Transcript 구간 전파 |
| `transcript.status` | live STT와 현재 LIVE TranscriptVersion 상태 전파 |
| `question.created` | 새 질문 전파 |
| `question.updated` | 질문 리소스 자체의 상태 변경 전파 |
| `clustering.updated` | watermark·revision·generation·active/last Job 상태 전파 |
| `reaction.updated` | 질문별 반응 수 변경 전파 |
| `answer.updated` | 답변 연결과 완료 상태 전파 |
| `answer.deleted` | 취소로 hard delete된 Answer와 target 무효화 전파 |
| `recording.updated` | 안전한 Recording 상태 변경 전파 |
| `transcript.version.updated` | 영구 TranscriptVersion 상태·canonical 포인터 변경을 알리는 invalidation |
| `job.updated` | AI 작업 진행·완료·실패 상태 전파 |
| `session.updated` | 강의 시작·종료와 후처리 상태 전파 |

LIVE Summary·Chat, REVIEW Chat과 모든 `REQUESTER_ONLY` Job 결과는 공용 Session WebSocket으로 보내지 않는다. MVP는 `POST → 202 + AIJob → GET Job polling → 저장 결과 GET`만 사용하고 성공한 최종 결과만 DB에 저장한다. 별도 공용 Summary 완료 event, token delta, SSE, streaming HTTP와 private WebSocket은 제공하지 않는다.

### 연결 복구

- 서버는 주기적인 ping/pong으로 연결 상태를 확인한다.
- 재접속 시 클라이언트는 마지막으로 반영한 opaque `cursor`를 전달한다. cursor가 없거나 만료·미지원이면 서버는 `resync.required`와 다시 조회할 리소스 범위를 반환한다.
- 누락 데이터는 REST API로 다시 조회해 화면 상태를 복원한다. `/record` manifest로 상태·count·조회 경로를 복구한 뒤 Transcript·질문·Cluster·Answer·공용 Job은 각 cursor·단건 API에서 읽으며, 이 전용 REST 결과와 공개 `clustering_state`를 최종 진실로 사용한다.
- 이벤트는 at-least-once 전달될 수 있으므로 `event_id`로 중복을 제거하고, 같은 리소스의 더 작은 `resource_version`은 오래된 갱신으로 무시한다.
- `transcript.version.updated`를 받으면 기존 canonical timeline과 cursor cache를 무효화하고 version 목록과 canonical Transcript를 REST로 다시 조회한다. event payload만으로 Segment·Gap이나 canonical 결과를 재구성하지 않는다.
- audio WS에서 첫 성공 `audio.start`의 `client_stream_id`가 Session의 단독 publisher를 claim한다. 다른 ID의 연결은 `AUDIO_PUBLISHER_CONFLICT`와 close code `4409`로 음성 전송만 거부하고, 같은 ID는 새 ticket으로 reconnect·resume할 수 있다.
- 수락한 PCM frame은 45초 liveness를 갱신한다. liveness 만료는 관측·장애 판단용이며 claim을 다른 탭·장치로 넘기지 않는다. Session 종료 전에는 동일 `client_stream_id`만 reconnect할 수 있다.

## 7. 주요 처리 파이프라인

### 7.1 PDF 전처리

1. API 서버가 Course의 `PROFESSOR` 권한, PDF MIME·parsing 가능 여부와 `1..100000000` bytes 크기를 검사한다.
2. 원본 PDF를 파일 스토리지에 server-generated storage key로 저장한다. 이 키는 내부 식별자이며 API·공유 event·로그에 노출하지 않는다.
3. Session을 잠그고 `READY`, `LIVE`, `COMPLETED`인지, 연결된 Material이 10개 미만인지 다시 확인한다. `PROCESSING` Session의 업로드는 거부한다.
4. 원본명과 별도인 안정적인 `display_name`을 할당하고 Material, `MATERIAL_PROCESSING` Job, queue outbox와 제공된 멱등성 키의 terminal 응답을 한 transaction에서 만든다. 이름 충돌은 확장자 앞 ` (1)`, ` (2)` suffix로 해결하고 기존 이름은 재번호를 매기지 않는다.
5. Material 처리 Worker가 PyMuPDF로 PDF를 다시 열어 페이지 수와 텍스트 추출 가능 여부를 확인한다. 이 단계는 Material을 사용할 수 있는 PDF로 판정하는 경계이며, KnowledgeChunk 분할·embedding 생성은 PR-18의 RAG Worker가 `READY` Material을 대상으로 수행한다.
6. Worker는 현재 Job attempt·run token·상태와 Material version·`detached_at IS NULL`을 다시 확인한다.
7. 현재 Material version과 연결 상태가 유효할 때만 page_count, Material `READY`와 Job `SUCCEEDED`를 한 transaction으로 확정한다. chunk·vector·RAG 결과는 이 transaction에 포함하지 않는다.

MVP의 `MAX_UPLOAD_BYTES`는 decimal `100000000`이다. 업로드 시 MIME·PDF signature·PyMuPDF parsing을 모두 확인해 유효하지 않은 파일은 `415 UNSUPPORTED_MEDIA_TYPE`로 거부한다. 처리 Worker의 예기치 않은 읽기·추출 실패는 안전한 실패 Job과 Material `FAILED`로 남기며 원본 provider·파서 오류 원문을 공개하지 않는다. 같은 내용과 같은 원본 파일명의 PDF는 별도 Material로 업로드할 수 있다.

로컬 개발에서는 API와 별도로 `cd backend && uv run python -m tbd.jobs.material_worker`를 실행해 `MATERIAL_PROCESSING` Job을, `cd backend && uv run python -m tbd.jobs.knowledge_worker`를 실행해 `KNOWLEDGE_INDEXING` Job을 polling한다. Material이 `READY`가 되거나 final Segment가 저장될 때 세션당 활성 색인 Job 하나를 예약한다. 이 feature Worker들은 공통 Worker registry를 만들거나 다른 Job type을 claim하지 않는다. container·drain·공통 registry 운영은 PR-29의 범위다.

#### 7.1.1 Material 연결 해제

- 해당 Course의 `PROFESSOR`는 Session `READY`, `LIVE`, `COMPLETED`에서 Material을 연결 해제할 수 있다. Session `PROCESSING`에서는 `SESSION_STATE_CONFLICT`로 거부한다.
- transaction은 `Session → Material → AIJob` 순서로 잠그고 `detached_at`, Material `version`, 내부 object cleanup 요청 outbox와 필수 멱등성 키의 terminal `204`를 함께 commit한다. `PENDING`·`RUNNING` Job은 `CANCELLED`로 끝내고, `FAILED` Job은 재시도 불가로 남겨 늦은 worker 결과를 fence한다. 결과 provenance로 참조되는 `SUCCEEDED` Job도 보존한다.
- commit 즉시 Material 목록·상세·content와 RAG 검색에서 제외한다. `processing_status`에 `DETACHED`를 추가하지 않고, cleanup 실패가 tombstone을 되돌리지 않는다.
- 원본 content 조회는 연결된 `UPLOADED`, `PROCESSING`, `READY` Material만 허용한다. `FAILED`, detached, 권한 밖 자료는 `404 MATERIAL_NOT_FOUND`로 존재를 숨기며, RAG는 이보다 좁게 연결된 `READY`만 사용한다.
- Worker claim과 결과 commit은 연결된 Material만 대상으로 하고 version을 확인하므로 연결 해제 전 작업의 늦은 결과를 저장하지 않는다.
- 현재 outbox에는 내부 object cleanup 요청만 기록한다. PR-18 이후에는 참조되지 않은 Material source KnowledgeChunk도 같은 멱등 정리 대상으로 추가한다. 연결 해제 Material의 원문 content link는 제공하지 않는다. 기존 Chat Evidence는 저장된 안전한 label을 계속 표시하고 공개 link를 `null`로 반환한다. 참조 Chunk 보관 기간, 별도 source snapshot으로의 FK 전환과 최종 hard delete 시점은 정책 확정 전까지 바꾸지 않는다.

### 7.2 실시간 STT와 로컬 녹음 분기

1. 교수자 브라우저가 하나의 microphone source를 실시간 PCM 경로와 로컬 binary 녹음 경로로 동시에 분기한다.
2. 실시간 경로는 audio WS로 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk를 전송한다. event WS와 녹음 resumable upload에는 이 frame을 보내지 않는다.
3. 로컬 경로는 브라우저에서 수업 전체 녹음을 지속하고 Session 종료 뒤 HTTP resumable upload로 전송한다. codec·container와 브라우저 저장 방식은 미정이다.
4. 실시간 전송과 로컬 녹음은 독립된 상태·오류 경계를 갖는다. 한 경로가 실패해도 다른 경로, 질문과 이미 저장된 final Transcript는 계속 사용할 수 있다.
5. 첫 성공 `audio.start`는 publisher `client_stream_id` claim과 논리 SessionRecording 생성·`CAPTURING` 전이를 한 transaction에서 확정한다. 다른 ID는 `4409 AUDIO_PUBLISHER_CONFLICT`로 거부하고 같은 ID만 reconnect·sequence resume한다.
6. class 시작 뒤 `source=LIVE`, `status=FINALIZING`인 TranscriptVersion을 만들고 실시간 STT의 임시 결과는 저장하지 않는다. 확정 결과만 이 LIVE version의 Segment로 저장한 뒤 event WS로 전파한다.
7. 순서가 바뀐 live chunk를 복구할 수 있도록 Session ID, stream ID와 sequence를 사용하되 stream ID 원문과 PCM payload를 영구 원장이나 로그에 남기지 않는다.

#### 7.2.1 녹음 upload·playback

1. Session 종료 transaction은 Session을 즉시 `PROCESSING`으로 바꾸고 새 audio 입력·resume을 차단하며 기존 Recording을 `CAPTURING → UPLOAD_PENDING`으로 전이한다.
2. 같은 publisher의 교수자 client가 Recording upload를 초기화하면 `UPLOAD_PENDING → UPLOADING`으로 전이한다. init, offset 조회, binary chunk append와 complete endpoint는 resumable resource 경계를 제공하지만 method·header·chunk·checksum·expiry 세부는 provisional이다.
3. 서버가 확인한 연속 offset과 전체 byte를 만족한 complete transaction만 final metadata와 논리 storage locator, `UPLOADED`, `source=RECORDING`·`status=FINALIZING`인 다음 TranscriptVersion, `RECORDING_TRANSCRIPTION` Job과 안전한 outbox를 함께 commit한다. 녹음 upload가 완료되기 전에는 HQ STT를 시작하지 않는다.
4. SessionRecording은 외부에 하나인 논리 aggregate다. final locator와 temporary handle은 단일 파일, 여러 part·fragment 또는 manifest 중 어느 물리 구조도 의미할 수 있으며 구체 cardinality는 미정이다.
5. playback은 `UPLOADED` Recording에만 제공하고 요청할 때마다 현재 인증·Course 접근과 추후 확정할 녹음 정책을 다시 검증한다. proxy streaming 또는 짧은 opaque delivery URL 방식은 미정이며 내부 key·경로·manifest는 반환하지 않는다.
6. capture·upload·storage 검증 실패는 Recording `FAILED`로 표현한다. 실패 단계별 upload 재시도 방식은 미정이지만 Session 전체 `PROCESSING` 10분 제한을 넘긴 비terminal Recording·Upload gate는 `FAILED`로 종료해 다음 class를 막지 않는다.

#### 7.2.2 Transcript version·canonical 전환

1. `TranscriptVersion.source`는 실시간 확정 발화 원장인 `LIVE`와 업로드 녹음 전체 재처리 결과인 `RECORDING`으로 구분한다. `FINALIZING`은 조립 중, `FINALIZED`는 Segment가 하나 이상인 기술적 성공, `EMPTY`는 정상 처리됐지만 Segment가 0개, `FAILED`는 처리 실패다.
2. class 시작 transaction은 LIVE version을 만들고 실시간 REST·RAG 기본 source용 canonical 포인터로 설정한다. 현재 audio runtime은 final Segment·Gap을 이 `FINALIZING` version에 기록하고, Session 종료 시 새 frame을 fence한다. LIVE version을 Segment 수에 따라 `FINALIZED` 또는 `EMPTY`로 terminal 처리하는 시점은 후처리 coordinator가 담당한다. 정상 RECORDING `FINALIZED` 또는 `EMPTY`가 이 포인터를 교체한다. HQ `FAILED` 또는 HQ 결과 없이 Session timeout이 된 뒤 보존된 LIVE 포인터를 완료 기록의 final source로 인정할지는 미정이며, 구현이 status만 보고 final source를 확정하지 않는다.
3. `RECORDING_TRANSCRIPTION`은 `visibility=SHARED`, `blocks_session_completion=true`이고 Recording 하나를 target으로 한다. 재시도는 같은 Job 행의 `attempt + 1`을 사용하되 실패 version을 덮어쓰지 않고 attempt마다 새 RECORDING version을 만든다.
4. HQ Worker는 비canonical version 아래 Segment·Gap만 비공개로 조립한다. 둘 다 같은 `transcript_version_id`에 속하며 Gap의 `start_ms`, `end_ms`는 class(Session) timeline의 누락 구간만 뜻한다. Segment만 녹음 seek용 `recording_start_ms`, `recording_end_ms`를 저장하고, RECORDING version에서 재처리 후에도 복구하지 못한 Gap만 `is_final=true`다.
5. Worker는 Job ID·attempt·run token·`RUNNING`, version `FINALIZING`, Session과 Recording 범위를 다시 확인한다. 전체 녹음 재처리, Segment·Gap 저장, Segment의 class·recording 시간 매핑과 Gap의 class timeline 검증이 성공하면 version 상태, Session `canonical_transcript_version_id`, Job `SUCCEEDED`와 `transcript.version.updated` outbox를 짧은 transaction으로 commit한다. Segment가 있으면 `FINALIZED`, 없으면 `last_sequence=0`인 `EMPTY`다. `EMPTY`에도 확정 Gap이 있으면 해당 Gap만 timeline으로 조회할 수 있다.
6. Session 종료 transaction은 `SESSION_POSTPROCESSING`을 `SHARED`, `blocks_session_completion=true`, `PENDING` coordinator로 미리 만든다. Worker claim은 같은 Session의 Recording·HQ source가 terminal인지 다시 검증하며 `UPLOAD_PENDING`, `UPLOADING` 또는 실행 중인 `RECORDING_TRANSCRIPTION`이 있으면 이 Job을 가져가지 않는다.
7. 새 KnowledgeChunk는 `source_transcript_version_id`를 가진다. RAG SQL은 Session canonical 포인터와 source version이 같은 Chunk만 선택하므로 전환 직후 재연결이 끝나기 전에는 이전 LIVE Transcript Chunk를 새 canonical 근거로 사용하지 않는다. 과거 Evidence는 당시 version provenance를 유지한다.
8. HQ 처리 실패·Worker lease timeout·확정될 HQ Job timeout은 현재 attempt가 staged한 Segment·Gap을 삭제하거나 transaction rollback하고 `last_sequence=0`, RECORDING version `FAILED`, Job `FAILED`와 outbox를 원자적으로 commit한다. 일반 후처리의 기본 5분을 HQ에도 적용할지 별도 상한을 둘지는 미정이며 Session 전체 10분 deadline은 항상 적용한다. 기존 canonical 포인터는 유지하며 latest version 상태와 `is_canonical`은 독립적으로 조회한다. timeout된 이전 Worker 결과는 fence를 통과하지 못해 저장되지 않는다.
9. source가 `FINALIZED`, `EMPTY`, `FAILED` 또는 unavailable terminal이 되면 coordinator를 claim할 수 있다. Answer mapping과 Transcript KnowledgeChunk 재연결은 latest HQ source가 RECORDING `FINALIZED`일 때만 수행한다. `EMPTY`·`FAILED`·HQ 무결과 deadline·Recording source 없음에서는 해당 작업과 Summary 상태를 terminal로 확정하되 종료 transaction에서 이미 만든 Transcript 비의존 최종 질문 clustering은 계속 실행한다.
10. Answer의 `source_transcript_version_id`와 원본 LIVE Segment 범위는 변경하지 않고, 새 RECORDING version별 mapping을 `PENDING`, `SUCCEEDED`, `FAILED`로 저장한다. 일부 mapping 실패는 HQ Transcript 자체를 실패시키지 않는다. 매칭 tolerance·겹침률·동률 해소 방식은 미정이다.
11. 자동 `FINAL_SUMMARY`는 latest HQ source가 RECORDING `FINALIZED`이고 Segment가 하나 이상일 때만 만든다. RECORDING `EMPTY`는 `NOT_APPLICABLE`·`NO_FINAL_TRANSCRIPT`, RECORDING `FAILED`·HQ 무결과 deadline·Recording source 없음은 `SUMMARY_SOURCE_UNAVAILABLE`로 확정한다. 보존된 LIVE 포인터를 완료 기록 final source로 쓸지는 미정이므로 자동 `FINAL_SUMMARY` 입력으로 사용하지 않는다.
12. coordinator는 완료된 음성 Answer별 `ANSWER_ORGANIZATION`과 적용 가능한 `FINAL_SUMMARY` 같은 source 의존 downstream `SHARED` blocking Job을 만든다. `FINAL` `QUESTION_CLUSTERING`은 LIVE→PROCESSING 종료 transaction에서 대상이 있으면 즉시 생성되어 Recording·HQ source와 독립 실행한다. Answer 정리 입력은 성공한 HQ mapping을 우선하고 없으면 immutable 원본 LIVE 범위를 사용해 최초 Job 생성 시 고정한다. coordinator의 `SUCCEEDED` 또는 `FAILED`, mapping·Knowledge 결과, Summary 상태, child Job과 outbox를 같은 transaction에 commit하므로 terminal 순간 completion predicate가 새 원장을 빠뜨릴 수 없다.
13. Session이 이미 `COMPLETED`인 뒤 같은 `RECORDING_TRANSCRIPTION` 행의 retry가 성공하면 성공 transaction은 같은 `SESSION_POSTPROCESSING` 행을 `attempt + 1`로 자동 requeue한다. Session 상태·`completed_at`은 유지하고, 복구 coordinator가 새 canonical version의 Answer mapping·KnowledgeChunk 연결을 멱등 재조정한 뒤 새로 생성 자격을 얻은 FINAL Summary Job을 만든다. 기존 `ANSWER_ORGANIZATION` Job·결과의 immutable source는 바꾸거나 자동 재생성하지 않는다.
14. `transcript.version.updated`는 결과 본문이 아니라 cache invalidation 알림이다. 클라이언트는 `GET /api/v1/sessions/{id}/transcript/versions`와 version-bound union cursor 조회를 다시 호출하고, 한 page에서 분리된 `segments[]`·`gaps[]` REST 결과를 최종 진실로 사용한다.

### 7.3 질문 클러스터링

#### 7.3.1 LIVE incremental clustering

1. 실제 학생 질문을 commit할 때 `QuestionClusteringState.requested_sequence`를 하나 증가시키고 새 Question의 `clustering_sequence`에 같은 값을 기록한다.
2. 같은 Session에 active `QUESTION_CLUSTERING` `PENDING`·`RUNNING` Job과 retry-reserved Job이 모두 없으면 현재 `requested_sequence`를 `input_through_sequence`로, `current_revision`을 `base_revision`으로 capture한 `LIVE_INCREMENTAL`, `SHARED`, `blocks_session_completion=false` Job과 queue outbox를 같은 transaction에서 자동 생성한다. active 또는 retry-reserved Job이 있으면 Job을 더 만들지 않고 `requested_sequence > applied_sequence`만 남겨 대기 질문을 coalesce한다.
3. Worker는 `(applied_sequence, input_through_sequence]` 범위의 새 질문만 읽는다. 새 질문을 기존 Cluster에 배치하거나 새 Cluster를 만들 수는 있지만 기존 질문을 다른 Cluster로 이동시키지 않는다.
   같은 semantic LIVE Cluster는 generation별 물리 row가 바뀌어도 stable `logical_cluster_id`를 계승한다. 기존 질문의 logical ID가 바뀌면 결과 commit을 거부하며 FINAL full rebuild만 새 logical ID를 할당한다.
4. 새 질문이 배치된 영향 Cluster만 새 immutable `AIRepresentativeQuestion`을 생성한다. 모든 대상 Cluster의 대표질문 생성까지 완료되어야 Job을 `SUCCEEDED`로 바꿀 수 있다.
5. 성공 transaction은 Job의 `base_revision = current_revision` fence를 확인하고 새 generation·typed member·대표질문, Job `SUCCEEDED`, `applied_sequence = input_through_sequence`, `current_revision + 1`을 함께 commit한다. retry-reserved Job도 원래 capture한 `input_through_sequence`까지만 적용한다. 이 시점에 `requested_sequence > applied_sequence`면 현재 requested·revision을 capture한 다음 `LIVE_INCREMENTAL` Job과 outbox를 같은 transaction에서 자동 생성해 재시도 중 추가된 질문을 처리한다.
6. 실패 transaction은 staged Cluster·member·대표질문을 반영하지 않고 기존 generation·`applied_sequence`·`current_revision`을 유지한다. retryable `FAILED` 행이 해당 `(applied_sequence, input_through_sequence]` 구간의 retry-reserved Job으로 남고 공개 `clustering_state.retry_job_id`가 이를 식별한다. 예약 중 새 질문 commit은 active Job이 없어도 새 Job을 만들지 않고 `requested_sequence`만 올린다. scheduler는 같은 AIJob 행을 원래 `input_through_sequence`·`base_revision` 그대로 `attempt + 1`, `PENDING`으로 바꾸고 `retry_job_id`를 비운다. 이후에는 active Job 제약이 fresh Job 생성을 막으며 backoff·최대 횟수는 미정이다.

새 generation은 변경되지 않은 Cluster도 기존 대표질문 ID와 membership을 유지한 snapshot으로 복사한다. 새 generation을 원자 공개한 뒤 직전 Cluster·membership은 삭제해 일반 변경 이력을 보관하지 않되 `PRESERVED` 대표질문의 immutable text·Job provenance와 Answer target은 유지한다. 미답변 old 대표질문은 관련 KnowledgeChunk Evidence가 없으면 hard delete하고, 있으면 내부 `DISCARDED` tombstone으로 전이한다. 내부 `requested_sequence`·`applied_sequence`는 API `clustering_state.requested_through_sequence`·`applied_through_sequence`로 표현한다.

#### 7.3.2 대표질문 보존과 final full clustering

- LIVE generation이 교체될 때 Answer가 없는 과거 대표질문은 Cluster·공개 조회·UI·RAG에서 즉시 폐기한다. 관련 KnowledgeChunk를 참조하는 Evidence가 없으면 대표질문을 hard delete해 Chunk를 cascade하고, 있으면 `status=OPEN`, `lifecycle_status=DISCARDED`, `discarded_at` 내부 tombstone과 Chunk를 유지한다. Evidence는 원래 kind·label과 `link=null`을 반환한다. `CAPTURING` 또는 `COMPLETED` Answer의 target인 대표질문은 `lifecycle_status=PRESERVED`로 보존해 새 대표질문 아래 typed member 자식으로 둔다. `status=OPEN|SELECTED|ANSWERED`는 Answer 시작·완료·취소에 따라 별도로 전이한다.
- deferred integrity는 `DISCARDED` 대표질문이 Answer·Cluster 중앙·membership 0건, 관련 Evidence 최소 1건인지 검증한다. 마지막 Evidence 삭제 transaction은 tombstone 대표질문과 Chunk를 함께 hard delete해 orphan을 남기지 않는다. Course·Session aggregate 삭제는 Evidence와 이 행들을 같은 deferred transaction에서 cascade하며, 정확한 보관 기간은 Evidence 정책을 따른다.
- Session 종료 transaction은 active 또는 retry-reserved `LIVE_INCREMENTAL` Job의 run token과 retry 예약을 폐기하고 Session 상태·revision fence로 늦은 결과와 LIVE 재실행을 막는다. 같은 transaction의 FINAL Job이 LIVE 입력을 대체하므로 LIVE Job은 `SUPERSEDED`, `retryable=false`로 끝낸다. 대체 작업 없이 명시 중단하는 Job만 `CANCELLED`이며, `SESSION_ENDED` 같은 별도 상태를 만들지 않는다.
- LIVE→PROCESSING 종료 transaction은 종료 시점 `requested_sequence`를 학생 질문 상한, `ended_at`을 AI 대표질문의 `COMPLETED` Answer 상한으로 저장한다. 대상이 있으면 두 상한에 포함되는 input을 처음부터 다시 배치하고 각 final Cluster의 immutable 대표질문까지 생성하는 `FINAL` `QUESTION_CLUSTERING` `SHARED` blocking Job과 outbox를 같은 transaction에서 만든다. 성공 결과는 eligible input 각각을 정확히 하나의 Cluster에 포함하고 분류할 수 없는 입력도 `기타` Cluster에 넣는다. 이 Job은 Recording·HQ Transcript·coordinator와 독립 실행한다. 실패한 FINAL을 교수가 같은 행으로 명시 재시도하면 학생 watermark는 유지하고 `base_revision=current_revision`, Answer 상한은 재시도 수락 시각으로 다시 캡처해 그 사이 새로 답변된 대표질문을 포함한다. 대표질문 생성과 membership 완전성 검증이 끝나기 전에는 Job을 성공 처리하지 않는다.
- final generation은 `is_final=true`, `finalized_at`과 생성 Job·attempt provenance를 가진다. AI 대표질문은 생성 당시 `created_in_generation`을 공개하고 Cluster child는 `source_kind=STUDENT_QUESTION|AI_REPRESENTATIVE`로 구분한다. Cluster 목록은 `ordinal ASC, id ASC`, child 목록은 generation 안에서 유일한 `ordinal ASC`로 안정적으로 정렬하며 공개 ordinal은 DB `position` projection이다. 공개 `clustering_state.last_job`은 마지막 clustering Job의 식별자·attempt·mode·`PENDING`·`RUNNING`·`SUCCEEDED`·`FAILED`·`CANCELLED`·`SUPERSEDED` 상태를 결과와 따로 복구한다.
- Cluster·member cursor endpoint, 기본 20·최대 100 page size와 안정 정렬은 확정한다. 정확한 embedding·유사도 기준, model·prompt, LIVE 재시도 backoff·최대 횟수, FINAL 입력 0건의 Job·빈 generation 표현, generation 교체 cursor 재시작, 한 번에 preload할 page 수와 대규모 mind map 점진 loading·layout은 미정이다.

### 7.4 교수자 답변 연결

1. LIVE에서 교수자가 학생 질문 또는 AI 대표질문을 선택하면 `answer_type=VOICE`, typed target과 현재 LIVE Transcript 위치를 가진 `CAPTURING` Answer를 만든다. AI 대표질문의 exact text는 Answer에 immutable snapshot으로 복사한다.
2. 완료 처리는 같은 `source_transcript_version_id`의 선택 이후 live final Segment 구간을 원본 범위로 저장하고 Answer를 `COMPLETED`로 바꾼다. 대표질문 target을 완료해도 해당 Cluster의 학생 질문들은 각각 미답변 상태를 유지한다.
3. LIVE 답변 취소는 `CAPTURING` Answer 행을 hard delete한다. 학생 질문과 현재 `ACTIVE` 대표질문은 미답변 상태로 돌린다. 이미 `PRESERVED`인 과거 대표질문은 membership을 제거하고 관련 Evidence가 없으면 hard delete, 있으면 `status=OPEN`, `PRESERVED → DISCARDED`, `discarded_at`으로 전이한다. 두 분기 모두 기존대로 `current_revision + 1`을 적용하고 active·retry-reserved LIVE Job을 `CLUSTER_REVISION_CHANGED`, `retryable=false`로 fence한 뒤 pending 질문이 있으면 현재 revision의 fresh Job을 만든다. `answer.deleted`와 `clustering.updated` outbox를 같은 transaction에 기록하되 `CANCELLED` 상태나 취소 기록은 보관하지 않는다. 마지막 Evidence 삭제 transaction의 원자 hard delete는 비공개 tombstone만 지우므로 revision을 다시 올리지 않는다.
4. Session `COMPLETED`에서 교수자는 미답변 학생 질문에만 `answer_type=TEXT`인 text-only `COMPLETED` Answer를 새로 만들 수 있다. 학생 질문 또는 Answer 때문에 보존된 AI 대표질문의 기존 `COMPLETED` Answer에는 텍스트를 추가·수정할 수 있지만 최종 복습용·미답변 `ACTIVE` 대표질문에 새 text-only Answer를 만들 수 없다. LIVE에서는 text Answer를 작성하지 않는다.
5. canonical 전환 뒤 `SESSION_POSTPROCESSING`은 음성 기반 Answer의 원본 LIVE 범위를 녹음 시간축으로 다시 찾아 `answer_transcript_mappings`에 기록한다. 성공 mapping은 canonical Segment를 우선 표시하고 실패하면 원본 LIVE 범위와 실패 상태를 유지한다.
6. coordinator는 완료된 음성 Answer마다 `ANSWER_ORGANIZATION`, `SHARED`, `blocks_session_completion=true` Job 하나를 만든다. 성공한 HQ mapping 범위를 우선하고 없으면 원본 LIVE 범위를 source로 고정한다.
7. Worker는 immutable target 질문 snapshot과 고정 Transcript 범위만으로 정리문을 생성하고 `AnswerOrganization` 결과와 Job `SUCCEEDED`를 원자 commit한다. 수업 후 수정 가능한 교수자 `text_content`는 모델 입력에서 제외한다. 실패·timeout은 해당 Job만 `FAILED`로 격리하며 Course 교수자는 같은 source와 Job 행의 `attempt + 1`로 재시도한다. 성공 결과의 재생성은 MVP에서 제공하지 않는다.
8. AI 정리문과 교수자 `Answer.text_content`는 별도 원장이다. 완료 기록은 교수자 text를 우선 표시하되 AI 정리문과 원본 음성 범위를 별도 표시하고, AI Worker는 교수자 text를 수정하지 않는다. 정리문 형식·최대 길이·model·prompt·품질 기준과 교수자 text 수정 뒤 재색인은 미정이다.

### 7.5 강의 기반 AI 채팅

1. Course membership과 Chat owner를 확인한다. 역할은 구분하지 않고 `mode=LIVE`는 Session `LIVE`, `mode=REVIEW`는 `COMPLETED`인지 검증한다.
2. USER Message를 trim·Unicode NFC 정규화한 뒤 Unicode code point 1~2,000자인지 검증한다. 위반은 `422 VALIDATION_ERROR`다. purge-scoped 멱등성 행을 ID 순서로 먼저 잠근 뒤 `Session → Chat` 순서로 잠그고 정규화된 USER Message, 이 행을 immutable `target_user_message_id`로 가진 `CHAT_RESPONSE` requester-only Job, 내부 queue outbox와 terminal `202`를 한 transaction으로 commit한다. Worker는 이 commit 뒤에만 claim한다.
3. 질문을 임베딩하고 같은 세션의 PDF KnowledgeChunk, canonical Transcript와 Q&A를 검색한다. PDF Chunk는 연결된 `READY` Material만, Transcript Chunk는 Session `canonical_transcript_version_id`와 source version이 같은 행만, AI 대표질문 Chunk는 `lifecycle_status IN ('ACTIVE', 'PRESERVED')`인 source만 SQL에서 선택한다.
4. 검색 결과와 제한된 최근 대화 내역으로 LLM 컨텍스트를 구성한다.
5. LLM이 검색된 근거를 우선 사용해 최종 답변을 생성한다. 검색 근거가 부족하면 이를 명시한다.
6. Worker와 retry는 Job의 immutable USER Message target과 그 시점까지의 허용된 history를 입력으로 사용한다. 최종 Assistant Message, 필수 생성 Job·attempt, 내부 `knowledge_chunk_id`, 순위, non-null 안전 `label_snapshot`과 nullable model·prompt metadata를 같은 Job 성공 transaction에 저장한다. 생성 중 delta·실패 attempt 본문은 저장하지 않는다. 앞서 저장한 USER Message에는 Job·attempt·model·prompt metadata를 두지 않는다.
7. 요청자는 Job을 polling하고 성공 뒤 저장된 Message를 REST로 조회한다. 새로고침 복구를 위해 USER Message의 공개 `response_job_id`는 자신을 `target_user_message_id`로 가진 유일한 Job에서 계산한다. Assistant는 producer Job을 `job_id`로 공개하고 `response_job_id=null`이며, 이 두 projection을 Message 중복 컬럼으로 저장하지 않는다. 개인 결과는 공용 Session event로 발행하지 않는다.
8. 조회 시 typed source를 `MATERIAL|TRANSCRIPT|QUESTION|ANSWER`로 projection하고 `label_snapshot`을 공개 `label`로, 현재 접근 가능한 상대 경로를 `link`로 계산한다. 학생 질문과 AI 대표질문은 모두 `QUESTION`이다. 접근 가능한 source의 link는 non-null이며, Material이 연결 해제되었거나 AI 대표질문이 `DISCARDED`인 경우처럼 안전한 경로가 없으면 source kind·label은 유지하고 link만 `null`이다.

Evidence link는 cursor와 `/record` 배열 위치에 의존하지 않는다. Material은 권한 검사를
거치는 content 경로와 page가 있을 때만 붙이는 fragment, 학생 질문·AI 대표 질문·Answer는 각 단건 공개 ID,
Transcript는 Session·TranscriptVersion과 stable sequence 또는 time range를 사용하며 내부 Chunk ID·storage key·서버
경로와 signed storage URL을 반환하지 않는다.

### 7.6 강의 종료 후처리

1. 종료 요청 transaction은 purge-scoped idempotency 행을 Session보다 먼저 잠근다. 그 뒤 Session과 개인 결과 aggregate·Job을 잠그고 LIVE Summary·Chat·Message·Evidence와 관련 `LIVE_SUMMARY`·`CHAT_RESPONSE` requester-only Job을 삭제한 다음 세션을 `PROCESSING`으로 변경한다. purge-scoped 멱등성 record의 조기 삭제와 종료 뒤 polling·재요청 응답은 미정이다. FINAL Summary와 REVIEW Chat은 유지한다. 새 audio·질문·반응·Answer·개인 LIVE AI 입력과 audio resume을 차단하고 active 또는 retry-reserved LIVE clustering Job의 run token·retry 예약을 폐기해 Session 상태·revision fence로 늦은 결과를 막는다. 같은 종료 transaction의 FINAL clustering이 LIVE 입력을 대체하므로 LIVE Job은 `SUPERSEDED`, `JOB_SUPERSEDED`, `retryable=false`로 terminal 처리한다. `SESSION_POSTPROCESSING`, `SHARED`, `blocks_session_completion=true`, `PENDING` coordinator를 만들고, FINAL 대상이 1건 이상이면 종료 시점 `requested_sequence`, `current_revision`, `ended_at`을 입력 상한으로 고정한 별도 `FINAL` `QUESTION_CLUSTERING` Job과 queue·event outbox도 같은 transaction에서 만들어 늦은 결과와 조기 완료를 막는다. 종료 API 응답은 coordinator와 이 조건부 FINAL Job을 함께 반환한다.
2. Recording이 있으면 `CAPTURING → UPLOAD_PENDING`으로 바꾸고 브라우저 로컬 녹음의 resumable upload를 기다린다. `audio.stop`·drain 성공 여부로 Session 전이를 늦추지 않는다. LIVE TranscriptVersion은 drain 뒤 `FINALIZED` 또는 `EMPTY`로 끝낸다.
3. upload complete transaction이 Recording `UPLOADED`, RECORDING TranscriptVersion과 `RECORDING_TRANSCRIPTION` Job을 함께 commit한 뒤에만 전체 녹음 기반 HQ STT를 시작한다. 이 Job은 `SHARED`이며 Session 완료를 차단한다.
4. Recording·Upload·HQ source가 terminal이 될 때까지 coordinator는 claim 대상에서 제외한다. HQ Worker는 version별 Segment·Gap을 비공개로 준비하고 성공 transaction에서 canonical 포인터를 원자적으로 전환하며, `FAILED`·`EMPTY`도 source terminal wakeup을 발생시킨다.
5. source terminal 뒤 coordinator는 latest HQ source가 RECORDING `FINALIZED`일 때만 Answer mapping과 canonical KnowledgeChunk 재연결을 수행한다. RECORDING `EMPTY`는 `NOT_APPLICABLE`·`NO_FINAL_TRANSCRIPT`, RECORDING `FAILED`·HQ 무결과 deadline·Recording source 없음은 `SUMMARY_SOURCE_UNAVAILABLE`로 Summary 상태를 확정한다. final clustering input은 이미 종료 transaction에서 학생 질문 sequence와 Answer 완료 시각 상한으로 고정됐으므로 coordinator는 이를 다시 캡처하거나 새 FINAL Job을 만들지 않는다.
6. latest HQ source가 RECORDING `FINALIZED`이고 Segment가 하나 이상일 때만 `source_transcript_version_id`를 보존하는 `FINAL_SUMMARY` Job을 예약한다. 보존된 LIVE 포인터는 완료 기록 final source 인정 여부가 미정이므로 자동 Summary에 사용하지 않는다. LIVE에서 live final Segment가 0개인 개인 Summary 요청은 Job 없이 `409 SUMMARY_TRANSCRIPT_NOT_READY`로 거부한다.
7. coordinator 결과 transaction은 mapping·Knowledge와 Summary 상태, coordinator terminal, 완료 음성 Answer별 `ANSWER_ORGANIZATION`과 적용 가능한 `FINAL_SUMMARY` `SHARED` blocking Job, queue·event outbox를 한 번에 commit한다. 종료 transaction에서 먼저 만든 `FINAL` `QUESTION_CLUSTERING`을 포함해 모든 blocking Job 원장을 확인한 뒤에만 completion predicate를 다시 계산하므로 조기 `COMPLETED` race가 없다.
8. 각 결과를 독립적으로 저장하고 성공한 결과부터 조회 가능하게 한다. 녹음·HQ STT·Answer mapping·AI 작업 실패가 기존 PDF·질문·Answer와 이미 저장된 version 조회를 가리지 않는다.
9. 모든 완료 차단 Job과 Recording·Upload gate가 `SUCCEEDED` 또는 `FAILED` terminal이면 일부 실패가 있어도 Session을 `COMPLETED`로 바꾼다. 10분 deadline watchdog도 coordinator handoff invariant를 유지한다. Session·coordinator를 잠근 채 완료 음성 Answer별 누락 `ANSWER_ORGANIZATION`을 포함해 적용 가능하지만 아직 만들지 못한 downstream blocking Job을 `FAILED`, `retryable=true`, `error_code=SESSION_PROCESSING_TIMEOUT`, `started_at=NULL`인 terminal 행으로 원자 생성하고 Summary 상태·남은 gate·coordinator를 terminal 처리한 뒤 completion predicate를 평가한다. 완료 뒤 재시도해도 Session을 `PROCESSING`으로 되돌리지 않는다.

LIVE Chat Evidence cascade가 `DISCARDED` 대표질문의 마지막 Evidence를 삭제하면 기존 deferred cleanup이 tombstone과 KnowledgeChunk를 같은 종료 transaction에서 hard delete한다. 다른 Evidence가 남으면 tombstone을 유지한다. purge된 개인 Job의 늦은 Worker 결과는 Job·target 부재와 현재 attempt·run token·`RUNNING` fence를 통과하지 못하므로 삭제된 Summary·Chat을 되살리지 못한다.

FINAL Summary 공개 상태는 nullable Summary 한 필드만 보고 추측하지 않고 latest HQ RECORDING source, Session당 하나인 `FINAL_SUMMARY` Job과 현재 attempt의 결과 원장을 함께 대조한다.

- source gate가 아직 비terminal이면 Job 없이 `PENDING`이 정상이다.
- RECORDING `FINALIZED`와 Segment 1건 이상이면 coordinator가 FINAL Summary Job·queue outbox를 원자 생성한다. Session `PROCESSING`의 `PENDING`·`RUNNING` Job, `COMPLETED`에서 `attempt>1`인 명시적 Summary retry, 또는 HQ retry 성공 뒤 `SESSION_POSTPROCESSING attempt>1` 복구가 새로 만든 최초 `attempt=1` active FINAL Summary Job은 `PENDING`이다. 호환되는 결과를 같은 transaction에 저장한 `SUCCEEDED` Job은 `AVAILABLE`, 결과가 없는 `FAILED` Job은 `FAILED`다.
- RECORDING `EMPTY`는 Job 없이 `NOT_APPLICABLE/NO_FINAL_TRANSCRIPT`다. Recording source 없음, RECORDING `FAILED` 또는 HQ 무결과 deadline은 Job 없이 `FAILED/SUMMARY_SOURCE_UNAVAILABLE`다.
- 유효한 RECORDING `FINALIZED` source에 Job이 없거나, `SUCCEEDED` Job의 현재 attempt 결과가 없거나 source version·Segment 범위가 맞지 않거나, HQ recovery coordinator와 무관한 `COMPLETED`의 최초 `attempt=1` Job이 active이면 `DATA_INTEGRITY_ERROR`다. `EMPTY`·source unavailable에 Job·결과가 존재하는 반대 모순도 같은 오류이며 정상 null이나 watchdog 합성 실패 Job으로 숨기지 않는다.
- `READY`와 아직 개인 LIVE Summary를 요청하지 않은 `LIVE`의 `summary=null`·관련 Job 없음은 정상이다. LIVE 결과는 FINAL 상태 계산에 포함하지 않는다.

## 8. AI 실행 원칙

- 모델 입력에 실제 학생 식별 정보를 포함하지 않는다.
- 개인 AI 기능은 Course 역할을 구분하지 않고 현재 멤버에게 동일하게 제공하되 결과·Job은 요청자에게만 공개한다.
- 개인 AI 전송은 polling-only다. 성공한 최종 결과만 저장하고 생성 중 delta·부분 Assistant Message를 DB, shared event나 별도 stream에 남기지 않는다.
- 모든 검색은 `course_id`와 `session_id` 범위를 강제한다.
- 프롬프트에는 검색된 강의 문맥 밖의 내용을 추측하지 않도록 명시한다.
- 모델명, 프롬프트 버전, 처리 시각과 작업 결과를 추적할 수 있게 저장한다.
- STT·임베딩·LLM 모델은 인터페이스 뒤에 두어 교체 가능하게 구성한다.
- LLM과 Embedding 호출은 application 내부 비동기 provider protocol로만 수행한다. 이 경계는 HTTP API·AIJob producer·Worker registry가 아니며, 실제 외부 API와 local GPU runtime을 선택하거나 설정하지 않는다.
- provider 요청은 목적·입력·선택적 prompt version과 호출 deadline을 전달하고, 결과에는 생성 text 또는 입력 순서를 보존한 vector와 선택적 model 식별자를 반환한다. PR-18의 development/CI RAG profile은 `fake-embedding-v1`, 8차원, cosine distance를 사용하며 HNSW `m=16`, `ef_construction=64`를 적용한다. 실제 외부 provider·credential·GPU runtime은 선택하지 않았고, 다른 차원 도입은 migration과 전체 재색인 계약이 필요하다.
- provider는 `PROVIDER_TIMEOUT`, `PROVIDER_UNAVAILABLE`, `PROVIDER_RATE_LIMITED`, `PROVIDER_INVALID_RESPONSE`처럼 안전한 오류 code와 retryable 여부만 호출자에게 전달한다. 호출 task의 cancellation은 실패 결과로 변환하지 않고 전파하며, 입력 원문·secret·외부 provider 원문 오류를 로그·공개 Job·API 응답에 넣지 않는다.
- 로컬 deterministic fake는 외부 네트워크·GPU 없이 단위 테스트를 재현하기 위해 구현됐다. 실제 모델 adapter, credential·환경 변수와 GPU 동시성 설정은 아직 없다.
- `KNOWLEDGE_INDEXING`은 SHARED·non-blocking Session Job이다. 같은 Session의 READY Material, canonical Transcript final Segment와 이미 존재하는 수동 Answer text만 멱등 색인한다. PDF는 1,000 Unicode code point 이하의 공백 경계 page Chunk, Transcript는 final Segment 하나를 안정적인 sequence 범위 Chunk로 사용한다. 새 READY Material과 final Segment commit은 Session당 하나의 active 색인 Job을 예약할 수 있다. Answer writer는 PR-16 범위이므로 text 생성·수정 후 재색인 enqueue는 그 PR의 명시적 연계 작업이다.
- GPU 작업의 동시 실행 수를 제한해 메모리 부족이 다른 요청으로 확산되지 않게 한다.

## 9. 파일 저장 및 보안

- 현재 로컬 개발은 `STORAGE_ROOT` 아래 temporary/final namespace를 사용하는 filesystem `Storage` adapter를 제공한다. 녹음 원본 저장·resumable upload·playback 구현과 외부 Object Storage는 아직 없다.
- 파일을 실제로 다루는 업무 코드는 provider SDK나 filesystem 경로를 직접 사용하지 않고 application 내부 `Storage` protocol만 사용한다. 이 protocol은 HTTP API가 아니며 권한·MIME 검사·Course 상태 전이도 담당하지 않는다.
- `StorageKey`는 서버가 생성하는 불투명 내부 식별자다. logical key와 physical path를 분리하고, temporary namespace와 final namespace를 구분한다. 사용자 파일명·URL·storage provider handle을 key로 사용하지 않는다.
- temporary object append는 caller가 제공한 `expected_offset`과 현재 길이가 같을 때만 성공하고 확인된 다음 offset을 반환한다. bytes checksum은 SHA-256으로 검증하며, final promotion은 검증된 temporary object만 원자적으로 공개한다. final key가 이미 같은 SHA-256 object를 가리키면 멱등 성공으로, 다른 object를 가리키면 충돌로 처리한다.
- 내부 Range read는 `[start, end)` byte 범위로 정의한다. object가 없는 delete는 성공으로 취급하고, retry 가능·불가능한 storage 실패를 구분한다. API의 HTTP Range/오류 변환은 Recording playback PR에서 정의한다.
- database transaction과 storage write는 하나의 원자 transaction이 아니므로, domain service는 DB commit 실패 뒤 stage된 temporary object를 보상 삭제해야 한다. commit 뒤의 물리 삭제 재시도·outbox consumer·quota 반환과 orphan 실제 정리는 lifecycle·운영 PR의 책임이다.
- 녹음 원본 저장과 권한 확인 playback은 MVP 목표다. 97 GiB KCloud 루트 디스크는 운영 녹음 원장의 장기 저장소로 충분하다고 가정하지 않으며, 운영 MVP에는 외부 Object Storage 또는 동등한 내구성·용량 경계와 capacity monitoring이 필요하다.
- 배포 환경에서는 파일 경로를 데이터베이스에 직접 저장하지 않고 서버가 생성한 논리 storage key를 저장한다. Recording final key와 Upload temporary key는 물리 파일 하나를 뜻하지 않는다.
- object key, 서버 경로, fragment key와 manifest는 API 응답·공유 event·애플리케이션 로그에 공개하지 않는다. PDF는 업로드 원본명과 안정적인 공개 `display_name`을 분리해 보관한다.
- Evidence 응답에는 내부 `knowledge_chunk_id`, Chunk content와 source storage locator를 넣지 않는다. 공개 label은 생성 시 저장한 안전한 snapshot이고 link는 권한 검사를 다시 거치는 상대 REST 경로 또는 `null`이다.
- 업로드 파일은 확장자뿐 아니라 MIME 형식과 PDF 파싱 가능 여부를 검사한다.
- 모든 HTTP와 WebSocket 요청에 인증 및 역할 기반 권한 검사를 적용한다.
- 녹음 metadata, upload와 playback은 매 요청에서 인증과 Course 접근 권한을 재검증한다. 녹음의 정확한 역할별 접근 범위와 동의 정책은 미정이다.
- 참여 코드 HMAC key와 암호화 key를 분리한다. 제품의 코드 회전은 현재 코드 값을 바꾸는 작업이고 비밀키 회전은 별도 운영 절차다.
- 인증 토큰, 참여 코드, 질문 원문과 개인정보를 애플리케이션 로그에 남기지 않는다.
- 브라우저 녹음 codec·container·local storage, resumable protocol·checksum·expiry·최대 크기·fragment manifest는 확정 전까지 구성값이나 구현 전제로 만들지 않는다.
- 녹음 동의, 역할별 접근, 보관 기간과 삭제 정책은 외부 공개 전에 확정해야 한다. 현재 문서는 default 동의나 무기한 보관을 가정하지 않는다.

## 10. 비동기 작업과 장애 처리

- PDF 처리, 클러스터링, 요약과 후처리는 요청과 분리된 작업으로 실행한다.
- 작업은 `PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED`, `CANCELLED`, `SUPERSEDED` 상태와 안전한 오류 정보를 저장한다. timeout·worker lease 만료는 재시도 후보인 `FAILED`이고, 명시 중단은 `CANCELLED`, 대체 실행은 `SUPERSEDED`이며 마지막 두 상태는 재시도하지 않는다.
- 외부 모델 호출에는 타임아웃과 Job 유형별 재시도 정책을 적용한다. LIVE clustering의 backoff·최대 횟수는 확정 전까지 지수 백오프나 특정 횟수를 가정하지 않는다.
- 실패한 작업의 사용자 재시도는 새 Job을 만들지 않는다. 같은 행을 `attempt + 1`, `version + 1`, `PENDING`으로 바꾸고 progress·error·실행 시각·run token을 초기화한다.
- worker 결과 commit은 Job ID, 현재 attempt, run token과 `RUNNING` 상태가 모두 일치할 때만 허용해 이전 attempt의 늦은 결과를 폐기한다. Material 처리에는 대상 Material version과 `detached_at IS NULL` 조건도 추가한다.
- 개인 Job은 generic Job-first claim에서 제외한다. 후보는 잠금 없이 찾고 LIVE Summary claim·result·retry는 `purge-scoped IdempotencyRecord → Session → AIJob`, LIVE Chat은 `purge-scoped IdempotencyRecord → Session → Chat → target USER Message → AIJob` 순서로 잠근다. REVIEW Chat은 purge prefix 없이 같은 suffix를 사용한다. 마지막 Job 잠금 뒤 status·available time·attempt·run token·requester·immutable target·Session mode를 모두 재검증한다.
- 공개 Job 표현은 `visibility`, `attempt`, `version`, 안전한 progress, `retryable`, `blocks_session_completion`, result와 nullable `result_unavailable_reason`, `updated_at`을 포함하고 provider 내부 정보는 제외한다.
- Job 재시도 queue event와 멱등성 `202` 응답은 같은 DB transaction에 기록한다. `IdempotencyRecord`의 `PROCESSING` lease는 60초이고, 유효한 동일 요청은 중복 실행 대신 `409 IDEMPOTENCY_REQUEST_IN_PROGRESS`를 반환한다. terminal 응답은 AES-256-GCM으로 암호화해 정확히 24시간 보관하며, 이 암호화 키는 production 환경 변수 또는 KMS에서만 주입한다.
- Course 참여 코드는 별도의 AES-256-GCM 표시용 키와 HMAC-SHA-256 조회용 키를 사용한다. production에서는 `COURSE_JOIN_CODE_ENCRYPTION_KEY`와 `COURSE_JOIN_CODE_LOOKUP_KEY`를 서로 다른 base64 key material로 주입하며 원문과 key를 로그에 남기지 않는다.
- 한 작업의 실패가 다른 강의 세션이나 핵심 질문 기능을 차단하지 않게 격리한다.
- LIVE `QUESTION_CLUSTERING` 실패는 기존 generation·`applied_sequence`·`current_revision`을 유지하고 `requested_sequence > applied_sequence`를 pending 신호로 남긴다. 미적용 captured 구간은 공개 `clustering_state.retry_job_id`가 가리키는 retry-reserved Job이 소유하며, 예약 중 새 질문은 `requested_sequence`에만 합쳐진다. scheduler는 그 행을 원래 `input_through_sequence`·`base_revision` 그대로 `attempt + 1`, `PENDING`으로 바꾸면서 retry 예약을 해제하고 queue outbox를 commit한다. backoff·최대 횟수는 미정이다.
- 질문 watermark 증가, active·retry Job 생성·예약·requeue, 성공 generation, Session 종료 예약 해제, FINAL Job 생성·terminal 등 공개 `QuestionClusteringState`가 변하는 모든 transaction은 최신 projection의 `clustering.updated` event outbox를 함께 commit한다.
- `ANSWER_ORGANIZATION`은 Answer당 하나인 `SHARED` blocking Job이고 source Transcript version·Segment 범위를 최초 생성 때 고정한다. 결과 row와 성공은 원자 commit하고 실패 재시도는 같은 source·Job 행의 `attempt + 1`을 사용한다.
- Worker는 RUNNING Job의 lease를 15초마다 갱신하고 만료 시각을 현재부터 60초 뒤로 연장한다. 60초 동안 갱신되지 않으면 watchdog이 해당 Job을 `FAILED`로 terminal 처리한다. 이 신호는 브라우저 WebSocket ping/pong과 별개다.
- 일반 후처리 Job은 `started_at`부터 최대 5분이다. HQ STT에도 이 기본값을 적용할지 별도 상한을 둘지는 미정이며, 제한이 확정된 뒤 초과 Job을 `FAILED`로 끝내고 현재 run token과 lease를 제거한다.
- Session 전체 `PROCESSING` 제한은 `ended_at`부터 10분이다. deadline watchdog은 Session과 `SESSION_POSTPROCESSING` coordinator를 잠근 뒤 latest HQ source와 질문 대상을 기준으로 적용 가능하지만 아직 생성되지 않은 downstream blocking Job을 계산한다. `FINAL_SUMMARY`를 제외한 빠진 Job은 `status=FAILED`, `retryable=true`, `error_code=SESSION_PROCESSING_TIMEOUT`, `started_at=NULL`인 terminal 행으로 만든다. 누락된 FINAL clustering은 `input_through_sequence=requested_sequence`, `base_revision=current_revision`, `final_answered_through_at=lecture_sessions.ended_at`으로 원래 종료 상한을 복원한다. Summary 상태·남은 Recording·Upload·source gate·기존 blocking Job·coordinator terminal 전이를 같은 transaction에 commit하고, eligible source의 FINAL Summary Job 누락은 합성하지 않고 `DATA_INTEGRITY_ERROR`로 관측한다. completion predicate는 이 원장이 모두 생긴 뒤에만 평가한다.
- `SESSION_POSTPROCESSING`은 종료 transaction에서 미리 만든 `SHARED` blocking coordinator다. Recording·HQ source terminal 전에는 claim하지 않고, terminal 뒤 mapping·Knowledge·Summary 상태를 정리한다.
- coordinator의 terminal 전이와 완료 음성 Answer별 `ANSWER_ORGANIZATION`, 적용 가능한 `FINAL_SUMMARY` blocking Job, queue·event outbox 생성은 같은 transaction이다. `FINAL` `QUESTION_CLUSTERING`과 attempt별 input snapshot은 종료 transaction에서 먼저 생성하며 completion reconciliation은 두 transaction의 blocking 원장을 모두 확인한다.
- 모든 완료 차단 Job과 gate가 terminal이면 실패가 있어도 Session을 `COMPLETED`로 전환한다. watchdog이 만든 누락 downstream terminal 행은 FINAL Summary 누락을 제외하고 일반 실패 Job과 같은 재시도 원장이며, 실패한 AI 기능을 완료 기록에서 재시도해도 Session은 다시 `PROCESSING`으로 돌아가지 않는다.
- 모든 결과 commit은 현재 Job ID·attempt·run token·`RUNNING`과 대상 version·resource fence가 일치해야 한다. watchdog 이후 늦게 도착한 이전 Worker 결과는 저장하지 않는다.
- 개인 LIVE 결과 purge 뒤의 Worker도 동일 fence에 더해 Job과 Summary·Chat target 존재 검증에 실패하므로 결과를 commit하지 않는다.
- Recording upload complete는 HQ STT 시작의 선행 gate이며 `RECORDING_TRANSCRIPTION` Job과 RECORDING TranscriptVersion을 같은 transaction에서 만든다.
- Recording finalize는 검증된 final metadata·`UPLOADED`, RECORDING TranscriptVersion, `RECORDING_TRANSCRIPTION` Job과 queue outbox를 한 transaction에 commit한다. object publish 뒤 DB commit이 실패하면 동기 보상 또는 내부 cleanup outbox로 orphan을 제거한다.
- upload 만료·실패, aggregate 삭제와 중단된 temporary object 정리는 outbox consumer가 멱등 처리하고 object가 이미 없으면 성공으로 간주한다. exact reconciliation 주기와 재시도 한도는 미정이다.

### 10.1 Aggregate 삭제

- Material이 포함된 흐름은 `Session → Material → AIJob`, Recording 흐름은 `Session → Recording → Upload → TranscriptVersion → AIJob` 순서를 사용한다. class 삭제는 `purge-scoped IdempotencyRecord → Session → Material → Recording → Upload → TranscriptVersion → AIJob`, Course 삭제는 앞에 `Course`를 추가한 순서로 잠가 worker·시작·upload·삭제 요청과 교착을 피한다. 개인 LIVE 요청과 Session 종료도 purge-scoped idempotency 행을 Session보다 먼저 잠그는 동일한 예외 순서를 사용한다.
- DB 행을 삭제하기 전에 PDF, Recording final object와 active Upload temporary object의 내부 key를 수집하고 같은 transaction에 멱등 cleanup outbox task를 남긴다. 물리 storage object는 FK cascade로 정리됐다고 간주하지 않는다.
- aggregate와 함께 삭제된 Job은 현재 attempt·run token·상태와 target 존재 조건을 더 이상 만족하지 않으므로 늦은 worker 결과를 저장할 수 없다. LIVE Chat Evidence cascade가 마지막 참조를 지운 `DISCARDED` 대표질문·Chunk도 같은 deferred transaction에서 정리한다.
- 삭제와 멱등성 `204` terminal 응답을 같은 transaction에 기록해 리소스가 사라진 뒤 같은 요청을 반복해도 기존 응답을 반환한다.

## 11. 관측성

- 모든 요청과 AI 작업에 추적 가능한 요청 ID 또는 작업 ID를 부여한다.
- API 오류율, 응답 시간, WebSocket 연결 수와 재접속률을 수집한다.
- STT 확정 지연, HQ STT 실행 시간·queue depth·oldest age, TranscriptVersion 상태 체류·canonical 전환 시간과 실패율, attempt별 staged Segment·Gap rollback·삭제 실패, `last_sequence=0` reset 실패와 `EMPTY` final Gap-only timeline 수를 수집한다.
- Worker heartbeat 갱신 지연, 60초 lease timeout, 일반 Job 5분 timeout, HQ 실행 시간, 10분 Session watchdog terminal 처리 수, `SESSION_PROCESSING_TIMEOUT`으로 합성한 누락 downstream Job 수, eligible FINAL Summary Job 누락 `DATA_INTEGRITY_ERROR` 수와 늦은 결과 폐기 수를 수집한다.
- Answer canonical mapping 상태·실패율과 KnowledgeChunk canonical source 불일치 차단 수를 수집한다.
- `/record` manifest 조회 latency와 리소스별 count·전용 cursor page 조회 실패율을 분리해 수집한다. count와 page가 다른 시점 snapshot일 수 있는 정상 변화는 무결성 오류로 집계하지 않는다.
- Session별 clustering `requested_sequence`·`applied_sequence`·`input_through_sequence` 격차, current/base revision 불일치, coalesced 질문 수, active Job 중복 차단, generation 적용 지연·실패·재시도, 종료 fence와 late result 폐기 수를 수집한다. 종료 Job의 terminal 상태별 지표는 enum 결정 뒤 추가한다.
- `SESSION_POSTPROCESSING` dependency 대기 시간, source terminal부터 claim까지의 지연, coordinator terminal과 downstream Job 원자 생성 실패·reconciliation 수, completion 전에 적용 가능한 downstream 원장이 모두 존재하는지 검사한 handoff invariant 위반 수를 수집한다.
- audio publisher 충돌·동일 stream resume, Recording 상태 체류 시간, upload offset·재개·실패, playback 권한 거부와 orphan cleanup 실패를 수집한다.
- PDF·녹음 final·upload temporary object의 사용량과 증가율, KCloud 루트 디스크 여유·소진 예상 시간을 함께 모니터링한다. quota와 alert threshold는 미정이다.
- GPU 메모리, 사용률과 모델별 동시 실행 수를 모니터링한다.
- 로그는 구조화 형식으로 기록하고 민감정보 필터를 적용한다.

## 12. 배포 기준

- 로컬 개발에서는 PostgreSQL을 Docker Compose로 실행한다.
- 애플리케이션 설정은 환경 변수로 주입하고 비밀값을 저장소에 커밋하지 않는다.
- 데이터베이스 변경은 Alembic 마이그레이션으로 관리한다.
- 배포 서버의 GPU는 STT 또는 로컬 AI 추론에 사용할 수 있으며 상세 사양은 VM 문서를 따른다.
- API, AI 작업 프로세스와 데이터베이스의 상태 점검 방법을 각각 제공한다.
- 데이터베이스, PDF와 녹음 object는 독립적인 백업·복구 절차를 갖는다. 녹음 원장과 resumable temporary object를 KCloud 루트 디스크 하나에만 두는 운영 구성은 사용하지 않는다.
- 외부 Object Storage의 provider·region·암호화·수명주기와 DB metadata 사이 복구 절차, quota·backup·RPO/RTO는 운영 배포 전에 확정한다.
