Redis와 별도 벡터 데이터베이스는 실제 부하가 확인될 때 검토한다.

위 표는 현재 저장소의 구성 상태이고, 아래의 Course·class·녹음·멱등성·AIJob 규칙은 구현이 따라야 할 MVP 목표 계약이다. 목표 계약을 현재 구현 완료 상태로 해석하지 않는다.

### 처리 경계

- API 서버는 요청 검증, 권한 확인, 상태 변경과 실시간 전파를 담당한다.
- 오래 걸리는 PDF·STT·LLM 작업은 API 요청 처리와 분리한다.
- 데이터베이스를 최종 상태의 기준으로 사용하고 WebSocket은 변경 알림 용도로 사용한다.
- AI 기능이 실패해도 수업방, 질문과 저장된 기록 기능은 계속 동작해야 한다.
- 실시간 PCM 전송과 브라우저 로컬 녹음은 하나의 microphone source에서 분기하되 독립 상태로 관리한다. 한 경로의 실패가 다른 경로와 질문·이미 저장된 final Transcript를 자동 실패 처리하지 않는다.

## 4. 데이터 구조 원칙

상세 스키마는 별도 DB 문서에서 정의하되 다음 데이터 영역을 기준으로 설계한다.

| 영역 | 핵심 엔티티 |
|---|---|
| 사용자·권한 | User, CourseMember |
| 수업 | Course, LectureSession, LectureMaterial |
| 강의 기록 | SessionRecording, RecordingUpload, TranscriptSegment, LectureSummary |
| 질문·답변 | Question, QuestionReaction, QuestionCluster, Answer |
| AI 채팅 | ChatSession, ChatMessage |
| 비동기 처리 | AIJob, KnowledgeChunk |

### 제약조건 원칙

- 모든 하위 데이터는 `course_id` 또는 `session_id`를 통해 소속 범위를 확인할 수 있어야 한다.
- Course 생성자는 해당 Course의 불변 owner이자 유일한 `PROFESSOR`다. 교수자 추가와 owner 이전은 제공하지 않으며 DB가 owner와 교수자 membership의 정확히 한 개 불변식을 검증한다.
- 참여 코드는 trim·대문자 정규화 후 `[A-Z]{6}`이고 자동 만료하지 않는다. 조회용 HMAC과 표시용 암호문을 분리하며 owner 회전이 commit되면 이전 코드는 즉시 무효화하고 이력을 남기지 않는다.
- Course에는 `READY`, `LIVE`, `PROCESSING` 상태를 합쳐 active class가 최대 하나다. partial UNIQUE를 최종 동시성 방어선으로 사용한다.
- 같은 날짜 class는 순차적으로 여러 개 허용하고 완료 목록은 `lecture_date`, 실제 `started_at`, `id` 내림차순으로 안정적으로 정렬한다.
- class 제목은 모든 상태에서 수정할 수 있다. 날짜와 이미 기록된 시작·종료·완료 시각은 변경하지 않는다.
- `detached_at IS NULL`인 Material만 class에 연결된 자료다. Session당 최대 10개, 파일당 decimal `100000000` bytes를 DB constraint·trigger와 서비스 transaction이 함께 검증한다.
- 업로드 원본명과 생성 뒤 바꾸지 않는 `display_name`을 분리한다. 표시 이름 충돌은 확장자 앞 ` (1)`, ` (2)` suffix로 해결하고 같은 내용의 재업로드는 허용한다.
- 질문 반응은 `(question_id, user_id)` 조합을 고유하게 만들어 중복을 방지한다.
- Session당 외부에 노출하는 논리 SessionRecording은 최대 하나다. 첫 `audio.start`의 `client_stream_id` claim과 `CAPTURING` 전이는 원자적으로 처리하고 원문 stream ID는 저장하지 않는다.
- SessionRecording은 물리 object 개수와 무관한 논리 aggregate다. 단일 파일인지 fragment·manifest 집합인지와 DB·스토리지의 물리 cardinality는 미정이며 외부 계약에 노출하지 않는다.
- Transcript는 시간 순서 복구가 가능하도록 시작·종료 시각과 확정 여부를 저장한다.
- 임베딩 검색에는 반드시 수업방 및 세션 범위 필터를 적용한다. Material source는 같은 SQL에서 `processing_status = 'READY' AND detached_at IS NULL`도 강제한다.

### 상태 표현

```text
LectureSession: READY → LIVE → PROCESSING → COMPLETED
Recording:      CAPTURING → UPLOAD_PENDING → UPLOADING → UPLOADED | FAILED
RecordingUpload: ACTIVE → COMPLETED | EXPIRED | FAILED
Question:       OPEN → SELECTED → ANSWERED
AIJob:          PENDING → RUNNING → SUCCEEDED | FAILED
```

Course 자체에는 종료·보관 상태를 두지 않는다. 상태 전이는 서버에서 검증하며 잘못된 순서의 요청은 거부한다. 재시도가 필요한 AI 작업은 같은 Job 행의 attempt를 증가시키고 현재 attempt·run token을 확인해 중복·늦은 결과를 차단한다.

## 5. API 설계 원칙

- REST API는 수업방, 세션, 자료, 녹음, 질문, 반응, 답변, 요약과 채팅 리소스를 중심으로 구성한다.
- 인증된 사용자 ID를 요청 본문에서 신뢰하지 않고 인증 컨텍스트에서 가져온다.
- 모든 리소스 접근 시 수업방 소유자 또는 참여 학생 여부를 검사한다.
- owner 전용 Course 참여 코드 회전·Course 삭제·class 관리 요청은 생성자와 현재 `PROFESSOR` membership이 모두 일치하는지 확인한다.
- 질문 반응 추가·취소와 강의 종료 요청은 반복 호출에 안전하게 설계한다.
- 멱등성 terminal 응답은 terminal 전이 시각부터 정확히 24시간 같은 status와 body를 재사용한다.
- 목록 조회는 커서 기반 페이지네이션을 기본으로 한다.
- 오류 응답은 코드, 사용자 메시지와 요청 ID를 일관된 형식으로 반환한다.
- PDF 다운로드는 권한 확인을 거친 API 또는 만료 시간이 짧은 서명 URL을 사용한다.
- 녹음 metadata·upload·playback 요청은 매 요청에서 현재 인증과 Course 접근 권한을 다시 확인한다. storage key, 서버 경로, fragment key와 manifest는 응답·공유 event·로그에 노출하지 않는다.

정확한 경로, 요청·응답 스키마와 오류 코드는 API 문서에서 관리하며 구현 코드의 OpenAPI 출력을 최종 계약으로 사용한다.

### Course·class lifecycle

- `current_session`은 해당 Course의 유일한 `READY`, `LIVE`, `PROCESSING` class이며 없으면 `null`이다. active class가 `COMPLETED`가 된 뒤에만 다음 class를 생성한다.
- class 생성 경쟁은 하나만 성공하고 나머지는 `ACTIVE_SESSION_EXISTS`로 응답한다. 허용되지 않은 상태 전이·삭제는 `SESSION_STATE_CONFLICT`로 구분한다.
- 빈 class 제목은 Course 제목·class 날짜·시각을 포함한 서버 자동 제목으로 치환한다. 정확한 문자열 형식, `READY`에서 사용할 시각 원장과 timezone은 미정이며 사용자는 `lecture_date`, `started_at`, `ended_at`, `completed_at`을 수정할 수 없다.
- class 시작은 Session과 연결된 Material을 잠근 뒤 `PROCESSING`이 하나라도 있으면 `MATERIAL_PROCESSING_ACTIVE`로 거부한다. PDF가 0개이거나 연결된 Material이 `READY`, `UPLOADED`, `FAILED`뿐이면 시작할 수 있다.
- owner는 `READY`, `COMPLETED` class만 aggregate 단위로 삭제할 수 있다. `LIVE`는 종료하고 `PROCESSING`은 완료될 때까지 기다린다.
- owner는 Course를 삭제할 수 있지만 active class가 있을 때의 허용 여부와 삭제 후 복구 유예는 아직 미정이다. 구현은 이 두 정책을 임의로 확정하지 않는다.

## 6. 실시간 통신

클라이언트는 강의 세션 단위로 서로 다른 두 WebSocket에 연결한다. Session event WS는 Course 멤버의 공용 변경 알림을 전달하고, audio WS는 Course `PROFESSOR`의 실시간 PCM과 ack·flow control·resume만 담당한다. 두 채널은 별도 ticket·권한·연결 상태를 가지며 event WS로 audio frame이나 recording upload chunk를 보내지 않고 audio WS로 공용 Session event를 broadcast하지 않는다.

| 이벤트 | 용도 |
|---|---|
| `transcript.partial` | 아직 확정되지 않은 실시간 STT 결과 전파 |
| `transcript.final` | 저장된 확정 Transcript 구간 전파 |
| `question.created` | 새 질문 전파 |
| `question.updated` | 질문 상태 또는 클러스터 변경 전파 |
| `reaction.updated` | 질문별 반응 수 변경 전파 |
| `answer.updated` | 답변 연결과 완료 상태 전파 |
| `recording.updated` | 안전한 Recording 상태 변경 전파 |
| `summary.completed` | 요청된 실시간 요약 결과 전파 |
| `job.updated` | AI 작업 진행·완료·실패 상태 전파 |
| `session.updated` | 강의 시작·종료와 후처리 상태 전파 |

### 연결 복구

- 서버는 주기적인 ping/pong으로 연결 상태를 확인한다.
- 재접속 시 클라이언트는 마지막으로 반영한 이벤트 시각 또는 버전을 전달한다.
- 누락 데이터는 REST API로 다시 조회해 화면 상태를 복원한다.
- 이벤트 중복 수신을 허용하고 클라이언트는 리소스 ID와 버전을 기준으로 중복을 제거한다.
- audio WS에서 첫 성공 `audio.start`의 `client_stream_id`가 Session의 단독 publisher를 claim한다. 다른 ID의 연결은 `AUDIO_PUBLISHER_CONFLICT`와 close code `4409`로 음성 전송만 거부하고, 같은 ID는 새 ticket으로 reconnect·resume할 수 있다.
- 비정상 단절 뒤 publisher lease 만료, 재획득과 다른 탭·장치 takeover는 미정이다. 클라이언트나 서버가 임의 timeout으로 소유권을 넘기지 않는다.

## 7. 주요 처리 파이프라인

### 7.1 PDF 전처리

1. API 서버가 Course의 `PROFESSOR` 권한, PDF MIME·parsing 가능 여부와 `1..100000000` bytes 크기를 검사한다.
2. 원본 PDF를 파일 스토리지에 server-generated storage key로 저장한다. 이 키는 내부 식별자이며 API·공유 event·로그에 노출하지 않는다.
3. Session을 잠그고 `READY`, `LIVE`, `COMPLETED`인지, 연결된 Material이 10개 미만인지 다시 확인한다. `PROCESSING` Session의 업로드는 거부한다.
4. 원본명과 별도인 안정적인 `display_name`을 할당하고 Material, `MATERIAL_PROCESSING` Job, queue outbox와 제공된 멱등성 키의 terminal 응답을 한 transaction에서 만든다. 이름 충돌은 확장자 앞 ` (1)`, ` (2)` suffix로 해결하고 기존 이름은 재번호를 매기지 않는다.
5. AI 작업 프로세스가 페이지별 텍스트와 메타데이터를 추출하고 검색 가능한 크기로 분할해 임베딩을 생성한다.
6. Worker는 현재 Job attempt·run token·상태와 Material version·`detached_at IS NULL`을 다시 확인한다.
7. 문서 조각·페이지 번호·벡터 저장, Material `READY`와 Job `SUCCEEDED`를 한 transaction으로 확정하고 클라이언트에 알린다.

MVP의 `MAX_UPLOAD_BYTES` 목표값은 decimal `100000000`이다. 현재 저장소는 PyMuPDF dependency와 `STORAGE_ROOT` directory 설정만 있고 upload handler·크기 제한·전처리 pipeline은 아직 구현하지 않았다. 같은 내용과 같은 원본 파일명의 PDF는 별도 Material로 업로드할 수 있다.

#### 7.1.1 Material 연결 해제

- 해당 Course의 `PROFESSOR`는 Session `READY`, `LIVE`, `COMPLETED`에서 Material을 연결 해제할 수 있다. Session `PROCESSING`에서는 `SESSION_STATE_CONFLICT`로 거부한다.
- transaction은 `Session → Material → AIJob` 순서로 잠그고 `detached_at`, Material `version`, 내부 object·Knowledge cleanup outbox와 필수 멱등성 키의 terminal `204`를 함께 commit한다. 결과가 없는 `PENDING`·`RUNNING`·`FAILED` Material Job은 제거하고, 결과 provenance로 참조되는 `SUCCEEDED` Job은 보존한다.
- commit 즉시 Material 목록·상세·content와 RAG 검색에서 제외한다. `processing_status`에 `DETACHED`를 추가하지 않고, cleanup 실패가 tombstone을 되돌리지 않는다.
- Worker claim과 결과 commit은 연결된 Material만 대상으로 하고 version을 확인하므로 연결 해제 전 작업의 늦은 결과를 저장하지 않는다.
- Outbox consumer는 object와 참조되지 않은 Material source KnowledgeChunk를 멱등 삭제한다. 연결 해제 Material의 원문 content link는 제공하지 않는다. 기존 Chat Evidence가 참조하는 Chunk의 보관·FK·snapshot·과거 표시 방식·최종 hard delete 시점은 정책 확정 전까지 바꾸지 않는다.

### 7.2 실시간 STT와 로컬 녹음 분기

1. 교수자 브라우저가 하나의 microphone source를 실시간 PCM 경로와 로컬 binary 녹음 경로로 동시에 분기한다.
2. 실시간 경로는 audio WS로 `PCM_S16LE`, 16 kHz, mono, 500 ms chunk를 전송한다. event WS와 녹음 resumable upload에는 이 frame을 보내지 않는다.
3. 로컬 경로는 브라우저에서 수업 전체 녹음을 지속하고 Session 종료 뒤 HTTP resumable upload로 전송한다. codec·container와 브라우저 저장 방식은 미정이다.
4. 실시간 전송과 로컬 녹음은 독립된 상태·오류 경계를 갖는다. 한 경로가 실패해도 다른 경로, 질문과 이미 저장된 final Transcript는 계속 사용할 수 있다.
5. 첫 성공 `audio.start`는 publisher `client_stream_id` claim과 논리 SessionRecording 생성·`CAPTURING` 전이를 한 transaction에서 확정한다. 다른 ID는 `4409 AUDIO_PUBLISHER_CONFLICT`로 거부하고 같은 ID만 reconnect·sequence resume한다.
6. 실시간 STT의 임시 결과는 저장하지 않고 event WS로 빠르게 전파한다. 확정 결과는 시작·종료 시각과 함께 Transcript로 저장한 뒤 전파한다.
7. 순서가 바뀐 live chunk를 복구할 수 있도록 Session ID, stream ID와 sequence를 사용하되 stream ID 원문과 PCM payload를 영구 원장이나 로그에 남기지 않는다.

#### 7.2.1 녹음 upload·playback

1. Session 종료 transaction은 Session을 즉시 `PROCESSING`으로 바꾸고 새 audio 입력·resume을 차단하며 기존 Recording을 `CAPTURING → UPLOAD_PENDING`으로 전이한다.
2. 같은 publisher의 교수자 client가 Recording upload를 초기화하면 `UPLOAD_PENDING → UPLOADING`으로 전이한다. init, offset 조회, binary chunk append와 complete endpoint는 resumable resource 경계를 제공하지만 method·header·chunk·checksum·expiry 세부는 provisional이다.
3. 서버가 확인한 연속 offset과 전체 byte를 만족한 complete transaction만 final metadata와 논리 storage locator, `UPLOADED`, 안전한 `recording.updated`와 HQ STT wakeup outbox를 함께 commit한다. 녹음 upload가 완료되기 전에는 HQ STT를 시작하지 않는다.
4. SessionRecording은 외부에 하나인 논리 aggregate다. final locator와 temporary handle은 단일 파일, 여러 part·fragment 또는 manifest 중 어느 물리 구조도 의미할 수 있으며 구체 cardinality는 미정이다.
5. playback은 `UPLOADED` Recording에만 제공하고 요청할 때마다 현재 인증·Course 접근과 추후 확정할 녹음 정책을 다시 검증한다. proxy streaming 또는 짧은 opaque delivery URL 방식은 미정이며 내부 key·경로·manifest는 반환하지 않는다.
6. capture·upload·storage 검증 실패는 Recording `FAILED`로 표현할 수 있다. 실패 단계별 재시도, upload 만료 뒤 Recording 전이와 Session 완료 규칙은 미정이며 HQ STT Job·Transcript version·canonical 상태·timeout·Segment offset·Answer 재매핑은 PR4에서 확정한다.

### 7.3 질문 클러스터링

1. 새 질문의 임베딩을 생성한다.
2. 같은 세션의 열린 질문 또는 클러스터와 의미 유사도를 계산한다.
3. 기준 이상이면 기존 클러스터에 배치하고, 아니면 새 클러스터를 만든다.
4. LLM이 AI 대표 질문의 정확한 text를 Cluster `title`로 생성한다.
5. 같은 결과 세트의 Session별 증가 `generation`, 안정적인 `ordinal`, `is_final`, `finalized_at`, `created_by_job_id`, `created_by_job_attempt`을 저장하고 `question.updated` 이벤트로 알린다.

유사도 기준과 자동 재계산 주기는 평가 데이터로 결정한다. generation 원자 할당과 최신 watermark·late-result fence의 전체 설계는 후속 계약으로 남긴다. 처리 실패 시 질문의 `cluster_id`를 비워 두고 원본 질문 흐름을 유지한다.

### 7.4 교수자 답변 연결

1. 교수자가 질문을 선택하면 서버가 선택 시각과 현재 Transcript 위치를 기록한다. Cluster 선택이면 Cluster ID와 당시 `title`, 즉 AI 대표 질문 exact text도 immutable snapshot으로 보관한다.
2. 완료 처리 시 선택 이후의 확정 Transcript 구간을 답변 후보로 조회한다.
3. 연결 범위를 저장하고 질문을 `ANSWERED` 상태로 변경한다.
4. 후처리 과정에서 LLM이 답변 구간을 간결하게 정리할 수 있다.

### 7.5 강의 기반 AI 채팅

1. 질문자의 세션 접근 권한을 확인한다.
2. 질문을 임베딩하고 같은 세션의 PDF 조각, Transcript와 Q&A를 검색한다. PDF 조각은 연결된 `READY` Material만 SQL에서 선택한다.
3. 검색 결과와 제한된 최근 대화 내역으로 LLM 컨텍스트를 구성한다.
4. LLM이 검색된 근거를 우선 사용해 답변을 생성한다.
5. 검색 근거가 부족하면 이를 명시하는 응답을 생성한다.
6. 질문, 답변, 사용한 근거 ID와 모델 정보를 저장한다.

### 7.6 강의 종료 후처리

1. 종료 요청 transaction에서 세션 상태를 즉시 `PROCESSING`으로 변경하고 새 audio·질문·반응·Answer 입력과 audio resume을 차단한다.
2. Recording이 있으면 `CAPTURING → UPLOAD_PENDING`으로 바꾸고 브라우저 로컬 녹음의 resumable upload를 기다린다. `audio.stop`·drain 성공 여부로 Session 전이를 늦추지 않는다.
3. upload complete가 Recording `UPLOADED`와 wakeup outbox를 commit한 뒤에만 전체 녹음 기반 HQ STT를 시작한다. 구체 Job type과 완료 차단 predicate는 PR4에서 정의한다.
4. HQ STT 결과를 기반으로 Final Transcript를 만들고 이후 전체 Transcript, PDF와 Q&A에 대한 최종 요약·클러스터링을 실행해야 한다. Transcript version·canonical 전환과 기존 Answer 재매핑은 PR4 계약을 따른다.
5. 각 결과를 독립적으로 저장하고 성공한 결과부터 조회 가능하게 한다. 녹음·HQ STT·AI 작업 실패가 기존 PDF·질문·Answer와 이미 저장된 Transcript 조회를 가리지 않는다.
6. Recording 미생성·`FAILED`, upload 만료·timeout과 HQ STT 실패일 때의 최종 Session 완료 조건은 PR4 전까지 임의로 확정하지 않는다.

## 8. AI 실행 원칙

- 모델 입력에 실제 학생 식별 정보를 포함하지 않는다.
- 모든 검색은 `course_id`와 `session_id` 범위를 강제한다.
- 프롬프트에는 검색된 강의 문맥 밖의 내용을 추측하지 않도록 명시한다.
- 모델명, 프롬프트 버전, 처리 시각과 작업 결과를 추적할 수 있게 저장한다.
- STT·임베딩·LLM 모델은 인터페이스 뒤에 두어 교체 가능하게 구성한다.
- GPU 작업의 동시 실행 수를 제한해 메모리 부족이 다른 요청으로 확산되지 않게 한다.

## 9. 파일 저장 및 보안

- 현재 로컬 개발은 `STORAGE_ROOT` directory만 준비한다. 녹음 원본 저장·resumable upload·playback 구현과 외부 Object Storage는 아직 없다.
- 녹음 원본 저장과 권한 확인 playback은 MVP 목표다. 97 GiB KCloud 루트 디스크는 운영 녹음 원장의 장기 저장소로 충분하다고 가정하지 않으며, 운영 MVP에는 외부 Object Storage 또는 동등한 내구성·용량 경계와 capacity monitoring이 필요하다.
- 배포 환경에서는 파일 경로를 데이터베이스에 직접 저장하지 않고 서버가 생성한 논리 storage key를 저장한다. Recording final key와 Upload temporary key는 물리 파일 하나를 뜻하지 않는다.
- object key, 서버 경로, fragment key와 manifest는 API 응답·공유 event·애플리케이션 로그에 공개하지 않는다. PDF는 업로드 원본명과 안정적인 공개 `display_name`을 분리해 보관한다.
- 업로드 파일은 확장자뿐 아니라 MIME 형식과 PDF 파싱 가능 여부를 검사한다.
- 모든 HTTP와 WebSocket 요청에 인증 및 역할 기반 권한 검사를 적용한다.
- 녹음 metadata, upload와 playback은 매 요청에서 인증과 Course 접근 권한을 재검증한다. 녹음의 정확한 역할별 접근 범위와 동의 정책은 미정이다.
- 참여 코드 HMAC key와 암호화 key를 분리한다. 제품의 코드 회전은 현재 코드 값을 바꾸는 작업이고 비밀키 회전은 별도 운영 절차다.
- 인증 토큰, 참여 코드, 질문 원문과 개인정보를 애플리케이션 로그에 남기지 않는다.
- 브라우저 녹음 codec·container·local storage, resumable protocol·checksum·expiry·최대 크기·fragment manifest는 확정 전까지 구성값이나 구현 전제로 만들지 않는다.
- 녹음 동의, 역할별 접근, 보관 기간과 삭제 정책은 외부 공개 전에 확정해야 한다. 현재 문서는 default 동의나 무기한 보관을 가정하지 않는다.

## 10. 비동기 작업과 장애 처리

- PDF 처리, 클러스터링, 요약과 후처리는 요청과 분리된 작업으로 실행한다.
- 작업은 `PENDING`, `RUNNING`, `SUCCEEDED`, `FAILED` 상태와 오류 정보를 저장한다.
- 외부 모델 호출에는 타임아웃과 제한된 지수 백오프 재시도를 적용한다.
- 실패한 작업의 사용자 재시도는 새 Job을 만들지 않는다. 같은 행을 `attempt + 1`, `version + 1`, `PENDING`으로 바꾸고 progress·error·실행 시각·run token을 초기화한다.
- worker 결과 commit은 Job ID, 현재 attempt, run token과 `RUNNING` 상태가 모두 일치할 때만 허용해 이전 attempt의 늦은 결과를 폐기한다. Material 처리에는 대상 Material version과 `detached_at IS NULL` 조건도 추가한다.
- 공개 Job 표현은 `visibility`, `attempt`, `version`, 안전한 progress, `retryable`, `blocks_session_completion`, `updated_at`을 포함하고 provider 내부 정보는 제외한다.
- Job 재시도 queue event와 멱등성 `202` 응답은 같은 DB transaction에 기록한다.
- 한 작업의 실패가 다른 강의 세션이나 핵심 질문 기능을 차단하지 않게 격리한다.
- 프로세스 재시작 후 `RUNNING` 상태에 남은 작업을 복구하거나 실패 처리한다.
- Recording upload complete는 HQ STT 시작의 선행 gate다. 구체 HQ STT Job type·timeout·Transcript 결과와 Session 완료 predicate는 PR4 전까지 기존 AIJob 규칙에 끼워 넣지 않는다.
- Recording finalize는 검증된 final metadata·`UPLOADED`·HQ STT wakeup outbox를 한 transaction에 commit한다. object publish 뒤 DB commit이 실패하면 동기 보상 또는 내부 cleanup outbox로 orphan을 제거한다.
- upload 만료·실패, aggregate 삭제와 중단된 temporary object 정리는 outbox consumer가 멱등 처리하고 object가 이미 없으면 성공으로 간주한다. exact reconciliation 주기와 재시도 한도는 미정이다.

### 10.1 Aggregate 삭제

- Material이 포함된 흐름은 `Session → Material → AIJob`, Recording 흐름은 `Session → Recording → Upload → AIJob` 순서를 사용한다. class 삭제는 `Session → Material → Recording → Upload → AIJob`, Course 삭제는 앞에 `Course`를 추가한 순서로 잠가 worker·시작·upload·삭제 요청과 교착을 피한다.
- DB 행을 삭제하기 전에 PDF, Recording final object와 active Upload temporary object의 내부 key를 수집하고 같은 transaction에 멱등 cleanup outbox task를 남긴다. 물리 storage object는 FK cascade로 정리됐다고 간주하지 않는다.
- aggregate와 함께 삭제된 Job은 현재 attempt·run token·상태 조건을 더 이상 만족하지 않으므로 늦은 worker 결과를 저장할 수 없다.
- 삭제와 멱등성 `204` terminal 응답을 같은 transaction에 기록해 리소스가 사라진 뒤 같은 요청을 반복해도 기존 응답을 반환한다.

## 11. 관측성

- 모든 요청과 AI 작업에 추적 가능한 요청 ID 또는 작업 ID를 부여한다.
- API 오류율, 응답 시간, WebSocket 연결 수와 재접속률을 수집한다.
- STT 확정 지연, AI 작업 처리 시간, 작업 대기열과 실패율을 수집한다.
- audio publisher 충돌·동일 stream resume, Recording 상태 체류 시간, upload offset·재개·실패, playback 권한 거부와 orphan cleanup 실패를 수집한다.
- PDF·녹음 final·upload temporary object의 사용량과 증가율, KCloud 루트 디스크 여유·소진 예상 시간을 함께 모니터링한다. quota와 alert threshold는 미정이다.
- GPU 메모리, 사용률과 모델별 동시 실행 수를 모니터링한다.
- 로그는 구조화 형식으로 기록하고 민감정보 필터를 적용한다.

## 12. 배포 기준

- 로컬 개발에서는 PostgreSQL을 Docker Compose로 실행한다.
- 애플리케이션 설정은 환경 변수로 주입하고 비밀값을 저장소에 커밋하지 않는다.
- 데이터베이스 변경은 Alembic 마이그레이션으로 관리한다.
- 배포 서버의 GPU는 STT 또는 로컬 AI 추론에 사용할 수 있으며 상세 사양은 VM 문서를 따른다.
- API, AI 작업 프로세스와 데이터베이스의 상태 점검 방법을 각각 제공한다.
- 데이터베이스, PDF와 녹음 object는 독립적인 백업·복구 절차를 갖는다. 녹음 원장과 resumable temporary object를 KCloud 루트 디스크 하나에만 두는 운영 구성은 사용하지 않는다.
- 외부 Object Storage의 provider·region·암호화·수명주기와 DB metadata 사이 복구 절차, quota·backup·RPO/RTO는 운영 배포 전에 확정한다.
