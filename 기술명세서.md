# 실시간 AI 강의 보조 서비스 기술 명세서

## 1. 문서 목적

본 문서는 ToBeDetermined의 1차 MVP를 구현하기 위한 기술 요구사항을 정의한다. 핵심 범위는 수업방과 강의 세션 관리, 실시간 강의 기록, 익명 질문, 질문 클러스터링, 강의 기반 AI 채팅 및 수업 종료 후 기록 생성이다.

## 2. 시스템 구성

| 구성 요소 | 책임 |
|---|---|
| 웹 클라이언트 | 교수자·학생 화면, PDF 열람, 질문 및 AI 채팅 UI |
| API 서버 | 인증, 수업방·세션·질문·반응·답변 관리 및 권한 검증 |
| 실시간 서버 | Transcript, 질문, 반응, 답변 상태를 참여자에게 즉시 전파 |
| AI 작업 서버 | STT, 요약, 질문 클러스터링, 문서 임베딩 및 RAG 답변 생성 |
| 관계형 데이터베이스 | 사용자, 수업방, 세션, 질문, 답변 및 작업 상태 저장 |
| 객체 스토리지 | 강의자료 PDF와 선택적으로 녹음 파일 저장 |
| 벡터 저장소 | PDF, Transcript, Q&A 검색용 임베딩 저장 |

### 권장 기술 조합

아래 조합은 구현 권장안이며 팀의 개발 환경에 따라 대체할 수 있다.

- 프론트엔드: React 또는 Next.js
- 백엔드: FastAPI 또는 NestJS
- 데이터베이스: PostgreSQL
- 실시간 통신: WebSocket
- 작업 큐 및 캐시: Redis 기반 큐
- 객체 스토리지: S3 호환 스토리지
- AI 처리: STT 모델 + 임베딩 모델 + LLM 기반 RAG

## 3. 사용자 및 권한

| 역할 | 주요 권한 |
|---|---|
| 교수자 | 수업방·세션 생성, 자료 업로드, 강의 시작·종료, 질문 선택 및 답변 완료 처리 |
| 학생 | 수업 코드 참여, 세션 입장, 질문 작성, 반응, Transcript 조회 및 AI 채팅 |

- 모든 자료와 기록은 해당 수업방의 교수자 또는 참여 학생만 접근할 수 있다.
- 학생 질문은 화면에 익명으로 표시하되, 중복 반응 방지와 신고 대응을 위해 서버에는 작성자 식별자를 보관한다.
- 수업 코드만으로 자료에 접근할 수 없으며, 코드 확인 후 수업방 참여 정보가 생성되어야 한다.

### 주요 상태값

- 강의 세션: `READY` → `LIVE` → `PROCESSING` → `COMPLETED`
- 질문: `OPEN` → `SELECTED` → `ANSWERED`
- AI 작업: `PENDING` → `RUNNING` → `SUCCEEDED` 또는 `FAILED`

## 5. 주요 API

세부 URL과 요청 형식은 구현 과정에서 조정할 수 있으나 아래 동작은 반드시 제공한다.

| 영역 | 메서드 및 경로 | 기능 |
|---|---|---|
| 수업방 | `POST /courses` | 수업방과 참여 코드 생성 |
| 수업방 | `POST /courses/join` | 수업 코드로 참여 |
| 수업방 | `GET /courses/{courseId}` | 수업방과 강의 목록 조회 |
| 세션 | `POST /courses/{courseId}/sessions` | 강의 세션 생성 |
| 세션 | `POST /sessions/{sessionId}/start` | 강의 시작 |
| 세션 | `POST /sessions/{sessionId}/end` | 강의 종료 및 후처리 시작 |
| 자료 | `POST /sessions/{sessionId}/materials` | PDF 업로드 |
| 질문 | `POST /sessions/{sessionId}/questions` | 익명 질문 작성 |
| 질문 | `GET /sessions/{sessionId}/questions` | 질문·클러스터 목록 조회 |
| 반응 | `PUT /questions/{questionId}/reaction` | ‘나도 궁금해요’ 추가 |
| 반응 | `DELETE /questions/{questionId}/reaction` | 반응 취소 |
| 답변 | `POST /questions/{questionId}/select` | 답변 대상 선택 |
| 답변 | `POST /questions/{questionId}/answers` | 답변 내용 또는 Transcript 구간 연결 |
| 요약 | `POST /sessions/{sessionId}/summaries` | 현재까지의 강의 요약 요청 |
| 채팅 | `POST /sessions/{sessionId}/chat` | 해당 강의 기반 AI 질의응답 |
| 기록 | `GET /sessions/{sessionId}/record` | 수업 후 통합 기록 조회 |

- 반응 API는 사용자와 질문의 조합에 대해 멱등성을 보장한다.
- 목록 API는 생성 시각 또는 커서를 이용한 페이지네이션을 지원한다.
- 파일 다운로드는 인증된 단기 접근 URL을 사용한다.

## 6. 실시간 통신

클라이언트는 강의 세션 단위의 WebSocket 채널에 접속한다. 연결 시 인증 토큰과 수업방 참여 권한을 검증한다.

| 이벤트 | 전달 내용 |
|---|---|
| `transcript.partial` | 아직 확정되지 않은 실시간 인식 결과 |
| `transcript.final` | 저장이 완료된 Transcript 구간 |
| `question.created` | 새 질문 |
| `question.updated` | 질문 상태 또는 클러스터 변경 |
| `reaction.updated` | 질문별 반응 수 변경 |
| `answer.updated` | 답변 연결 또는 완료 상태 |
| `summary.completed` | 요청한 실시간 요약 결과 |
| `job.updated` | AI 작업 진행·완료·실패 상태 |
| `session.updated` | 강의 시작·종료 및 후처리 상태 |

- 연결이 끊어진 클라이언트는 재접속 후 마지막 수신 시각을 기준으로 누락 데이터를 REST API에서 다시 조회한다.
- 실시간 이벤트는 화면 갱신용이며, 최종 데이터의 기준은 데이터베이스로 한다.

## 7. 기능별 처리 흐름

### 7.1 PDF 업로드

1. 서버가 파일 형식, 크기 및 접근 권한을 검사한다.
2. 원본 PDF를 객체 스토리지에 저장한다.
3. 비동기 작업이 페이지별 텍스트를 추출하고 적절한 크기로 분할한다.
4. 각 조각의 임베딩과 페이지 정보를 벡터 저장소에 저장한다.
5. 처리 상태를 `READY` 또는 `FAILED`로 갱신한다.

### 7.2 실시간 Transcript

1. 교수자 클라이언트가 음성을 일정 크기의 조각으로 전송한다.
2. STT 작업 서버가 임시 결과와 확정 결과를 생성한다.
3. 임시 결과는 즉시 전파하고, 확정 결과만 `TranscriptSegment`로 저장한다.
4. 각 구간에 시작·종료 시각을 기록해 질문 답변과 연결할 수 있게 한다.

### 7.3 익명 질문과 반응

1. 학생의 세션 참여 권한과 질문 내용을 검증한다.
2. 작성자 정보는 서버에 저장하되 다른 사용자에게는 노출하지 않는다.
3. 질문 생성 및 반응 수 변경을 세션 참여자에게 실시간 전파한다.
4. 교수자 화면은 미답변 질문을 반응 수 내림차순으로 정렬할 수 있어야 한다.

### 7.4 질문 클러스터링

1. 열린 질문의 임베딩을 생성한다.
2. 의미 유사도를 기준으로 기존 클러스터에 배치하거나 새 클러스터를 만든다.
3. LLM이 클러스터 제목과 대표 질문을 생성한다.
4. 질문 수가 증가하면 일정 주기 또는 임계 개수마다 다시 계산한다.
5. 클러스터링 실패 시 개별 질문 목록은 계속 사용할 수 있어야 한다.

### 7.5 교수자 음성 답변 연결

1. 교수자가 질문 또는 클러스터를 선택한다.
2. 서버가 선택 시점을 답변 시작 지점으로 기록한다.
3. 교수자가 완료 처리하면 해당 시간 범위의 확정 Transcript를 답변으로 연결한다.
4. 연결된 질문을 `ANSWERED`로 변경하고 모든 참여자에게 전파한다.

### 7.6 강의 기반 AI 채팅

1. 사용자 질문과 세션 접근 권한을 확인한다.
2. PDF, 현재까지의 Transcript 및 Q&A에서 관련 문맥을 검색한다.
3. 검색 결과와 최근 대화 내역을 LLM 입력으로 구성한다.
4. LLM은 제공된 강의 문맥을 우선 근거로 답변한다.
5. 근거가 부족하면 추측하지 않고 강의 기록에서 확인할 수 없음을 알린다.

### 7.7 강의 종료 후처리

1. 세션을 `PROCESSING`으로 변경하고 추가 실시간 입력을 차단한다.
2. 남은 Transcript와 답변 연결을 확정한다.
3. 강의 핵심 내용, 주요 개념, 질문·답변 및 질문 클러스터를 정리한다.
4. 결과 저장 후 기록 페이지를 생성하고 세션을 `COMPLETED`로 변경한다.
5. 일부 AI 작업이 실패하면 성공한 기록은 먼저 제공하고 실패 작업만 재시도할 수 있게 한다.

## 8. AI 컨텍스트 및 안전 원칙

- 실시간 요약은 해당 세션의 최근 Transcript와 강의자료 중 관련 부분을 사용한다.
- 복습 채팅은 선택한 강의 세션의 PDF, 전체 Transcript 및 Q&A만 기본 검색 범위로 사용한다.
- 서로 다른 수업방의 문서나 대화 기록이 검색 결과에 섞이지 않도록 모든 검색에 `course_id`와 `session_id` 필터를 적용한다.
- 프롬프트와 출력에 포함되는 개인정보를 최소화하고 학생의 실제 식별 정보를 LLM에 전달하지 않는다.
- AI 결과에는 생성 시각과 작업 상태를 저장하며, 실패 시 사용자에게 재시도 수단을 제공한다.

## 9. 보안 및 파일 관리

- 모든 API와 WebSocket 연결에 인증 및 역할 기반 권한 검사를 적용한다.
- 참여 코드 원문은 로그에 남기지 않으며 충분히 긴 무작위 값으로 생성한다.
- PDF는 허용된 형식과 용량만 업로드할 수 있도록 검사한다.
- 파일은 공개 경로에 두지 않고 인증된 사용자에게만 제한적으로 제공한다.
- 사용자 입력은 길이 제한과 유해 콘텐츠 검사를 적용하고 화면 출력 시 이스케이프 처리한다.
- 강의 음성 녹음을 활성화할 경우 녹음 시작 전에 참여자 동의와 보관 기간을 명시한다.

## 10. 오류 처리 및 관측성

- API는 일관된 오류 코드와 사용자용 메시지를 반환한다.
- AI 및 파일 처리 작업은 타임아웃, 제한된 자동 재시도, 수동 재시도를 지원한다.
- 서버 로그에는 요청 ID와 작업 ID를 포함하되 질문 내용, 참여 코드, 인증 토큰은 기록하지 않는다.
- API 오류율, WebSocket 연결 수, STT 지연, AI 작업 시간 및 작업 실패율을 모니터링한다.

## 11. MVP 완료 조건

1. 교수자가 수업방과 세션을 만들고 PDF를 업로드할 수 있다.
2. 학생이 수업 코드로 참여하고 진행 중인 세션에 입장할 수 있다.
3. Transcript, 질문, 반응 및 답변 상태가 새로고침 없이 참여자 화면에 반영된다.
4. 동일 학생은 하나의 질문에 반응을 한 번만 추가할 수 있다.
5. 유사 질문이 클러스터로 표시되며, AI 실패 시에도 개별 질문 기능은 정상 동작한다.
6. 교수자가 선택한 질문에 답변 내용 또는 Transcript 구간을 연결할 수 있다.
7. AI 요약과 채팅이 해당 강의의 PDF, Transcript 및 Q&A를 우선 근거로 동작한다.
8. 강의 종료 후 Transcript, 요약, 질문·답변이 포함된 기록 페이지가 생성된다.
9. 권한이 없는 사용자는 수업 자료, 기록 및 실시간 채널에 접근할 수 없다.
10. 실패한 AI 작업의 상태를 확인하고 다시 실행할 수 있다.

## 12. MVP 이후 범위

- 강의 음성 원본 저장 및 다시 듣기
- Transcript 선택 시 해당 음성 시점으로 이동
- 자동 주기 요약과 이해도 지표
- 부적절한 질문 필터링 및 Transcript 수정
- 예상문제와 복습 퀴즈 생성
- 답변의 PDF 페이지·Transcript 시각 출처 표시
- 한 학기 전체 통합 검색 및 AI 채팅
- 학습 자료 다운로드

## 13. 미정 사항

- 최종 프론트엔드·백엔드 프레임워크
- 인증 방식과 외부 로그인 연동 여부
- PDF 및 음성 파일의 최대 크기와 보관 기간
- 사용할 STT, 임베딩 및 LLM 모델
- 질문 클러스터링 주기와 유사도 기준
- 실시간 지연 시간 및 동시 접속자 목표
